# Task Type Assignment

## Prefix-Based Type System

All task types use a prefix-based naming convention that encodes the category and determines quality-gate routing.

### Prefix Rules

| Prefix | Category | Quality-Gate | Examples |
|--------|----------|-------------|----------|
| `coding.` | Code changes | Run gate (`compile -> fmt -> lint -> test`) | `coding.feature`, `coding.fix`, `coding.refactor` |
| `doc` | Docs only | Skip gate | `doc`, `doc.eval`, `doc.summary` |
| `test.` | Test pipeline | Special handling | `test.gen-cases`, `test.run`, `test.graduate` |
| `validation.` | Validation tasks | Special handling | `validation.code`, `validation.ux` |
| (other) | Meta types | Special handling | `gate`, `code-quality.simplify` |

### Quality-Gate Routing

```
IsTestableType(t) = strings.HasPrefix(t, "coding.")
isDocsOnlyType(t)  = strings.HasPrefix(t, "doc")  // covers "doc", "doc.eval", etc.
```

- `coding.*` tasks MUST pass the full quality gate before recording completion
- `doc*` tasks skip the quality gate entirely (no compile, fmt, lint, or test)
- `test.*` / `validation.*` / `gate` tasks have their own specialized workflows
- `noTest: true` in task frontmatter is an explicit override for edge cases (e.g., a code task that does not require tests)

### Type Assignment for New Tasks

When creating a new task, assign the type based on the work:

1. **Code changes** (new features, bug fixes, refactors, cleanups): use `coding.*`
2. **Documentation only** (markdown, yaml, json that does not compile): use `doc*`
3. **Test pipeline** (auto-generated by `forge task index`): uses `test.*`
4. **Validation** (auto-generated when `auto.Validation` is enabled): uses `validation.*`
5. **Stage gates** (auto-generated between phases): uses `gate`

The type is set in the task frontmatter `type` field. If omitted, `InferType()` derives it from the task ID pattern.
