# Forge Guide

## Directory Conventions

### Rules

- `manifest.md` тАФ Feature unique entry point. Read this first.
- `process/` тАФ Runtime state. Do NOT commit to git.
- `testing/` тАФ Generated by standard tasks. Do NOT hand-write.
- `tests/e2e/` тАФ Only through `/graduate-tests`. Do NOT add manually.
- `records/` тАФ Generated by `task record`. Do NOT write directly.
- `specs/` тАФ Extracted by `/consolidate-specs`. User confirms before integrating to project-level.

### Project-Level Documents

Non-skill documents shared across features:

```
docs/
  ARCHITECTURE.md       тАФ System architecture
  business-rules/       тАФ Cross-feature business rules (by domain, e.g. auth.md)
  conventions/          тАФ Technical specs (coding standards, API conventions, naming rules)
  reference/            тАФ System specs (environment, deployment, tech stack)
  decisions/            тАФ Technical decisions (/record-decision)
  lessons/              тАФ Lessons learned (/learn-lesson)
  sitemap/sitemap.json  тАФ Page element map (project-level, /gen-sitemap)
```

> Agents read `docs/business-rules/` and `docs/conventions/` during task execution for domain constraints and coding standards. These are populated by `/consolidate-specs`.

### Manifest

`manifest.md` is the single entry point for a Feature. An AI agent reads this file to understand the full context:
- **Documents** table: lists all document paths and auto-generated summaries
- **Traceability** table: PRD тЖТ Design тЖТ Tasks mapping
- **Status** (feature-level): prd тЖТ design тЖТ tasks тЖТ in-progress тЖТ completed
  - Not to be confused with task-level statuses in index.json: pending, in_progress, completed, blocked, skipped

## Quality Gate Protocol

All task-executing workflows (`/execute-task`, `task-executor` agent, `/fix-bug`, `error-fixer` agent) MUST pass the quality gate before recording completion.

### Scope Resolution

Before each `just <verb>` command, resolve scope from the task's `scope` field:

1. If `scope` is missing, empty, or `"all"` тЖТ `just <verb>` (no scope argument).
2. If `scope` is `"frontend"` or `"backend"`:
   a. Run `just project-type`, capture stdout (trimmed) and exit code.
   b. If exit code != 0, or output not in `frontend`/`backend`/`mixed` тЖТ fallback to `just <verb>`.
   c. If output == `"mixed"` тЖТ `just <verb> <scope>`.
   d. If output is `"frontend"` or `"backend"` (not mixed) тЖТ fallback to `just <verb>`.

### All-Completed Hook

After all tasks done, runs as final safety net (no scope тАФ project-wide):
1. Quality gate: `just compile тЖТ just fmt тЖТ just lint`
2. Project-wide tests: `just test`
3. E2E regression: `just e2e-setup тЖТ just probe тЖТ just test-e2e`

## Testing Lifecycle

Three layers of testing, each with distinct purpose and trigger:

| Layer | Command | Scope | When |
|---|---|---|---|
| Unit Tests | `just test [scope]` | Task-level | Every task verify step (Quality Gate) |
| Feature E2E | `just test-e2e --feature <slug>` | Feature-level | T-test-3 after scripts generated |
| Regression Suite | `tests/e2e/` | Project-level | all-completed hook; graduated via T-test-4 |

```
Unit (per task) тФАтФАтЖТ Feature E2E (T-test-3) тФАтФАтЖТ Regression (graduate to tests/e2e/)
       тЖС Quality Gate enforces              тЖС T-test-4 graduates
```

## Task-CLI

Task CLI manages task lifecycle within feature workflows.

**Typical flow**: Before starting work, run `task feature` тЖТ `task claim` to get a task тЖТ `task record` to save results + update task status.

### Key Commands

| Operation | Command | Description |
|-----------|---------|-------------|
| Switch feature | `task feature <slug>` | Set current work context |
| Claim task | `task claim` | Get next available task |
| Complete task | `task record <id> --data docs/features/{slug}/tasks/process/record.json` | **One step for record + status update** |
| Add task | `task add --title "..." [flags]` | Add task to current feature |
| Apply template | `task template <name>` | View task template content |

### `task record` Workflow

```
1. task claim           тЖТ writes process/state.json
2. During execution     тЖТ write progress to process/record.json
3. task record --data docs/features/{slug}/tasks/process/record.json  тЖТ generates records/*.md + updates index.json
```

**One command does 2 things:** generates and outputs record file тЖТ updates task status.

> For full command reference, run `task -h` or `task [command] -h`
