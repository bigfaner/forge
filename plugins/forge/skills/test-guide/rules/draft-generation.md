# Convention Draft Generation Rules (Surface-First)

This document defines the rules for generating per-surface Convention file drafts.

## Overview

The draft generation process takes Surface configuration from `.forge/config.yaml` and framework detection signals, then produces Convention files organized by Surface (not by framework). Each surface gets its own directory with a `core.md` containing 7 mandatory sections plus an assertion preference table.

The generated files conform to the structure rules in `rules/convention-structure.md` and the workflow defined in `SKILL.md`.

## Schema Validation (7+1 Section Completeness)

Every Convention draft (per-surface `core.md`) MUST include these required sections:

| # | Section | Purpose |
|---|---------|---------|
| 1 | **YAML frontmatter** | `title` and `domains` fields |
| 2 | **Auto-generated marker** | `<!-- auto-generated by forge:test-guide -->` |
| 3 | **文件位置** | Test directory, file naming, build tags, constraints |
| 4 | **隔离模型** | Isolation model specific to the surface |
| 5 | **断言重点** | Assertion dimensions with table (维度, 断言模式, 示例) |
| 6 | **超时策略** | Timeout strategy at multiple levels |
| 7 | **生命周期** | Setup, Execute, Capture, Assert, Teardown |
| 8 | **Contract/Journey 比例** | Balance between Contract and Journey tests |
| 9 | **反模式** | Table: 反模式, 危害, 替代方案 |
| 10 | **断言偏好表** | Per-framework rows: 断言库, mock 机制, fixture 模式 |

**Validation rule**: If any mandatory section is missing, the draft is INCOMPLETE and must not be presented to the user without noting the gap.

## Draft Generation Process

### Step 1: Load Surface Configuration

Read `.forge/config.yaml` and extract the `surfaces` field. Parse into `active_surfaces` array.

For each surface in `active_surfaces`:
1. Validate the surface key is one of: cli, api, web, tui, mobile.
2. Load the corresponding surface strategy template from `templates/surfaces/<surface>.md`.

### Step 2: Fill Assertion Preference Table

For each surface template, fill the `## 断言偏好表` section using framework detection results:

**When frameworks are detected**: Replace the template placeholder row with one row per detected framework.

| Detected Framework | Assertion Library | Mock Mechanism | Fixture Pattern |
|--------------------|-------------------|----------------|-----------------|
| Go testing + testify | testify/assert | interfaces + test doubles | TestMain setup |
| Ginkgo + Gomega | Gomega Expect | Ginkgo fake | BeforeEach/AfterEach |
| Vitest | expect (built-in) | vi.mock() | beforeEach/fixture |
| Jest | expect (built-in) | jest.mock() | beforeEach |
| pytest | Python assert + pytest.raises | unittest.mock / pytest-mock | @pytest.fixture |
| cargo test | assert! macro | mockall | #[cfg(test)] module |
| JUnit 5 | JUnit 5 Assertions | Mockito | @BeforeEach |

**When no framework is detected** (cold start): keep the placeholder row and add a comment:
```
<!-- TODO: Run /forge:test-guide after adding test dependencies to fill this table -->
```

### Step 3: Generate Per-Surface Files

For each surface in `active_surfaces`, generate:

#### 3a. core.md

Read the template at `templates/surfaces/<surface>.md`, fill the assertion preference table (Step 2), and produce the final `core.md` content.

All 7 sections must be present. The template provides surface-specific content for each section.

#### 3b. index.md

Create `docs/conventions/testing/<surface>/index.md` as a file index:

```markdown
---
title: "<Surface Name> 测试约定"
domains: [testing, <surface>]
---

<!-- auto-generated by forge:test-guide -->

# <Surface Name> 测试约定

- [测试策略 (core.md)](core.md)
```

### Step 4: Generate Top-Level Index

Generate `docs/conventions/testing/index.md` as a quick-reference table covering all active surfaces:

```markdown
---
title: "测试约定速查表"
domains: [testing]
---

<!-- auto-generated by forge:test-guide -->

# 测试约定速查表

| Surface | 测试类型 | 文件位置 | 断言重点 | 详细策略 |
|---------|---------|---------|---------|---------|
| cli | CLI 功能测试 | tests/<journey>/ | exit code + stdout + stderr | [cli/core.md](cli/core.md) |
| api | API 功能测试 | tests/<journey>/ | status code + response body + headers | [api/core.md](api/core.md) |
| web | Web E2E 测试 | tests/<journey>/ | DOM 可见性 + 用户操作 + URL | [web/core.md](web/core.md) |
| tui | 终端功能测试 | tests/<journey>/ | 精确文本 + 正则匹配 + 快照 | [tui/core.md](tui/core.md) |
| mobile | Mobile E2E 测试 | tests/<journey>/ | UI 可见性 + 操作响应 + 屏幕 ID | [mobile/core.md](mobile/core.md) |
```

Only include rows for surfaces present in `active_surfaces`.

### Step 5: Validate Draft Completeness

Before presenting to the user, validate each `core.md` against the 7+1 section schema:

1. **YAML frontmatter**: `title` and `domains` are present
2. **Auto-generated marker**: present immediately after frontmatter
3. **文件位置**: directory, file naming, build tag are non-empty
4. **隔离模型**: at least one isolation principle listed
5. **断言重点**: dimension table with at least one row
6. **超时策略**: at least one timeout level defined
7. **生命周期**: all 5 steps present (Setup, Execute, Capture, Assert, Teardown)
8. **Contract/Journey 比例**: ratio target is defined
9. **反模式**: table with at least one row
10. **断言偏好表**: table with columns (断言库, mock 机制, fixture 模式)

If validation fails, fill gaps with surface-appropriate defaults and flag them as "inferred (not detected from project)".

## Draft Output Format

Present the drafts to the user in this format:

```
Convention Drafts Generated
============================

Surfaces: <active_surfaces list>
Framework detected: <framework name> (confidence: <level>)

Files to write:
  docs/conventions/testing/index.md (quick-reference table)
  <for each surface>:
    docs/conventions/testing/<surface>/index.md
    docs/conventions/testing/<surface>/core.md

Review options:
  (a)ccept - Write all files
  (e)dit   - Tell me what to change
  (r)eject - Discard and regenerate with feedback
```

## User Review and Feedback Loop

### Acceptance

When user accepts (`a`):
1. Write all files (see File Write Rules).
2. Output confirmation with file paths and section summary.
3. Proceed to the report step in SKILL.md.

### Edit Request

When user requests edits (`e`):
1. Ask: "What would you like to change?"
2. Apply the requested changes to the affected draft(s).
3. Re-present the updated summary with diff markers.
4. Do NOT increment retry counter.

### Rejection with Feedback

When user rejects with feedback (`r`):
1. Parse the user's feedback to identify approved and rejected files/sections.
2. Regenerate ONLY rejected files/sections, preserving approved content verbatim.
3. Increment retry counter.

**Retry limit**: Maximum 2 retries (3 total presentations including the initial one).

### Retry Exhausted

After 2 retries are exhausted and the user still rejects:
1. Mark all files as `[DRAFT - needs manual review]`.
2. Write all drafts to `docs/conventions/testing/<surface>/core.draft.md`.
3. Output:

```
Retry limit reached. Drafts written to:
  docs/conventions/testing/<surface>/core.draft.md
  ...

Please manually edit the files, then rename from .draft.md to .md.
The pipeline will wait for confirmation before proceeding.
```

## File Write Rules

### Directory

Convention drafts are written to `docs/conventions/testing/` with per-surface subdirectories:

```bash
mkdir -p docs/conventions/testing
for surface in <active_surfaces>; do
  mkdir -p "docs/conventions/testing/$surface"
done
```

### File naming

- Accepted drafts: `docs/conventions/testing/<surface>/core.md` and `docs/conventions/testing/<surface>/index.md`
- Top-level index: `docs/conventions/testing/index.md`
- Pending drafts (retry exhausted): `docs/conventions/testing/<surface>/core.draft.md`

### Auto-generated marker

Every generated Convention file MUST include the auto-generated marker immediately after YAML frontmatter:

```markdown
---
title: "..."
domains: [...]
---

<!-- auto-generated by forge:test-guide -->
```

### Existing file handling

If a Convention file already exists at the target path:
- Without `--force`: Present a diff and ask for confirmation (handled by SKILL.md Step 0)
- With `--force`: Overwrite directly

## Inferred Defaults

When a field cannot be determined from project signals, use these framework-specific defaults:

| Framework   | Default runner          | Default assertion        | Default tag format        |
|-------------|------------------------|--------------------------|---------------------------|
| Go testing  | `go test -v -json`     | testify/assert           | `//go:build <surface>-<type>`          |
| Ginkgo      | `ginkgo -v --json-report=report.json` | Gomega Expect | `Label("<surface>-<type>")` |
| Vitest      | `vitest run --reporter=verbose` | `expect` from vitest | `describe('@tag', ...)` |
| Jest        | `jest --verbose`       | `expect` from @jest/globals | `describe('@tag', ...)` |
| pytest      | `pytest -v`            | Python assert            | `@pytest.mark.<tag>`     |
| JUnit 5     | `mvn test`             | JUnit 5 Assertions       | `@Tag("name")`           |
| Rust        | `cargo test`           | `assert!` macro          | `#[cfg(feature = "<surface>-<type>")]` |

These defaults are auxiliary reference for filling the assertion preference table, not for driving the generation flow.
