# Convention File Structure Rules (Surface-First)

<HARD-RULE>
Convention files are organized by Surface (cli/api/web/tui/mobile), not by framework.
Each surface has its own directory containing an index.md and a core.md.
Framework information is reduced to a single row in the assertion preference table within core.md.
</HARD-RULE>

## Directory Structure

```
docs/conventions/testing/
  index.md              # Quick-reference table: Surface -> type -> location -> assertion focus
  cli/
    index.md            # File index
    core.md             # CLI test strategy (language-agnostic)
  api/
    index.md
    core.md             # API test strategy (language-agnostic)
  web/
    index.md
    core.md             # Web E2E strategy (language-agnostic)
  tui/
    index.md
    core.md
  mobile/
    index.md
    core.md
```

## Per-Surface core.md

Each `core.md` contains 7 mandatory sections plus an assertion preference table. The content is language-agnostic -- LLM already knows framework syntax. Convention files only supplement project-specific rules that LLM cannot infer.

### Required Sections

1. **YAML frontmatter** with `title` and `domains`:
   ```yaml
   ---
   title: "<Surface Name> 测试策略"
   domains: [testing, <surface>]
   ---
   ```

2. **Auto-generated marker**: The first comment after frontmatter MUST be:
   ```markdown
   <!-- auto-generated by forge:test-guide -->
   ```

3. **Mandatory sections** (7 sections + 1 table):

   **文件位置**:
   ```markdown
   ## 文件位置

   - **目录**: <test directory>
   - **文件命名**: <file naming patterns per language>
   - **Build tag**: <tag syntax>
   - **约束**: <surface-specific constraints>
   ```

   **隔离模型**:
   ```markdown
   ## 隔离模型

   - <isolation principle 1>
   - <isolation principle 2>
   ...
   ```

   **断言重点**:
   ```markdown
   ## 断言重点

   <Surface>-specific assertion dimensions as a table with columns: 维度, 断言模式, 示例.

   **最低要求**: <minimum assertion requirement>
   ```

   **超时策略**:
   ```markdown
   ## 超时策略

   - <timeout level 1>
   - <timeout level 2>
   ...
   ```

   **生命周期**:
   ```markdown
   ## 生命周期

   1. **Setup**: <setup description>
   2. **Execute**: <execution description>
   3. **Capture**: <capture description>
   4. **Assert**: <assertion description>
   5. **Teardown**: <teardown description>
   ```

   **Contract/Journey 比例**:
   ```markdown
   ## Contract/Journey 比例

   <Surface>-specific ratio target and requirements.
   ```

   **反模式**:
   ```markdown
   ## 反模式

   | 反模式 | 危害 | 替代方案 |
   |--------|------|----------|
   | <anti-pattern> | <harm> | <alternative> |
   ```

   **断言偏好表** (per-framework rows):
   ```markdown
   ## 断言偏好表

   | 断言库 | mock 机制 | fixture 模式 |
   |--------|-----------|-------------|
   | <assertion library> | <mock mechanism> | <fixture pattern> |
   ```

4. **Content constraints**:
   - core.md MUST NOT include framework-specific code templates, import blocks, or runner commands. LLM has this knowledge.
   - The assertion preference table columns are fixed: 断言库, mock 机制, fixture 模式. Adding new columns requires a proposal review.
   - Each framework occupies exactly one row in the assertion preference table.

## Cold Start Framework Candidates

Used when no test files exist and framework detection has low confidence. The detection step presents these candidates for user selection:

| Language      | Primary         | Alternatives                          |
| ------------- | --------------- | ------------------------------------- |
| Go            | go testing + testify | Ginkgo v2 + Gomega               |
| JavaScript/TS | Vitest          | Jest, Mocha + Chai                    |
| Rust          | cargo test      |                                      |
| Python        | pytest          | unittest, nose2                       |
| Java          | JUnit 5         | TestNG, Spock                         |
| C#            | xUnit           | NUnit, MSTest                         |
