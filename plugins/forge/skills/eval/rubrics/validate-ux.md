# validate-ux Evaluation Rubric

**Total: 1000 points | iterations: 1 (single-pass, no revise loop)**

## Purpose

This rubric drives dynamic UX validation: compile and run the system, execute PRD user flows, capture outputs and effects, then evaluate against UX quality rules and flow correctness. The scorer evaluates a `ux-snapshot.md` artifact produced by Phase 1 pre-processing.

## Required Input

| Input | Source |
|-------|--------|
| ux-snapshot.md | Generated by Phase 1 pre-processing |
| Rubric | This file |
| Context files | conventions + business-rules per `context` frontmatter |

**Block A: UX Rules (400 pts)** -- Evaluates standalone UX quality independent of PRD flows.
**Block B: PRD Flow Validation (600 pts)** -- Evaluates whether PRD user flows actually work as described.

## Dimensions

### Block A: UX Rules

#### 1. Error Actionability (120 pts)

Evaluate error messages the system produces during both standalone checks and flow execution.

| Criterion | Points | What to check |
|-----------|--------|---------------|
| Diagnostic clarity | 0-40 | Error messages explain *what* went wrong in user-understandable language. Raw stack traces, internal error codes without explanation, or cryptic messages lose points. |
| Fix guidance | 0-50 | Each actionable error includes a suggestion for how to fix it (correct flag syntax, required argument format, valid alternatives). Errors that merely state "invalid" or "failed" without guidance lose points. |
| Context preservation | 0-30 | Error messages retain enough context (command attempted, input provided, current state) so the user can understand the failure without re-running. |

#### 2. Help Completeness (120 pts)

Evaluate the system's `--help` output.

| Criterion | Points | What to check |
|-----------|--------|---------------|
| Command coverage | 0-40 | All available commands/subcommands appear in help output. Missing commands lose points proportionally. |
| Parameter documentation | 0-40 | Every flag and argument has a description. Parameters with default values show the default. Required vs optional is clearly indicated. |
| Usage examples | 0-40 | Help includes at least one usage example for non-trivial commands. Examples cover the most common use cases. |

#### 3. Output Clarity (90 pts)

Evaluate the readability and structure of command output across all captured outputs.

| Criterion | Points | What to check |
|-----------|--------|---------------|
| Formatting & structure | 0-35 | Output uses consistent formatting (tables, lists, indentation). Key information is visually prominent, not buried in walls of text. |
| Terminology consistency | 0-30 | Output uses consistent terminology matching the help text and documentation. Same concept is not called different things in different outputs. |
| Signal-to-noise ratio | 0-25 | Output contains relevant information without excessive verbosity. Debug/internal information is not shown in normal output mode. |

#### 4. Platform UX Rules (70 pts)

Evaluate platform-specific UX conventions. The applicable sub-dimension is determined by project type (CLI / Web / TUI).

**CLI (70 pts)**:

| Criterion | Points | What to check |
|-----------|--------|---------------|
| Exit codes | 0-25 | Success returns 0. Different error types return distinct non-zero codes. No silent failures (exit 0 on error). |
| Pipe friendliness | 0-20 | Output can be piped to other commands. Progress bars/spinners use stderr, data uses stdout. |
| Progress feedback | 0-25 | Long-running operations show progress indication (spinner, percentage, dots). Operations complete silently only if truly instant. |

**Web (70 pts)**:

| Criterion | Points | What to check |
|-----------|--------|---------------|
| Loading states | 0-25 | Async operations show loading indicators. No frozen UI during network requests. |
| Form validation | 0-25 | Form inputs validate before submission with inline error messages. No silent failures or full-page errors for preventable issues. |
| Navigation & breadcrumbs | 0-20 | Current location is clear. Navigation supports expected patterns (back, deep links). No dead-end pages. |

**TUI (70 pts)**:

| Criterion | Points | What to check |
|-----------|--------|---------------|
| Key-binding discoverability | 0-25 | Available keybindings are visible or discoverable via a help overlay. No hidden-only-in-docs keybindings. |
| Layout & readability | 0-25 | Initial render shows a clear, well-aligned layout. Text does not overflow or clip. Status bar (if any) shows relevant state. |
| Status feedback | 0-20 | The TUI shows current mode/state. Non-interactive responses (help screen, invalid input) are clear and timely. |

### Block B: PRD Flow Validation

#### 5. Flow Completeness (120 pts)

For each PRD user flow, verify it executes from start to finish without blocking errors.

| Criterion | Points | What to check |
|-----------|--------|---------------|
| Flow execution success | 0-60 | Each PRD flow runs from step 1 to the final step. Flows that abort mid-way lose points proportionally. |
| Step coverage | 0-40 | Every step described in the PRD flow has a corresponding captured output. Steps skipped in snapshot without explanation lose points. |
| Missing flow identification | 0-20 | Any PRD flow not exercised in the snapshot is explicitly noted as "not executed" with reason, not silently omitted. |

#### 6. Output-Reality Consistency (120 pts)

Verify that what the system outputs matches the actual state of the world.

| Criterion | Points | What to check |
|-----------|--------|---------------|
| Output claims verified | 0-50 | If output says "created X.md", "deleted task 3", "updated 5 records" -- the snapshot verifies these claims actually happened. |
| Status accuracy | 0-40 | Output status messages (success, warning, error) accurately reflect the operation result. A "success" message for a failed operation is a severe deduction. |
| Information currency | 0-30 | Listed items, counts, and paths reflect the current state, not a stale cache. Output from list/query commands is up-to-date. |

#### 7. Data & Side Effect (120 pts)

Verify expected data changes occur and unexpected changes do not.

| Criterion | Points | What to check |
|-----------|--------|---------------|
| Expected data effects | 0-50 | Operations that should modify data (create, update, delete) show verified changes in the snapshot. Missing verification for data-modifying steps loses points. |
| Side effect isolation | 0-40 | `git diff --stat` shows changes only in expected files. Unrelated files modified by an operation lose points proportionally. |
| Effect documentation | 0-30 | Each flow step has its effect verification section filled in (not blank/placeholder). Partially filled sections lose points proportionally. |

#### 8. Idempotency & State Integrity (100 pts)

Verify repeated operations are safe and system state remains consistent.

| Criterion | Points | What to check |
|-----------|--------|---------------|
| Idempotency safety | 0-55 | Re-running the same operation does not produce duplicate data, errors, or corrupted state. The idempotency check section in the snapshot shows results. |
| State consistency | 0-45 | After multi-step flows, related state remains consistent (e.g., record count matches index, references are valid). State integrity checks are documented in the snapshot. |

#### 9. Cascade Effect (60 pts)

Verify downstream behaviors are correctly triggered (or correctly not triggered).

| Criterion | Points | What to check |
|-----------|--------|---------------|
| Triggered cascades | 0-35 | Operations that should trigger downstream behavior (hooks, notifications, dependent updates) show evidence in the snapshot that the cascade occurred. |
| Absence of spurious cascades | 0-25 | Operations that should NOT trigger downstream behavior show no unexpected cascade evidence. False triggers lose points. |

#### 10. Friction Detection (80 pts)

Detect UX friction: steps where a user would need to consult documentation to proceed.

| Criterion | Points | What to check |
|-----------|--------|---------------|
| Undocumented required input | 0-30 | Steps where the system prompts for input without explaining valid values or format. Users should not need to guess. |
| Ambiguous error recovery | 0-30 | Steps where an error occurs but the path forward is unclear from the output alone. If the user must search docs/Google to proceed, this is friction. |
| Workflow discontinuity | 0-20 | Steps that break expected workflow patterns (e.g., command succeeds but next step assumes a different state). Surprise requirements not communicated upstream lose points. |

## Deduction Rules

- **Missing flow execution** (flow in PRD but not in snapshot): -40 per flow
- **Empty effect verification** (section exists but has no content): -15 per step
- **False idempotency claim** (claims safe but evidence shows corruption): -30 per instance
- **Snapshot generated in project directory** (not in worktree/tmp): -50 (pollution risk)

## Scoring Guide

| Score Range | Interpretation |
|-------------|----------------|
| 900-1000 | Excellent UX. All flows pass, outputs match reality, errors are actionable, no friction detected. |
| 700-899 | Good UX with minor issues. Most flows complete, some friction or minor inconsistencies. Target met. |
| 500-699 | Significant UX problems. Multiple flows incomplete, several friction points, or output-reality mismatches. |
| Below 500 | Major UX failures. Most flows broken, pervasive friction, or system produces misleading output. |
