# Multi-Page Prototype: Define Navigation Contract Before Building

## Problem

多页面 HTML 原型在生成后出现三类系统性缺陷：
1. **函数冲突**：共享 JS (`app.js`) 和页面内联脚本定义同名函数，后者覆盖前者导致功能失效
2. **导航不一致**：6 个页面各自定义 Tab 栏，导致 Tab 数量不同（4-5个）、标签名不同、链接指向不存在的页面（如 `body.html`）、同一位置指向不同页面
3. **PRD 与原型脱节**：PRD 写了 8 个 UF，原型实际有 10 个页面（缺少统计页和动作详情页）；PRD 描述的交互（如"点击日期进入训练页"）与原型的实际交互（底部弹出详情卡片）不符

## Root Cause

### 因果链（3 层）

```
症状：日历点击无效、Tab 栏混乱、PRD 不匹配
  ↓
直接原因：每个页面独立生成，没有共享的导航和代码规范
  ↓
根本原因：原型生成流程缺少「先契约、后实现」的约束步骤
  ↓
触发条件：多页面原型项目 + 共享代码文件 + 页面特定代码混合
```

**根本原因是流程缺失，而非代码错误：**

1. **没有导航契约**：PRD 只写"从导航入口进入"，没有定义 Tab 有几个、叫什么、指向哪。每个页面独立生成时各自想象了一套导航结构。

2. **没有代码边界契约**：`app.js` 同时包含通用工具（tab 高亮、toast）和页面特定逻辑（`selectDay`），没有约定哪些函数属于共享层、哪些属于页面层。

3. **没有原型↔PRD 对齐步骤**：PRD 先写完，原型后生成。原型在生成过程中自然演变（新增统计页、交互方式改变），但没有回写 PRD 的机制。PRD 变成了历史文档而非需求真相。

## Solution

### 在生成多页面原型之前，必须先建立两个契约：

**契约 1：导航架构表**

在生成任何页面前，定义一个表格并作为所有页面的共享参考：

```markdown
| Tab序号 | 标签 | 目标文件 | 图标 |
|---------|------|---------|------|
| 1       | 日历 | calendar.html | calendar-icon |
| 2       | 计划 | plan.html     | clipboard-icon |
| ...     | ...  | ...           | ...           |

Push 页面：
| 页面 | 入口 | 返回目标 |
|------|------|---------|
| 训练执行 | 日历「开始训练」 | calendar.html |
| 动作详情 | 动作库点击动作 | exercise-library.html |
```

**契约 2：代码分层约定**

```markdown
共享层 (app.js)：tab 高亮、toast、通用 UI 组件（modal、accordion、slider）
页面层 (inline <script>)：页面特定业务逻辑、数据定义、事件处理

规则：
- 共享层不得定义任何页面特定的函数
- 页面层通过脚本加载顺序确保优先级（inline > app.js）
```

**对齐步骤：原型完成后回写 PRD**

原型是需求的最终表达。生成后必须：
1. 逐一对比原型页面与 PRD UF
2. 发现新增页面 → 补充 UF
3. 发现交互差异 → 更新 UF 的 User Interaction Flow
4. 更新 Page Composition 表

## Key Takeaway

**多页面原型的质量取决于页面间的「缝隙」处理，而非单个页面的精美程度。**

具体规则：
1. **先建导航表，再生成页面** — 所有页面从同一份导航契约生成 Tab 栏，消除不一致
2. **共享代码严禁包含页面特定逻辑** — `app.js` 只放通用工具，页面函数留在 inline script
3. **原型是需求的最终真相** — 生成后必须回写 PRD，否则 PRD 会变成过时的历史文档
4. **集成测试不可省略** — 生成后必须点一遍所有跨页面链接，而非只看单页面效果
