# 先分析再动手 — 并且分析本身要保持独立判断

## Problem

用户要求"优化生成器 skill，不要迁就评分器"，我直接读了评估器 rubric、对照模板找 gap、动手改了 5 个文件。整个过程没有先输出分析、没有等待确认。

## Root Cause

表面原因是跳过了确认环节。但更深层的问题是：

1. **以评分器为标准做 gap analysis，恰恰违背了用户"不要迁就评分器"的要求。** 我的分析框架是"rubric 维度 X 在模板中缺失 → 补上"。这不是客观中立，这是把 rubric 当成了 ground truth。正确的做法应该是独立判断每个 section 的质量标准，然后交叉参考 rubric，而非反过来。

2. **没有质疑前提。** 用户说优化了三个评分器，我默认评分器定义了"好文档"的标准。但评分器可能本身有偏差或遗漏。我没有问：这些 rubric 维度本身是否合理？是否有维度是冗余的？是否有重要维度缺失？

3. **把"对应地优化"理解成了"镜像补齐"。** 评分器有 6 个维度，我就确保生成器在每个维度上都有产出。但生成器的目标不是填满评分器的表格，而是产出对下游（写 PRD、写代码）真正有用的文档。有些 rubric 维度可能不需要显式 prompting 也能自然覆盖。

4. **违反了 CLAUDE.md 的核心原则。** "Think Before Coding" 要求先陈述假设——我没有声明"我假设 rubric 是正确标准"这个关键前提。"State your assumptions explicitly" — 我跳过了这一步直接执行。

## Solution

对于"优化 X 以匹配 Y"类任务：

1. **先独立评估 X 的现状**，不参考 Y — X 本身有什么问题？哪里弱？
2. **再参考 Y** — Y 指出了哪些 X 独立分析没发现的盲点？
3. **输出差距分析**，标注哪些是 X 自身的问题、哪些是 Y 视角补充的
4. **等待用户确认方向后**，再动手

## Key Takeaway

"不要迁就评分器"不只是一句修饰语 — 它意味着生成器的优化应该从"什么是一份好文档"出发，而不是从"评分器扣分项"出发。流程上：先分析（独立判断 + 交叉验证）、再呈现、再执行。
