# 大文件生成导致 Subagent 卡死 2 小时

## Problem

`/gen-test-scripts` 任务（T-test-2）在 subagent 中执行时卡了近 2 小时，最终被用户手动中断。

- 实际工具调用仅在 09:06–09:12（约 6 分钟）完成
- 写完 `api.spec.ts`（18 TC, 23KB）后，subagent 尝试一次性生成 `ui.spec.ts`（52 TC, 预估 60–70KB）
- 从 09:12:33 起卡住，直到 11:10:28 被用户中断，耗时约 **2 小时无产出**

## Root Cause

**单次输出量超出模型能力**：

1. **输入膨胀** — `test-cases.md` 本身 61KB / 1264 行，占满 context window
2. **输出膨胀** — 52 个 UI+Integration 测试用例需要生成约 60–70KB 代码，超出单次输出上限
3. **任务设计粒度过粗** — gen-test-scripts 作为单一任务，要求一次生成所有脚本文件（API + UI）
4. **无分块机制** — skill 和 task 都没有"逐文件生成"的策略，默认一次性输出全部内容

因果链：
- 症状：subagent 卡 2 小时无响应
- 直接原因：尝试一次性生成 60KB+ 的 `ui.spec.ts`
- 根本原因：任务未按输出大小拆分，导致单次生成超出模型输出能力

## Solution

当生成任务的预估输出超过 ~30KB 时，必须拆分为多个子任务：

```
# 拆分策略：按输出文件拆分
T-test-2a: 生成 API 测试脚本（TC-053~TC-070, ~23KB）
T-test-2b: 生成 UI 测试脚本（TC-001~TC-052, ~60KB）
  → 可进一步拆分为：
  T-test-2b1: 生成 Milestones 页面测试（TC-001~TC-030）
  T-test-2b2: 生成 Existing 页面修改测试（TC-031~TC-052）
```

**判断标准**：
- 单文件预估超过 30KB → 必须拆分
- 总测试用例超过 25 个 → 考虑按功能模块拆分
- 参考：18 个 API TC → 23KB，52 个 UI TC → 预估 60–70KB

## Key Takeaway

**大文件生成任务必须按输出大小拆分，不能让 subagent 一次性生成超过 30KB 的代码。**

- 拆分维度：按输出文件（api.spec.ts vs ui.spec.ts）或按功能模块
- 预估公式：每个 TC 约 1.3KB（API）或 1.2KB（UI），超过 25 TC 的文件应拆分
- 拆分后的子任务可以并行分派，反而更快

**反模式**：将大量测试用例的脚本生成作为单一任务 → subagent 卡死在超大输出上。
