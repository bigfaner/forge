# 1.1: MatchSurface API 增强 + forge surfaces --json

## Description

修改 `MatchSurface()` 返回 `SurfaceMatch{Key, Type}` 结构体（当前仅返回 type 字符串），并在 `forge surfaces` CLI 新增 `--json` flag 输出结构化 JSON。这是整个 surface 信息传播链的数据源——下游 breakdown-tasks、quick-tasks、fix-task、init-justfile、run-tests 均依赖此 CLI 查询 surface 信息。

## Reference Files

`design/tech-design.md#Interface-1a` — MatchSurface 新签名、SurfaceMatch 结构体定义、消费方列表
`design/tech-design.md#Interface-1b` — --json flag 三模式规格、JSON 输出 schema、stderr 格式覆盖声明
`design/tech-design.md#Dependencies` — 内部依赖关系（已存在/需修改签名/需新增）

## Acceptance Criteria

- Given `MatchSurface()` 接受 surfaces map 和文件路径
- When 调用 `MatchSurface()` 查询 `"src/components/App.tsx"`
- Then 返回 `SurfaceMatch{Key: "admin-panel", Type: "web"}` 而非仅 `"web"`
- And `forge surfaces src/components/App.tsx --json` 输出 `[{"key": "admin-panel", "type": "web"}]`
- And `forge surfaces --json`（无参数，列表模式）输出 `{"surfaces": [...]}`
- And `forge surfaces --types --json` 输出 `{"types": ["web", "api"]}`
- And surfaces 配置缺失时 `--json` 模式 stderr 输出 `{"error": "..."}` 并 exit 1
- And 现有调用方 `runSurfacesQuery` 适配新签名，无 `--json` 时保持文本输出不变
- And `go test ./pkg/forgeconfig/... ./internal/cmd/...` 全部通过

## User Stories

- Story 3 (surface-key 迁移): 前置依赖——breakdown-tasks 需要 `forge surfaces --json` 获取 surface-key
- Story 2 (自动化测试编排): run-tests 需要 CLI 查询 surface-type 加载执行策略

## Hard Rules

- `--json` 模式下 stderr 必须使用 `json.NewEncoder(cmd.ErrOrStderr()).Encode()` 输出 JSON 错误，不得使用 `fmt.Fprintf`
- SurfaceMatch 定义在 `pkg/forgeconfig` 包中

## Implementation Notes

参考 Interface 1a 的新旧签名对比和 Interface 1b 的三模式行为表。现有调用方仅 `surfaces.go` 的 `runSurfacesQuery`，需从 `SurfaceMatch.Type` 取值。标量形式（单 key "."）时 `Key` 返回 `"."`。
