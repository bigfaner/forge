# Journey: Automated Test Orchestration

## Overview

A Forge user runs `run-tests` which automatically detects the surface type from the task's frontmatter or `forge surfaces` CLI, loads the corresponding execution strategy rule, and executes the correct orchestration sequence (e.g., dev -> probe -> test -> teardown for web/api, build -> dev -> test for cli/tui).

## Setup

- A task file exists with `surface-type: web` in its frontmatter (or `surface-type: cli`, etc.)
- The project's justfile contains the surface-specific recipes generated by init-justfile
- The `just` binary is installed and version >= 1.4.0
- For web/api surfaces: the dev server recipe is configured to start in the background

## Happy Path

### Step 1: Run run-tests with task frontmatter surface-type

**User Action**: Execute `run-tests` for a task that has `surface-type: web` in its frontmatter

**Expected Result**: run-tests reads the surface-type from the task frontmatter. No fallback to `forge surfaces` CLI is needed.

### Step 2: Load execution strategy rule file

**User Action**: (Automatic -- run-tests loads the rule)

**Expected Result**: run-tests loads `skills/run-tests/rules/surfaces/web.md`. The rule defines the orchestration sequence: `dev (background) -> probe -> test -> teardown`.

### Step 3: Execute dev (background start)

**User Action**: (Automatic -- run-tests executes `just dev`)

**Expected Result**: The dev server starts in the background. run-tests records the PID in `.forge/test-state.json`. Exit code 0.

### Step 4: Execute probe (retry polling)

**User Action**: (Automatic -- run-tests executes `just probe`)

**Expected Result**: Probe checks service readiness with up to 3 retries at 30-second intervals (total timeout 90s). Logs format: `[probe] [retry <current>/<max>] <url> -- <reason>`. On success (exit 0), proceeds to test step.

### Step 5: Execute test

**User Action**: (Automatic -- run-tests executes `just test`)

**Expected Result**: Tests run. On exit 0, proceeds to teardown. On exit 1 (retryable), still proceeds to teardown.

### Step 6: Execute teardown

**User Action**: (Automatic -- run-tests executes `just test-teardown`)

**Expected Result**: Background processes are terminated (PID-based). `.forge/test-state.json` is cleaned up. Teardown is idempotent -- PID not existing is not an error.

## Edge Cases

### Step 1b: Task frontmatter missing surface-type, fallback to forge surfaces CLI

**Precondition**: Task frontmatter has no `surface-type` field

**User Action**: Run run-tests

**Expected Result**: run-tests falls back to `forge surfaces <file-path>` CLI to query the surface type via longest-prefix-match. On success, proceeds to load rule file.

### Step 2b: Both surface-type sources unavailable

**Precondition**: Task frontmatter has no `surface-type` AND `forge surfaces` CLI fails or returns no result

**User Action**: Run run-tests

**Expected Result**: run-tests outputs an error to stderr with the failure reason and a recovery hint ("configure surfaces in .forge/config.yaml or specify surface-type in task frontmatter"). Exits with exit code 2 (blocking).

### Step 3b: Execution strategy rule file not found

**Precondition**: Surface type is detected but the corresponding rule file does not exist (e.g., `skills/run-tests/rules/surfaces/web.md` is missing)

**User Action**: Run run-tests

**Expected Result**: run-tests outputs an error to stderr: "surface type '<type>' execution strategy rule file not found, supported types: web/api/cli/tui/mobile". Exits with exit code 2 (blocking).

### Step 4b: Probe fails with retryable error (exit 1)

**Precondition**: Dev server started but probe detects a transient error (connection timeout, DNS failure)

**User Action**: (Automatic -- probe retries up to 3 times)

**Expected Result**: After 3 consecutive probe failures (retryable), run-tests executes teardown, then exits with exit code 1 (retryable). HARD-GATE enforced: no probe retry or dev restart within the same orchestration cycle.

### Step 5b: Probe fails with blocking error (exit 2)

**Precondition**: Probe detects a non-transient error (port occupied, auth failure, config error)

**User Action**: (Automatic -- probe fails immediately)

**Expected Result**: run-tests executes teardown, then exits with exit code 2 (blocking). HARD-GATE enforced: no retry within this cycle. Upper-level scheduler (e.g., CI) distinguishes retryable(1) vs blocking(2).

### Step 6b: Dev server crashes during probe

**Precondition**: Dev server process terminates unexpectedly while probe is retrying

**User Action**: (Automatic -- probe continues retrying)

**Expected Result**: Probe retries exhaust (3 attempts, 90s total). run-tests executes teardown (PID already gone -- idempotent skip). Exits with exit code 1 (retryable). No orphan process remains.

### Step 7b: Teardown itself fails (kill error)

**Precondition**: Teardown attempts to kill a PID but the kill syscall fails

**User Action**: (Automatic -- teardown error handling)

**Expected Result**: Teardown retries the kill once. If it still fails, logs the process info and continues cleanup. Ensures `.forge/test-state.json` is cleaned up (deleted or marked completed).

### Step 8b: Session interruption leaves stale state

**Precondition**: A previous run-tests was interrupted (e.g., user Ctrl+C), leaving `.forge/test-state.json` with stale PIDs

**User Action**: Run run-tests again

**Expected Result**: run-tests detects the stale state file, cleans up residual processes, and proceeds with a fresh orchestration cycle.

### Step 9b: CLI/TUI surface orchestration (no probe, no teardown)

**Precondition**: Task has `surface-type: cli` or `surface-type: tui`

**User Action**: Run run-tests

**Expected Result**: Orchestration sequence is `build -> dev -> test`. No probe step, no teardown step. Dev is not a background server but a build/compile step.

### Step 10b: Mobile surface orchestration

**Precondition**: Task has `surface-type: mobile`

**User Action**: Run run-tests

**Expected Result**: Orchestration sequence is `test-setup -> dev -> test -> teardown`. `test-setup` prepares the emulator. `teardown` cleans up emulator and processes.

### Step 11b: forge surfaces CLI output format unexpected

**Precondition**: `forge surfaces` CLI returns output in an unexpected format (not valid JSON or missing expected fields)

**User Action**: Run run-tests

**Expected Result**: run-tests outputs an error to stderr with the actual output content and a recovery hint ("forge surfaces CLI output format unexpected, expected JSON format {surface-key, surface-type}, verify forge CLI version"). Exits with exit code 1 (retryable).

## Journey Invariants

- HARD-GATE: After probe failure, no probe retry or dev restart within the same orchestration cycle -- this is an inviolable constraint
- Every error message includes both (1) the specific failure reason and (2) a recovery hint (BIZ-error-reporting-002)
- Teardown is always idempotent: PID not existing is not an error; kill failure retries once then logs and continues
- Exit code semantics are consistent: 0=success, 1=retryable, 2=blocking
- `.forge/test-state.json` is always cleaned up (deleted or marked completed) regardless of orchestration outcome
- No orphan processes remain after run-tests completes (regardless of success or failure path)
