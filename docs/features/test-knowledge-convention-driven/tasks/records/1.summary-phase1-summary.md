## 4. Conventions Established

- **Convention-driven test generation**: Test framework/language knowledge lives in `docs/conventions/testing-*.md` files (user-editable markdown), not in Go code. Skills read Convention files via LLM, not CLI commands.
- **Recipe-existence over config lookup**: `pkg/just.ResolveScope` probes the actual justfile rather than reading project-type from config. This makes scope resolution behavior-driven rather than configuration-driven.
- **Extract-not-rewrite for config**: `pkg/forgeconfig/` was extracted from `pkg/profile/config.go` to preserve the subtle `raw` map default-tracking mechanism, rather than rewritten from scratch.
- **Silent migration**: Unknown YAML fields are silently ignored so existing config.yaml files with removed fields (project-type, languages, etc.) continue to work without user intervention.
- **Generated test tasks reference Convention path**: e2e task files now say "Read `docs/conventions/testing-*.md`" instead of embedding strategy content.
- **Test function naming**: Function names referencing "profile" (e.g., TestBuildIndex_MultiProfile) are acceptable when they test language-per-profile logic in testgen, not the deleted Profile struct.

## 5. Deviations from Design

- **[1.4 vs I-1]** Design doc I-1 specifies Config with only Auto and Worktree fields. Implementation added `TestFramework` and `TestCommand` fields to forgeconfig.Config because `forge test run-journey` and `forge config get` commands depend on them. These fields were NOT removed as the design stated — they were retained as implementation necessities.
- **[1.4 vs I-6]** Design says `forge test verify` and `forge test promote` work unchanged. Implementation confirmed: `promote` had zero changes (already profile-free); `verify` implementation lives in a separate file (test_verify.go) and was not modified.
