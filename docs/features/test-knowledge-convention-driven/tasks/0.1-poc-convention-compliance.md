# 0.1: POC: Validate LLM Convention compliance

## Description
Validate the foundational assumption: LLM reads Convention markdown and correctly applies it during test code generation. This POC determines whether the Convention approach is viable before committing to the full rewrite.

Create Convention files for 3 frameworks, run gen-test-scripts with Convention-only context (no Profile content), measure first-pass compile rate and Convention accuracy.

## Reference Files
- `plugins/forge/skills/gen-test-scripts/SKILL.md` — current skill to temporarily modify
- `docs/conventions/` — where Convention files will be placed
- `forge-cli/pkg/profile/languages/go/generate.md` — current Profile content (for comparison baseline)

## Acceptance Criteria
- Convention files created for 3 frameworks: Go testing (forge-cli's actual setup), Go ginkgo, TypeScript vitest
- gen-test-scripts invoked with Convention files loaded and NO Profile content (temporarily disable forge test detect/get calls in the skill)
- Measure: (a) first-pass `just e2e-compile` success rate, (b) Convention-specified import/assertion/tag accuracy in generated code
- Gate: first-pass compile rate >= 70% on Go testing framework (forge-cli's 126+ existing tests)

## User Stories
- Story 3 (cold start): validates that LLM can generate compilable code from Convention alone

## Hard Rules
- Do NOT modify any Go code — this POC only modifies the gen-test-scripts skill prompt temporarily
- Revert skill changes after POC (or keep if proceeding to Phase 2)
- Document POC results in `docs/features/test-knowledge-convention-driven/design/poc-results.md`

## Implementation Notes
- Write Convention files following the schema defined in tech-design.md Data Models section
- For Go testing: `assert (not require)`, `//go:build e2e`, `go test -json`
- For ginkgo: `ginkgo` imports, `Describe`/`It`/`Expect` style
- For vitest: `describe`/`it`/`expect`, `@Tag` decorators
- Run gen-test-scripts on 5-10 representative Journeys for each framework
