# 4.2: Diff equivalence validation

## Description
Compare Convention-generated code against Profile-generated code for framework core patterns. Imports, assertion functions, and tag syntax must be identical. Style differences (variable naming, comments) are allowed.

## Reference Files
- `tests/e2e/` — existing test files (Profile-generated)
- Regenerated test files (Convention-generated)

## Acceptance Criteria
- For each existing Journey: regenerate using Convention-driven gen-test-scripts
- Diff framework core patterns (import statements, assertion function calls, build tags) — must be 0 difference
- Style differences (variable names, comments, whitespace) are acceptable
- Convention file for forge-cli project correctly declares: Go testing + testify, assert (not require), //go:build e2e, go test -json

## User Stories
- Story 4 (backward compatibility): generated code diff equivalence

## Hard Rules
- Framework core pattern diff must be exactly 0 — any difference is a regression

## Implementation Notes
- Use `diff` or `git diff` to compare regenerated files against existing
- Focus comparison on: import blocks, assertion calls, build tags, test function signatures
- Ignore: variable names, comments, whitespace, helper function implementations
