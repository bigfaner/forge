# 2.3: Atomic write primitives: WithLock + AtomicWrite

## Description

Extend `pkg/index/lock.go` with `WithLock(indexPath, fn func() error)` callback for advisory locking. Extend `pkg/index/atomic.go` with `AtomicWrite(path, data, perm)` primitive for crash-safe file writes (temp+rename). Update `SaveStateAtomic` to use AtomicWrite.

## Reference Files

- design/tech-design.md — Interface 2: Atomic Write (full signatures, standard pattern, why WithLock)
- design/tech-design.md — Security Considerations § Stale lock mitigation

## Acceptance Criteria

- [ ] `WithLock(indexPath, fn func() error) error` wraps advisory lock with 5s timeout from `pkg/constants/`
- [ ] `AtomicWrite(path, data, perm) error` implements temp+rename
- [ ] `SaveIndexAtomic` uses AtomicWrite internally
- [ ] `SaveStateAtomic` uses AtomicWrite internally
- [ ] Unit tests: concurrent access (2 goroutines), crash safety (temp cleanup), atomic behavior
- [ ] `go build ./...` compiles

## User Stories

- Story 1: Safe Concurrent Task Operations — AC: lock conflict error (not corrupted index), valid JSON after concurrent writes, lock timeout AIError

## Hard Rules

- `WithLock` is the ONLY way to acquire the index lock. Never call `LockFile`/`UnlockFile` directly.
- `pkg/index/` must remain a leaf package — no internal deps. LoadIndex/SaveIndexAtomic stay in the caller's callback.
- Verify POSIX flock auto-release on process death (Linux + macOS).

## Implementation Notes

The WithLock callback pattern absorbs the encapsulated-index-lock proposal. It wraps the entire read-modify-write cycle, eliminating the TOCTOU gap that SaveIndexLocked had. See design Interface 2 for the standard caller pattern.
