# 2.3: WithLock callback + atomic state writes

## Description

Add `WithLock` callback wrapper to `pkg/index/lock.go` (wraps existing `LockFile`/`UnlockFile`). Convert non-atomic state writes to atomic (temp+rename) in `pkg/feature/forge_state.go` and `pkg/task/state.go`.

Existing primitives (`SaveIndexAtomic`, `LockFile`, `UnlockFile`) are unchanged — this task adds the callback wrapper and extends atomic writes to state files.

## Reference Files

- design/tech-design.md — Interface 2: WithLock + SaveStateAtomic
- `forge-cli/pkg/index/lock.go` — existing LockFile/UnlockFile
- `forge-cli/pkg/index/atomic.go` — existing SaveIndexAtomic
- `forge-cli/pkg/feature/forge_state.go` — non-atomic state writes
- `forge-cli/pkg/task/state.go` — non-atomic SaveState

## Acceptance Criteria

- [ ] `WithLock(indexPath string, fn func() error) error` wraps LockFile/UnlockFile with 5s timeout
- [ ] `pkg/feature/forge_state.go`: WriteForgeState/EnsureForgeState/MarkFeatureCompleted use atomic write (temp+rename)
- [ ] `pkg/task/state.go`: SaveState uses atomic write (temp+rename)
- [ ] `pkg/feature/forge_state.go`: ClearForgeState writes `false` instead of deleting file
- [ ] Unit tests: WithLock concurrent access, lock conflict, atomic write crash safety
- [ ] `go build ./...` compiles

## Hard Rules

- `WithLock` is the ONLY way to acquire the index lock going forward
- `pkg/index/` must remain a leaf package — no internal deps
- All state file writes must use atomic pattern

## Implementation Notes

`WithLock` should handle lock release in defer (safe even on panic). For atomic state writes, use the same temp+rename pattern as `SaveIndexAtomic` in `pkg/index/atomic.go`. Extract a shared `AtomicWrite(path, data, perm)` helper if it reduces duplication.
