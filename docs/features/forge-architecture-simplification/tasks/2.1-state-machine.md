# 2.1: State machine: ValidateTransition + CheckTransitionDeps

## Description

Create `pkg/task/statemachine.go` implementing centralized transition validation. Two-phase design: `ValidateTransition` (pure state check) + `CheckTransitionDeps` (dependency resolution). Includes `TransitionRole` enum (submit/claim/reopen/auto), `canAutoUnblock` unexported helper, and the transition rule table.

## Reference Files

- design/tech-design.md — Interface 1: Centralized Transition Validation (full signature, transition table, caller pattern)
- design/state-transition-diagram.md — 转换规则表, 按角色分类的规则

## Acceptance Criteria

- [ ] `TransitionRole` enum defined with 4 values: submit, claim, reopen, auto
- [ ] `ValidateTransition(current, target, role) error` implements full transition table
- [ ] Terminal state protection: completed → * blocked (all roles), rejected/skipped → * blocked (except RoleReopen → pending)
- [ ] `CheckTransitionDeps(index, taskID) ([]string, error)` for blocked → pending/in_progress
- [ ] `canAutoUnblock` unexported helper checks active fix-task count
- [ ] Unit tests cover all 6×6 state combinations + role isolation (7 scenarios from design)
- [ ] `go build ./...` compiles, `go test ./pkg/task/...` passes

## User Stories

- Story 2: Correct State Machine Enforcement — AC: ValidateTransition returns error for completed task submit; reopen works for rejected/skipped; completed reopen blocked
- Story 7: Reopen Rejected or Skipped Tasks — AC: ValidateTransition allows rejected/skipped → pending via RoleReopen

## Hard Rules

- `canAutoUnblock` is unexported — not a standalone interface.
- No `--force` / `force bool` parameter anywhere.
- Transition table is the single authority — no other code path may validate transitions.

## Implementation Notes

Follow the caller pattern from design exactly. The transition table should be data-driven (slice of TransitionRule) with a lookup function, not a chain of if/else. See design Interface 1 for the complete table with From/To/Role/Allowed/Guard columns.
