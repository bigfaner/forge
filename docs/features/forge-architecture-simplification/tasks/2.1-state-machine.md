# 2.1: State machine: ValidateTransition + CheckTransitionDeps

## Description

Create `pkg/task/statemachine.go` implementing centralized transition validation. Two-phase design: `ValidateTransition` (pure state check) + `CheckTransitionDeps` (dependency resolution). Includes `TransitionRole` enum, `canAutoUnblock` unexported helper, and the transition rule table.

## Reference Files

- design/tech-design.md — Interface 1: Centralized Transition Validation
- design/state-transition-diagram.md — transition rules by role

## Acceptance Criteria

- [ ] `TransitionRole` enum defined with 4 values: submit, claim, reopen, auto
- [ ] `ValidateTransition(current, target, role) error` implements full transition table
- [ ] Terminal state protection: completed → * blocked (all roles), rejected/skipped → * blocked (except RoleReopen → pending)
- [ ] `CheckTransitionDeps(index, taskID) ([]string, error)` for blocked → pending/in_progress
- [ ] `canAutoUnblock` unexported helper checks active fix-task count
- [ ] Unit tests cover all state combinations + role isolation
- [ ] `go build ./...` compiles, `go test ./pkg/task/...` passes

## Hard Rules

- `canAutoUnblock` is unexported — not a standalone interface
- No `--force` / `force bool` parameter anywhere
- Transition table is the single authority — no other code path may validate transitions

## Implementation Notes

Follow the caller pattern from design exactly. The transition table should be data-driven (slice of TransitionRule) with a lookup function, not a chain of if/else.
