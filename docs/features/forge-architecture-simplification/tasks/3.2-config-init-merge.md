# 3.2: Config init merge (huh TUI)

## Description

Delete the bufio-based config init path (`internal/cmd/config.go` configInitCmd), keep only the huh TUI path (`internal/cmd/init.go` configInitFunc). Ensure the unified init exposes all 7 auto configuration fields.

Current state: `forge init` uses huh TUI (interactive, styled), `forge config init` uses bufio (plain text prompts). Both produce the same config structure.

## Reference Files

- design/tech-design.md — CC-2, CE-3
- `forge-cli/internal/cmd/init.go` — huh TUI path (keep)
- `forge-cli/internal/cmd/config.go` — bufio path (remove)

## Acceptance Criteria

- [ ] Bufio config init code path deleted from `internal/cmd/config.go`
- [ ] `forge config init` delegates to huh TUI path (or becomes alias)
- [ ] All 7 auto fields (e2eTest, consolidateSpecs, cleanCode, validation, runTasks, gitPush, knowledgeSave) exposed in init wizard
- [ ] `go build ./...` + `go test ./...` pass

## Hard Rules

- Must preserve all existing init behavior that users rely on

## Implementation Notes

`validation`, `runTasks`, `knowledgeSave` are ModeToggle type (Quick/Full sub-fields); `e2eTest`, `consolidateSpecs`, `cleanCode` are also ModeToggle; `gitPush` is plain bool. The huh TUI path in init.go already uses `huh.NewConfirm`, `huh.NewInput`, etc.
