# 2.6: Migrate index writers to WithLock pattern

## Description

Migrate claim, add, build_index commands from direct `task.SaveIndex` (non-atomic) to `WithLock + SaveIndexAtomic` pattern. Also migrate `quality_gate.go` addFixTask. Ensures all index writers use advisory locking and atomic writes.

Submit migration is handled separately in task 2.4.

## Reference Files

- design/tech-design.md — Interface 2 § Standard pattern for all index writers
- `forge-cli/internal/cmd/claim.go` — NO lock, uses `task.SaveIndex` (non-atomic)
- `forge-cli/pkg/task/build.go` — uses `task.SaveIndex` (non-atomic)
- `forge-cli/internal/cmd/quality_gate.go` — addFixTask: NO lock

## Acceptance Criteria

- [ ] `internal/cmd/claim.go` uses `WithLock(indexPath, fn)` for index writes
- [ ] `internal/cmd/add.go` uses `WithLock(indexPath, fn)` for index writes
- [ ] `pkg/task/build.go` uses `SaveIndexAtomic` for index save
- [ ] `internal/cmd/quality_gate.go` addFixTask uses `WithLock` for index writes
- [ ] No remaining calls to `task.SaveIndex` on index.json paths (grep verified)
- [ ] `go build ./...` + `go test ./...` pass

## Hard Rules

- Every index write path MUST go through WithLock. No exceptions.
- grep for `task.SaveIndex` to verify complete migration

## Implementation Notes

Follow the standard pattern from design Interface 2. Each command wraps its read-mutate-write cycle in `WithLock`. The submit migration is handled in task 2.4.
