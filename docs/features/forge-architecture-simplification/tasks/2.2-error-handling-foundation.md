# 2.2: Error handling foundation: AIError factories + Exit codes

## Description

Add 4 new AIError factory functions (`ErrInvalidTransition`, `ErrEvalParseFailure`, `ErrInvalidPath`, `ErrContractUnverifiable`) to `cmd/errors.go`. Update `Exit()` to use differentiated exit codes (1 for retryable, 2 for blocking). Add `ExitCode()` method to AIError.

## Reference Files

- design/tech-design.md — Error Handling § Error Types & Codes, Propagation Strategy, Exit Code Strategy

## Acceptance Criteria

- [ ] 4 new factory functions added to `cmd/errors.go`
- [ ] `AIError.ExitCode()` returns 2 for `ErrInvalidTransition`, `ErrInvalidPath`, `ErrContractUnverifiable`; 1 for all others
- [ ] `Exit(err)` calls `printAIError` + `os.Exit(aiErr.ExitCode())`
- [ ] Unit tests for exit code mapping (4 scenarios from design)
- [ ] Existing error behavior unchanged — no regression in commands not yet migrated
- [ ] `go build ./...` compiles

## User Stories

- Story 3: Actionable Error Messages — AC: error format consistent, includes Code/Message/Cause/Hint/Action

## Hard Rules

- Do NOT migrate individual commands yet — that's tasks 2.4-2.10. This task only adds the foundation.
- `verify_task_done.go` already uses exit code 2 as hook — do NOT modify it.

## Implementation Notes

See design Error Handling section for the full exit code mapping table and per-file migration plan. The dual-context explanation (hook vs Bash tool) is documented but doesn't affect implementation — exit codes are the same regardless of context.
