# 2.2: Error handling: new error codes + exit code differentiation

## Description

Add 4 new error codes and factory functions to existing `internal/cmd/errors.go`. Add `ExitCode()` method to AIError for differentiated exit codes (1 for retryable, 2 for blocking). Update `Exit()` to use `ExitCode()`.

The AIError system already exists with 17+ factory functions and `Exit()`. This task only extends it — no migration of individual commands.

## Reference Files

- design/tech-design.md — Error Handling section (exit code strategy, new error codes)
- `forge-cli/internal/cmd/errors.go` — existing AIError system

## Acceptance Criteria

- [ ] 4 new error codes: `ErrInvalidTransition`, `ErrInvalidPath`, `ErrEvalParseFailure`, `ErrContractUnverifiable`
- [ ] 4 new factory functions with appropriate Message/Cause/Hint/Action fields
- [ ] `AIError.ExitCode()` returns 2 for blocking errors, 1 for retryable
- [ ] `Exit(err)` calls `os.Exit(aiErr.ExitCode())` instead of always `os.Exit(1)`
- [ ] Unit tests for exit code mapping
- [ ] Existing error behavior unchanged
- [ ] `go build ./...` compiles

## Hard Rules

- Do NOT migrate individual commands — that's tasks 2.4-2.10
- `verify_task_done.go` already uses exit code 2 as hook — do NOT modify it
- Existing error factory functions must keep their current behavior

## Implementation Notes

See design Error Handling for the full exit code mapping table. The dual-context explanation (hook vs Bash tool) is documented but doesn't affect implementation — exit codes are the same regardless of context.
