# 2.5: Reopen command + Status read-only

## Description

Add `cmd/reopen.go` — new cobra subcommand `forge task reopen <id>` for re-activating rejected/skipped tasks to pending. Make `cmd/status.go` read-only: remove state mutation logic, error when called with status argument.

## Reference Files

- design/tech-design.md — Interface 1 (RoleReopen, status read-only), Layer Placement § reopen.go
- design/state-transition-diagram.md — RoleReopen 规则, task reopen 子命令, task status 只读化

## Acceptance Criteria

- [ ] New file `cmd/reopen.go` with cobra subcommand `task reopen`
- [ ] `forge task reopen <id>` calls `ValidateTransition(current, "pending", RoleReopen)`
- [ ] rejected → pending ✓, skipped → pending ✓, completed → error, non-terminal → error
- [ ] `cmd/status.go`: state mutation code removed, status argument → error "task status is read-only. Use forge task submit to complete a task."
- [ ] Characterization test `TestStatus_AllowsMutation` updated to expect error
- [ ] `go build ./...` + tests pass

## User Stories

- Story 7: Reopen Rejected or Skipped Tasks — all 4 ACs covered
- Story 2: Correct State Machine Enforcement — AC: status with argument returns read-only error

## Hard Rules

- Reopen target is always `pending` — user cannot specify other states.
- Completed tasks are NEVER re-openable. Error message: "task already completed, create a subtask if re-work needed".
- Status must error on ANY status argument, not just specific values.

## Implementation Notes

Register reopen as a subcommand of the task command group. The reopen command needs WithLock for index write (depend on 2.3 indirectly through claim/add pattern). For status read-only, simply remove the state-update code path and return an error if len(args) > 0.
