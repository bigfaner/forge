# 2.5: Reopen command + Status read-only

## Description

Add `internal/cmd/reopen.go` — new cobra subcommand `forge task reopen <id>` for re-activating rejected/skipped tasks to pending. Make `internal/cmd/status.go` read-only: remove state mutation logic (2-arg update mode) and `--force` flag.

## Reference Files

- design/tech-design.md — Interface 1 (RoleReopen, status read-only)
- design/state-transition-diagram.md — RoleReopen rules, status read-only behavior
- `forge-cli/internal/cmd/status.go` — current 2-arg update mode with --force

## Acceptance Criteria

- [ ] New file `internal/cmd/reopen.go` with cobra subcommand registered under taskCmd
- [ ] `forge task reopen <id>` calls `ValidateTransition(current, "pending", RoleReopen)`
- [ ] rejected → pending ✓, skipped → pending ✓, completed → error, non-terminal → error
- [ ] Reopen uses `WithLock` for index write
- [ ] `internal/cmd/status.go`: state mutation code removed, 2-arg call returns error "task status is read-only. Use forge task submit to complete a task."
- [ ] `--force` flag removed from status command
- [ ] Characterization test `TestStatus_AllowsMutation` updated to expect error
- [ ] `go build ./...` + tests pass

## Hard Rules

- Reopen target is always `pending` — user cannot specify other states
- Completed tasks are NEVER re-openable
- Status must error on ANY status argument, not just specific values

## Implementation Notes

Register reopen as a subcommand of the task command group in `internal/cmd/root.go`. For status read-only, remove the 2-arg code path entirely and return an error if `len(args) > 1`.
