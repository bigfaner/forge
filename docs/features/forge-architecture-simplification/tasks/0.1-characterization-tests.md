# 0.1: Characterization tests for current behavior

## Description

Write characterization tests that lock in current (including "illegal but allowed") behavior for claim, submit, status, add, quality_gate, and build commands. These tests form the regression safety net for Phase 1 (must pass unchanged) and Phase 2 (expectations updated to reflect behavior changes).

## Reference Files

- design/tech-design.md — Testing Strategy section (17 named test scenarios)
- design/state-transition-diagram.md — 6-state transition diagram
- docs/proposals/forge-architecture-simplification/defect-inventory.md — SM-1~SM-8, QG-1~QG-3, GI-1 defect descriptions

## Acceptance Criteria

- [ ] `TestSubmit_CurrentBehavior_AllowsCompletedResubmit` — submit on completed task currently succeeds (will fail in Phase 2)
- [ ] `TestAdd_BlockSource_CurrentBehavior_AllowsCompletedToBlocked` — block-source on completed task currently succeeds
- [ ] `TestClaim_AutoUnblock_CurrentBehavior` — auto-unblock with/without BlockedReason
- [ ] `TestQualityGate_SourceTaskID_IsSentinel` — verify current sentinel `"quality-gate:N"` behavior
- [ ] `TestQualityGate_CountFixTasks_CountsAll` — verify counts completed fix-tasks (not active-only)
- [ ] `TestBuildIndex_Orphan_WarningOnly` — verify orphan is only warned, not cleaned
- [ ] `TestStatus_AllowsMutation` — verify status command currently allows state changes
- [ ] `TestSubmit_AutoDowngrade_NoBlockedReason` — auto-downgrade sets status to blocked but NOT BlockedReason (SM-6)
- [ ] All 8 tests pass against current codebase without modification

## User Stories

- Story 1: Safe Concurrent Task Operations — tests lock behavior gaps
- Story 2: Correct State Machine Enforcement — tests transition rule gaps
- Story 6: Quality Gate Actually Works — tests QG sentinel/cap gaps

## Hard Rules

- Tests must capture **current** behavior exactly, including behaviors we consider bugs. Do NOT fix bugs in these tests.
- Use `testing + testify` — no external test frameworks.
- Place tests in `forge-cli/internal/cmd/` alongside the commands they test (or `forge-cli/pkg/` for package-level tests).

## Implementation Notes

Use Go table-driven tests. Each test should document the "expected current behavior" vs "desired future behavior" in comments, so Phase 2 knows which expectations to update.
