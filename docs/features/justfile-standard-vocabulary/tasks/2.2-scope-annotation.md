# 2.2: breakdown-tasks scope annotation prompt

## Description

Insert the Scope Assignment section into `breakdown-tasks/SKILL.md` after Step 4a (Business Tasks). This adds the scope classification algorithm that determines each task's scope (frontend/backend/all) based on file paths, and writes it into the generated index.json.

## Reference Files

- `plugins/forge/skills/breakdown-tasks/SKILL.md` — Target file for scope annotation logic insertion
- `docs/features/justfile-standard-vocabulary/design/tech-design.md` — Interface 3 (scope field), Interface 4 (Skill Scope Resolution Protocol), breakdown-tasks Scope 标注 section
- `docs/features/justfile-standard-vocabulary/prd/prd-spec.md` — Section 5.3: 任务级 scope 标记

## Acceptance Criteria

- [ ] Scope Assignment section inserted in SKILL.md after Step 4a
- [ ] Classification algorithm: paths matching frontend patterns (`ui/`, `src/`, `components/`, `pages/`, `styles/`, `public/`) → frontend
- [ ] Classification algorithm: paths matching backend patterns (`cmd/`, `internal/`, `pkg/`, `api/`) → backend
- [ ] Scope computation: all frontend → `"frontend"`, all backend → `"backend"`, mixed/undetermined → `"all"`
- [ ] Non-mixed projects: all tasks receive `scope: "all"`
- [ ] Scope Resolution Protocol (Interface 4) available for skill files to reference
- [ ] Generated index.json tasks include `scope` field per task

## User Stories

- **Story 3: 智能任务 scope 标记**
  - Given 一个混合项目的 tech design, When `/breakdown-tasks` 拆分任务时, Then `index.json` 中每个任务包含 `scope` 字段
  - Given 一个任务仅涉及前端目录文件, When `/breakdown-tasks` 处理该任务时, Then scope 标记为 `frontend`
  - Given 一个任务涉及前后端目录文件, When `/breakdown-tasks` 处理该任务时, Then scope 标记为 `all`
  - Given 一个非混合项目, When `/breakdown-tasks` 拆分任务时, Then 所有任务的 scope 标记为 `all`

## Implementation Notes

- This is prompt engineering — the scope classification logic is expressed as instructions in SKILL.md that Claude follows when running breakdown-tasks
- The exact text to insert is defined in the tech-design doc under "breakdown-tasks Scope 标注"
- The scope field is optional in index.json (omitempty), so existing index.json files without scope continue to work
- Consumers should treat missing scope as `"all"` (default behavior)
