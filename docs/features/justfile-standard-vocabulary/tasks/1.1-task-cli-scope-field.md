# 1.1: task-cli scope field: Go structs + JSON schema

## Description

Add `scope` field to task-cli's Go `Task` and `TaskState` structs, update the JSON schema for index.json validation, and add unit tests for scope serialization/deserialization. This is the data layer foundation that enables scope-aware task execution.

## Reference Files

- `task-cli/pkg/task/types.go` — Task and TaskState struct definitions (Model 1, Model 2)
- `task-cli/pkg/task/types_test.go` — Existing struct tests
- `plugins/forge/skills/breakdown-tasks/templates/index.schema.json` — JSON Schema for index.json (Model 3)
- `docs/features/justfile-standard-vocabulary/design/tech-design.md` — Model 1-3 definitions

## Acceptance Criteria

- [ ] `Task` struct has `Scope string` field with json tag `"scope,omitempty"`
- [ ] `TaskState` struct has `Scope string` field mirroring `Task.Scope`
- [ ] `index.schema.json` `tasks.additionalProperties.properties` includes `scope` property with enum `["frontend", "backend", "all"]` and default `"all"`
- [ ] Unit test: `Task` with `Scope: "frontend"` serializes to JSON containing `"scope":"frontend"`
- [ ] Unit test: `Task` with empty `Scope` omits the field from JSON output (omitempty)
- [ ] Unit test: JSON `{"scope": "backend"}` deserializes into `Task` with `Scope == "backend"`
- [ ] `task validate` accepts index.json files with scope fields
- [ ] Existing tests still pass (`go test ./...` in task-cli)

## User Stories

- **Story 3: 智能任务 scope 标记**
  - Given 一个混合项目的 tech design, When `/breakdown-tasks` 拆分任务时, Then `index.json` 中每个任务包含 `scope` 字段
  - Given 一个非混合项目, When `/breakdown-tasks` 拆分任务时, Then 所有任务的 scope 标记为 `all`

## Implementation Notes

- The `scope` field uses `omitempty` so existing index.json files without scope continue to validate
- Default value `"all"` is a convention enforced by consumers, not a Go zero value — document in struct comment
- The schema change is additive (new optional property), so existing index.json files remain valid
