# 1.1: Create go-test profile verification-strategies.md

## Description

Create the reference implementation of `verification-strategies.md` for the go-test profile. This file defines per-capability verification dimensions, boundary scenarios, and test data requirements for TUI, API, and CLI capabilities.

This is the first strategy file and serves as the template for the remaining 5 profiles (Phase 3).

## Reference Files

- `docs/features/typed-verification-strategies/design/tech-design.md` — Interface 1: Strategy File Format (template + concrete TUI example)
- `forge-cli/pkg/profile/profiles/go-test/manifest.yaml` — profile capabilities `[tui, api, cli]`
- `docs/features/typed-verification-strategies/prd/prd-spec.md` — PRD §类型验证策略矩阵 (standard strategy matrix with all 5 interface types)

## Acceptance Criteria

- [ ] File exists at `forge-cli/pkg/profile/profiles/go-test/verification-strategies.md`
- [ ] Contains exactly 3 capability sections: `## tui`, `## api`, `## cli` (matching manifest.yaml capabilities)
- [ ] Each section contains `### Verification Dimensions` with >= 3 items
- [ ] Each section contains `### Boundary Scenarios` with >= 2 items
- [ ] TUI section uses PRD-defined dimensions: golden file comparison, dimensional constraint, ANSI consistency, layout integrity
- [ ] TUI boundary scenarios include CJK characters, long paths, narrow terminal
- [ ] API section includes contract verification, error paths, boundary values
- [ ] CLI section includes output golden file, exit codes, parameter combinations
- [ ] File is <= 200 lines (PRD performance requirement)

## User Stories

- **Story 3**: As a Profile author, I want to define verification strategies per capability
  - Given a profile directory (go-test)
  - When I create verification-strategies.md
  - Then each capability section has required subsections with minimum item counts

## Implementation Notes

Use the PRD §类型验证策略矩阵 table as the primary source for dimension and scenario content. The design doc's TUI example (tech-design.md lines 99-120) provides the concrete template — extend the same pattern for API and CLI sections.
