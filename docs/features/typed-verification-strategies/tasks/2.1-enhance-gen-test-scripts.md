# 2.1: Enhance gen-test-scripts SKILL.md for Level-based code generation

## Description

Modify gen-test-scripts SKILL.md Step 4 to branch on the Level field from test-cases.md, generating structurally different test code for e2e and integration levels. Integration tests write to `tests/integration/features/<slug>/` instead of `tests/e2e/features/<slug>/`.

## Reference Files

- `plugins/forge/skills/gen-test-scripts/SKILL.md` — current skill definition (~400 lines), especially Step 4 (lines 278-398)
- `docs/features/typed-verification-strategies/design/tech-design.md` — Interface 3: Level-based Code Generation
- `docs/features/typed-verification-strategies/prd/prd-spec.md` — PRD §gen-test-scripts 级别化代码生成
- `docs/features/typed-verification-strategies/prd/prd-user-stories.md` — Story 6
- `plugins/forge/skills/gen-test-scripts/templates/` — existing templates (helpers, config, spec files)

## Acceptance Criteria

- [ ] Step 4 reads Level field from each TC in test-cases.md
- [ ] Level=e2e TCs generate code with golden file comparison imports (e.g., `os/exec` for Go) and write to `tests/e2e/features/<slug>/`
- [ ] Level=integration TCs generate code with HTTP/assertion imports (e.g., `net/http` + assert lib for Go) and write to `tests/integration/features/<slug>/`
- [ ] TCs with empty Level use current default behavior (write to `tests/e2e/features/<slug>/`)
- [ ] e2e and integration code differ in at least one of: imports, assertion method, or directory structure (PRD S6 requirement)
- [ ] Shared infrastructure step (Step 3.5) handles both `tests/e2e/` and `tests/integration/` directories
- [ ] Output section updated to document the dual-directory staging area
- [ ] Error handling table updated for Level-specific scenarios

## User Stories

- **Story 6**: As a Forge user, I want gen-test-scripts to generate different code structures based on Level
  - Given test-cases.md contains Level=e2e (TUI) and Level=integration (API) TCs
  - When gen-test-scripts executes
  - Then e2e code includes golden file reading + writes to `tests/e2e/`; integration code includes HTTP assertions + writes to `tests/integration/`; both differ in imports, assertions, and directory

## Implementation Notes

The key change is in Step 4 where spec files are generated. Currently all output goes to `tests/e2e/features/<slug>/`. The modification adds a conditional:
- Parse Level from TC entry
- If Level=integration → output to `tests/integration/features/<slug>/`
- Otherwise → output to `tests/e2e/features/<slug>/` (current default)

The import/assertion differences are driven by the profile's `generate.md` strategy, which already differentiates by type (TUI/API/CLI). The Level branching adds directory separation and emphasizes the structural difference.

Read the existing gen-test-scripts SKILL.md carefully before editing — it has detailed template handling, compilation checks, and VERIFY marker resolution that must be preserved.
