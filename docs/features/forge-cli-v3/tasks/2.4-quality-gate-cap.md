# 2.4: Quality-gate max fix-task cap

## Description

Add a maximum fix-task cap (3 per step) to the quality-gate command. When a quality-gate step fails and there are already 3 active fix-tasks for that step, print an error instead of creating a new fix-task.

## Reference Files

- `design/tech-design.md` §Interfaces — 7. Quality-Gate Max Fix-Task Cap
- `design/tech-design.md` §Error Handling — New Error Cases (max fix-tasks reached)
- Current: `forge-cli/internal/cmd/all_completed.go` (being renamed to `quality_gate.go` in task 2.2)

## Acceptance Criteria

- [ ] `countActiveFixTasks(index *task.TaskIndex, step string) int` function added to `quality_gate.go`
- [ ] `maxFixTasksPerStep = 3` constant defined
- [ ] `addFixTask` checks cap before creating fix-task: returns `("", ErrMaxFixTasks)` when cap exceeded
- [ ] Fix-task identification: `SourceTaskID != ""` AND title has prefix `"fix <step>:"`
- [ ] Terminal states (completed, skipped) excluded from count
- [ ] When cap reached: stderr `"max fix-tasks reached for <step>, manual intervention required"`, exit code 1
- [ ] Unit tests: 3 active fix-tasks → cap error on 4th; completed/skipped fix-tasks not counted
- [ ] `go test ./...` passes

## User Stories

- Story 4: When 3 active fix-tasks exist for a step, `forge quality-gate` outputs "max fix-tasks reached for <step>, manual intervention required" and exits 1
- Story 4: New fix-task title includes step name and sequence number

## Implementation Notes

This task modifies `quality_gate.go` (renamed from `all_completed.go` in task 2.2). The cap check is inserted before the existing `task.AddTask → task.CreateTaskMarkdown → feature.EnsureForgeState` sequence. The `TaskIndex` API is unchanged — only the counting logic is new. Use `strings.HasPrefix` for title matching with the `"fix <step>:"` prefix convention.
