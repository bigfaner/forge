# 2.5: Concurrent write conflict handling for submit

## Description

Add advisory file locking to the submit command to handle concurrent write conflicts. When two agents submit results simultaneously for the same feature, one succeeds and the other receives a "concurrent write conflict, retry" error. Implement flock-based locking with a 5-second timeout.

## Acceptance Criteria

- [ ] `pkg/index/lock.go` created with `LockFile(indexPath string) (*os.File, error)` and `UnlockFile(f *os.File) error`
- [ ] `ErrLockConflict` error defined: `"concurrent write conflict, retry"`
- [ ] Lock file: `<feature-dir>/tasks/index.json.lock` (created if not exists, reused)
- [ ] 5-second timeout on lock acquisition; returns `ErrLockConflict` on timeout
- [ ] Windows support: use `LockFileEx` via Go's `syscall` package (or cross-platform library)
- [ ] `submit.go` (`runSubmit`): acquire lock before reading index.json, defer unlock
- [ ] Atomic index write: write to `.tmp` file, then `os.Rename` to `index.json`
- [ ] Error cases: lock timeout → exit 1, stderr "concurrent write conflict, retry"; lock creation failure → exit 1, stderr "failed to create lock file: <reason>"
- [ ] Unit tests: concurrent access simulation, timeout behavior, lock release on process exit
- [ ] `forge cleanup` does NOT touch `index.json.lock` files
- [ ] `go test ./...` passes

## User Stories

- Story 3: When two agents submit concurrently, one succeeds (exit 0) and the other gets "concurrent write conflict, retry" (exit 1). `index.json` remains valid JSON after conflict.

## Implementation Notes

Use Go's `golang.org/x/sys/unix` for `flock(2)` on Unix and `syscall.LockFileEx` on Windows. Alternatively, use a cross-platform file locking approach. The lock scope is per-feature (each feature has its own `index.json.lock`). Lock files are not deleted after use — they persist for reuse. The atomic write pattern (tmp + rename) ensures `index.json` is never in a partial state.
