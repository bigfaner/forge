# 2.1: Create Surface Rules + Detection

## Description

创建 5 个 surface rule 文件（CLI/TUI/WebUI/Mobile/API），每个包含检测信号、测试指导原则、测试策略指导和必须 Outcome 参考。同时更新 gen-journeys SKILL.md 以加载和应用 surface rules，并将检测结果持久化到 `.forge/config.yaml` 的 `surface` 字段。

Surface 与 Convention 正交分离：Surface rule 定义"测什么、怎么隔离、关注什么"（测试策略），Convention 定义"用什么框架、怎么写断言"（框架知识）。

## Acceptance Criteria

- [ ] 5 个 surface rule 文件创建于 `plugins/forge/skills/gen-journeys/rules/`：
  - `surface-cli.md` — 检测信号（main.go + cobra/urfave 等）、subprocess 测试策略、exit code/stdout/stderr 关注点、必须衍生 `not-found` + `already-exists`
  - `surface-tui.md` — 检测信号（tea.Program/tview 等）、terminal I/O 策略、异步 Cmd timeout、必须衍生 `timeout`
  - `surface-webui.md` — 检测信号（React/Vue/Svelte 等）、browser automation 策略、必须衍生 `validation-error` + `session-expired`
  - `surface-mobile.md` — 检测信号（AndroidManifest/xcodeproj 等）、Maestro YAML 骨架 + deep link、best-effort 策略
  - `surface-api.md` — 检测信号（http.Handler/gin/echo/express 等）、HTTP client 策略、必须衍生 `unauthorized`
- [ ] 每个 surface rule 文件包含 4 个 section：检测信号、通用测试指导原则、测试策略指导、必须 Outcome 参考
- [ ] gen-journeys SKILL.md 增强为：读取 surface rules → 检测项目类型 → 写入 `.forge/config.yaml` surface 字段
- [ ] 检测信号表覆盖 PRD § Flow Description 中全部信号组合（14 条规则 + 混合/未知 fallback）
- [ ] 无法匹配或匹配到多个类型时暂停管线，输出检测到的信号供用户确认
- [ ] 新增 surface 类型只需添加一个 `surface-<type>.md` 文件，无需修改管线代码

## User Stories

- **Story 3** (prd-user-stories.md): Surface 差异化 — 管线根据项目类型自动采用不同策略
- **Story 7** (prd-user-stories.md): 可扩展 surface 类型系统 — 新增类型只需添加配置文件

## Hard Rules

- Surface rule 文件是指导性内容（自然语言），不是可执行配置
- Surface 与 Convention 正交：surface rule 不引用 convention 文件，反之亦然
- 检测信号表必须与 PRD § Flow Description 的信号组合完全一致
- 使用相对路径引用 rule 文件（`rules/surface-cli.md`），不使用绝对路径

## Implementation Notes

- Surface 持久化到 `.forge/config.yaml` 的 `surface` 字段，读写方式：LLM 直接读写 YAML（或 `forge config get/set surface`，如 CLI 已支持）
- SurfaceDetectionResult 数据模型（tech-design § Data Model 1）记录检测结果，用于诊断
- 每个 surface rule 文件的"必须 Outcome 参考"是示例锚点，不是硬性清单——LLM 结合项目实际判断
- Mobile surface 为 best-effort：不要求完整覆盖，复杂场景标记 manual-only
