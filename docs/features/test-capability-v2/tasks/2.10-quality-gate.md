# 2.10: Update quality gate for multi-stage eval

## Description

将现有单一质量门禁（gen-test-cases 评分）替换为多阶段门禁（eval-journey → eval-contract → 置信度评级），每阶段独立 pass/fail 判定，门禁结果汇入统一质量报告。

## Reference Files

- `docs/features/test-capability-v2/prd/prd-spec.md` § Scope (质量门禁更新), § Quality Gate Integration (BIZ-quality-gate-001 关系)
- `docs/features/test-capability-v2/design/tech-design.md` § Quality Gate Integration
- `plugins/forge/hooks/hooks.json` — hook 注册
- `plugins/forge/hooks/guide.md` — forge 规范指南（可能引用管线步骤）
- `docs/conventions/forge-distribution.md` — 分发模型约束

## Acceptance Criteria

- [ ] 质量门禁从单一阶段更新为三阶段：eval-journey → eval-contract → 置信度评级
- [ ] 每阶段独立 pass/fail 判定，互不干扰
- [ ] eval-journey 失败不触发 eval-contract（串行依赖）
- [ ] eval-contract 失败不触发 gen-test-scripts（串行依赖）
- [ ] 置信度评级为信息性指标，不阻断管线（LOW 只标记，不阻止）
- [ ] 门禁结果汇入统一质量报告
- [ ] BIZ-quality-gate-001 关系正确：
  - BIZ-quality-gate-001 验证源代码质量（开发阶段）
  - 新 eval 门禁验证测试产物质量（测试生成阶段）
  - 串行执行、独立判定
  - BIZ-quality-gate-001 的 e2e 结果汇入 Fact Table 作为 runtime 来源

## User Stories

- **Story 4** (prd-user-stories.md): 评测门禁 — 多阶段门禁保障测试产物质量

## Hard Rules

- 门禁更新不能影响 BIZ-quality-gate-001 的现有逻辑
- eval 门禁的 pass/fail 判定由 rubric 阈值决定，不由代码逻辑硬编码
- hooks.json 中的质量门禁触发逻辑需要同步更新

## Implementation Notes

- 需要检查 hooks.json 中 Stop hook 的质量门禁配置，确保旧的单阶段门禁被替换
- guide.md 中可能引用了旧的管线步骤，需要更新为新的多阶段门禁流程
- 门禁配置可能需要在 hooks.json 或单独的配置文件中定义阶段和阈值
