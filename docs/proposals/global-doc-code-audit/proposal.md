# Proposal: 全局文档-代码一致性审计与知识库清理

## Problem

Forge 项目的用户文档、规范文档和知识库与实际代码实现之间存在未量化的不一致：文档描述的行为与代码实际行为矛盾、规范文档引用了已废弃的实现路径、知识库中积累了大量可能过时或重复的条目。这些不一致会误导 AI 代理执行错误操作，增加新成员上手成本——此处的"新成员"指源码仓库贡献者（含 AI 代理作为虚拟贡献者），他们需要同时理解"文档说的"和"代码实际做的"两套体系，耗费额外时间在冲突信息间辨别真伪。终端用户不直接接触 docs/ 目录，其受影响路径是间接的（通过 plugin 分发物的质量间接感知）。

**具体影响实例**：
- AI 代理按 `test-pipeline-consistency-audit` 审计前文档的旧术语（`tests/e2e/`、`graduation`、`staging`）生成测试代码，实际代码已迁移到新术语（`tests/<journey>/`、tag-based promotion），导致生成的测试无法运行
- AI 代理读取 `docs/conventions/` 中描述已不存在的代码结构的规范后，按规范在错误路径创建文件，需要人工排查修正
- 新成员（源码贡献者）学习项目时，先按 ARCHITECTURE.md 理解架构，再读代码发现实际 hook 执行顺序与文档描述不同，需反复确认以辨别哪个可信

### Evidence

- 已有 5 个局部审计提案（plugin-consistency-audit、skill-ecosystem-audit、skill-instruction-audit、prompt-template-audit、test-pipeline-consistency-audit）在分析阶段发现了不同层面的不一致（提案中的不一致发现基于实际代码阅读，虽未执行修复，但其诊断过程具有参考价值）
- `test-pipeline-consistency-audit` 已确认 Go 代码仍使用旧术语（`tests/e2e/`、`E2E`、`graduation`、`staging`），而文档层已迁移到新术语（`tests/<journey>/`、tag-based promotion）
- docs/lessons/ 下积累了 133 条经验教训，未经过系统性有效性审查

- docs/conventions/ 下 18 份规范文档（顶层15份 + testing/3份），因 v3.0.0 大幅重构代码结构，其中描述已不存在的代码路径或已废弃流程的文档比例较高——已确认实例：test-type-model.md 引用了已不存在的 tests/e2e/ 路径模型；naming.md 中的部分常量名与 CLI 代码当前定义不一致（已通过 grep 抽样验证：至少涉及 `SKILL_DIR`、`PLUGIN_DIR` 两个常量名在 naming.md 中的拼写与 CLI 代码中的实际导出名存在大小写或下划线差异）
- docs/business-rules/ 下 4 份业务规则文档，同样未经过系统性有效性审查

### Urgency

项目正处于 v3.0.0 分支开发阶段，文档-代码不一致会随版本迭代持续恶化。v3.0.0 计划在 2026 年 Q3 内发布（当前分支已存在且处于活跃开发），审计需在发布前 4 周完成，为修复预留至少 2 周时间窗口。审计产出应在 v3.0.0 合入 main 前为修复提供输入。

## Proposed Solution

执行一次性的三层系统性审计，产出结构化问题报告并转化为可执行 Task：

1. **L1 用户文档层**：审计根目录用户文档（README.md、DESIGN.md）和 docs/ 下用户文档（docs/ARCHITECTURE.md、docs/user-guide/、docs/official-references/）与代码实际行为的一致性

2. **L2 规范文档层**：审计 docs/business-rules/、docs/conventions/ 与实际实现的一致性（注：docs/reference/ 目录不存在；原引用的 test-type-model.md 位于 plugins/forge/skills/test-guide/references/，属于 plugin 内部文件，不在 L2 审计范围内）
3. **L3 知识库层**：审查 docs/lessons/（133条）和 docs/decisions/（10条）的有效性，标记过时、重复、无参考价值的条目

**L3 有效性判定规则**：
- **有效**：内容描述的场景、代码路径、工具仍然存在且适用，结论仍然正确
- **过时**：引用的代码路径已不存在、描述的工具/流程已被替代、结论与当前实现矛盾
- **重复**：与另一条目内容高度重叠（相同结论或相同场景），仅保留更完整的一条
- **需更新**：核心结论仍有效但部分细节（路径、参数、版本号）已变化，需局部修正

每层审计检查三个维度：过时/错误（文档与代码矛盾）、缺失（代码做了但文档没说，或反过来）、冗余（重复或无价值的内容）。


**层级间反馈机制**：三层审计按 L1 → L2 → L3 顺序执行（可部分并行，但 L1 优先启动）。L1/L2 审计中发现的代码结构不一致，须记录到"跨层影响清单"，供 L3 审计时参考——例如 L1 发现某 hook 执行顺序与文档矛盾，L3 审计时须检查引用该 hook 的 lessons/decisions 是否也需标记为过时。反向同理：L3 发现某经验教训引用的代码路径已不存在，须通知 L2 检查引用相同路径的 convention 是否也过时。跨层影响清单在 L1/L2 完成后集中汇总（约 1 小时），L3 执行时逐条核对（已计入 L3 时间估算），反向反馈在 L3 完成后追加到 L2 报告（约 1 小时）。

### 闭环路径

审计本身不直接修复文档-代码不一致，而是遵循"审计→修复→验证"闭环：
1. **审计阶段**（本提案范围）：产出结构化问题报告和可执行 Task
2. **修复阶段**（审计完成后立即启动）：task-executor 按优先级执行修复 Task，P0 问题优先处理；本提案承诺在审计报告产出后 1 周内启动 P0 修复
3. **验证阶段**（修复完成后）：对修复结果进行抽样复核，确保一致性达标

本提案聚焦闭环的第一步——审计，为后续修复和验证提供准确的输入。

### 审计结果消费流程

1. 审计完成后，按严重级别排序输出问题报告（P0 → P1 → P2 → P3）
2. P0 问题阻断 v3.0.0 发布流程，需在发布前修复
3. P1 问题在审计后第一轮修复批次中处理
4. P2/P3 问题可延后到后续迭代处理
5. 每层审计产出独立报告，可并行启动修复，无需等待全部层级完成

**审计报告结构**（每层报告遵循统一模板）：
```
# [L1/L2/L3] 审计报告
## 审计基准
- 基准 commit: <hash>
- 审计日期: <date>
- 审计范围: <文件列表>

## 问题汇总
- P0: <count> | P1: <count> | P2: <count> | P3: <count>
- 已标记过时/重复/需更新: <count>（仅 L3）

## 问题详情
### [P0] <问题标题>
- **文件**: <path>:<line_range>
- **声明**: <文档中的描述>
- **实际**: <代码中的实际情况>
- **建议动作**: <修复建议>

## 审计质量复核
- 抽样比例: 10% | 抽样结果: <通过/未通过> | 遗漏条目数: <X> | 是否扩展复核: <是/否>
```

### Innovation Highlights

本提案的核心差异化在于以下三点：(1) **层级间反馈闭环**——不同于扁平审计，三层架构通过"跨层影响清单"实现审计发现的双向传播（L1/L2 代码结构变化同步影响 L3 知识库条目判定，L3 发现的过时路径反向触发 L2 规范复审），形成审计内部的自校验网络；(2) **审计报告模板标准化**——统一的报告结构（基准 commit、问题详情、质量复核）使审计产出可直接转化为 task-executor 可消费的 Task，缩短从发现到修复的链路；(3) **与 consolidate-specs 的互补闭环**——consolidate-specs 在增量维度防止未来漂移，本审计在存量维度清理历史不一致，两者组合形成完整的文档一致性保障。可探索的辅助自动化手段（本次审计不引入，作为后续优化方向）：
- **AST 解析辅助路径验证**：解析代码 AST 提取实际导出的函数/常量名，与文档中引用的名称自动比对，用于 P1 级别（路径不存在）的批量预检
- **TF-IDF/向量相似度去重**：对知识库条目做文本相似度计算，自动标记可能重复的条目对，减少 L3 人工去重工作量
- **git blame 识别过时段落**：通过代码变更历史识别长期未更新的文档段落，优先审计变更距离最近的代码对应的文档

**跨领域方法借鉴**：本次审计的抽样复核和质量门控机制可借鉴金融审计中的 **inter-rater reliability 协议**——即由两个独立审计者对同一样本执行审计，用 Cohen's Kappa 系数衡量两者结果的一致性。若 Kappa ≥ 0.6，说明审计方法论具有可重复性；否则需调整流程。在当前场景下，"两个审计者"可以是同一 AI 代理的两个独立 session，对试点文件分别执行审计后比对结果。此协议为 blindspot B1（审计可重复性）提供了具体度量手段，但本次审计不强制执行，作为质量不达标时的升级方案。

### 审计执行流程

**L1/L2 文档审计流程**（每份文档按以下标准化流程执行）：
1. **声明提取**：从文档中提取所有事实性声明（代码路径、命令名、配置项、行为描述）
2. **代码定位**：在代码库中定位每个声明对应的实现代码（路径引用用 `find`/`grep` 验证，行为描述用代码阅读验证）
3. **逐条比对**：将声明与实现进行对比，判断方法分三类：
   - **路径/文件引用**：用 `find`/`grep` 验证目标路径是否存在；若文档声称某路径下有特定文件，验证该文件存在且内容与描述匹配
   - **行为/流程描述**：定位相关代码（函数、配置、hook），阅读代码逻辑，对比文档描述的步骤/顺序/参数是否与代码实际行为一致；重点关注：执行顺序差异、参数名/默认值差异、错误处理差异
   - **状态/配置声明**：对比文档声称的配置值、版本号、依赖项与代码中实际定义的值
   记录不一致（含文件路径和行号）
4. **缺失检测**：识别代码中存在但文档未描述的关键行为，以及文档描述但代码中不存在的功能
5. **结果记录**：按统一格式记录每个问题（路径、行号、严重级别、建议动作）

**L3 知识库审计流程**（适配知识库条目的特殊性，不假设每条目都有对应代码）：
1. **内容分类**：将条目分为代码引用类（引用具体路径/命令）、流程规范类（描述工作流程/决策）、经验总结类（泛化结论）
2. **引用验证**：对代码引用类条目，验证引用的路径/命令是否仍然存在（同 L1/L2 步骤 2-3）
3. **适用性判断**：对流程规范类和经验总结类条目，按以下结构化规则判定：
   - **工具链变化**：条目描述的工具/命令已不在当前工具链中 → **过时**
   - **流程矛盾**：条目描述的流程步骤与当前 CLAUDE.md 或团队约定矛盾 → **需更新**
   - **路径失效**：条目引用的代码路径已不存在（经 `find`/`grep` 确认） → **过时**
   - **结论泛化**：条目结论为通用最佳实践，不依赖特定实现细节 → **有效**
   - **部分过时**：核心结论仍成立但部分细节（版本号、参数名）已变化 → **需更新**
   每条判定须记录依据（如引用的当前代码路径或配置文件位置）
4. **去重检测**：通过主题聚类识别内容重叠的条目，标记为"重复"并推荐保留更完整的一条
5. **结果记录**：按 L3 判定规则标记每条条目状态，记录判断依据

## Requirements Analysis

### Key Scenarios

- **S1**：AI 代理读取 ARCHITECTURE.md 了解系统架构时，文档描述的流程与实际 hook 执行顺序一致
- **S2**：AI 代理读取 docs/conventions/naming.md 时，命名规则与 CLI 代码中的实际常量一致
- **S3**：新成员查阅 docs/lessons/ 时，所有条目仍有参考价值，不存在指向已删除代码的教训
- **S4**（整体项目目标，非审计阶段交付物）：审计完成后经修复执行，知识库条目总数减少至不超过100条（当前143条），每条剩余条目都有明确的适用范围；审计阶段标记为过时/重复的条目占比预期不低于20%——依据：docs/lessons/ 中约 40% 的条目创建于 v2.x 时期（v3.0.0 重构了目录结构和 hook 系统），其中引用旧路径（如已废弃的 tests/e2e/）或旧流程的条目比例预计较高，20% 是从 v2→v3 变更范围推算的保守估计

### 异常场景

- **审计发现大量 P0 问题**：若 P0 问题超过 10 条，暂停后续层审计，先输出 P0 问题报告供紧急修复；P0 修复完成后，对受影响的文件重新执行审计步骤（基于新 commit），避免后续 Task 基于过时代码状态
- **知识库条目状态有争议**（部分有效、部分过时）：标记为"需更新"，在报告中说明争议点，由人工判定最终状态
- **审计期间代码发生重大变更**：基准 commit 定义为"第一个审计 Task 启动前 v3.0.0 分支的最新 commit"，在报告头部标注该 commit hash；每个 Task 启动前执行 `git diff <基准commit> -- <审计目标路径>` 检测变更，若审计目标路径下有文件变更（新增/删除/修改超过 10 行），中止当前 Task 并基于新基准重新审计受影响文件

### Non-Functional Requirements

- 审计结果必须包含文件路径和行号，可直接定位问题
- 每个问题必须标注严重级别（P0/P1/P2/P3），便于优先级排序。分级标准：
  - **P0 (Critical)**：文档描述与代码行为直接矛盾，会导致 AI 代理执行破坏性操作（如删除错误文件、覆盖有效配置）
  - **P1 (High)**：文档引用了不存在的代码路径或已废弃的流程，会导致操作失败或生成的代码无法运行
  - **P2 (Medium)**：文档描述了正确方向但细节过时（参数名变化、路径微调），不导致操作失败但增加调试成本
  - **P3 (Low)**：文档风格、措辞或非关键细节不一致，不影响功能正确性

- 生成的 Task 分为三类模板：(1) **修复类 Task**：自包含且可由 task-executor 独立执行（含上下文信息，不依赖其他 Task 的输出）；(2) **审查类 Task**：包含人工确认节点，task-executor 执行到确认点时暂停等待人工输入，确认后才继续执行；(3) **跨层验证类 Task**：依赖同一层审计完成后的跨层影响清单作为输入，由 task-executor 在清单就绪后执行——此类 Task 在报告中标注前置依赖（所依赖的层级审计报告），task-executor 读取依赖报告后独立执行验证
- 审计产出的所有 Skill、Command、任务模板、提示词模板等统一采用英文撰写

- 审计完成时间：从审计启动到三份层级报告全部产出，不超过 3 个工作日（约 25 小时有效工作时间），与 v3.0.0 发布时间线对齐。其中层间反馈协调预留约 2 小时（L1/L2 完成后汇总跨层影响清单，L3 执行时逐条核对），L1 试点审计预留 1 小时（含人工复核和准确率基线报告）

### Constraints & Dependencies

- 审计基于 v3.0.0 分支当前代码状态
- 不修改任何代码或文档，只生成报告和 Task
- 知识库清理需人工确认，不可自动删除

- 审计产出的 Skill、Command、任务模板、提示词模板等统一采用英文（动机：审计产出的 Task 模板和 Skill 将作为 Forge plugin 的一部分分发，plugin 组件遵循英文规范以确保跨团队协作兼容性；审计报告本身不限语言）

- **分发模型约束**：docs/ 目录的内容仅在源码仓库中维护，不分发到用户环境——分发的仅是 plugins/forge/ 下的内容。因此 L1/L2 审计的消费者是源码仓库维护者（含 AI 代理作为虚拟成员），而非终端用户。L3 知识库条目仅存在于源码仓库中，对终端用户不可见

## Alternatives & Industry Benchmarking

### Industry Solutions

文档-代码一致性是经典问题。常见做法：
- **Doc-as-code + CI 检查**：文档与代码同仓库，通过 CI 流水线自动验证一致性。成熟实践包括 Docsify/Docusaurus 的构建时链接检查、Swagger/OpenAPI 的 spec-to-code 对比。Google 的 `devsite` 将文档与源码变更关联，当 API 源码变更时自动标记对应文档段落需更新；Microsoft 的 Docs 平台通过 `docfx` 工具从代码注释自动生成 API 文档，确保签名级别一致性。这些方案覆盖结构化/链接层面，但语义层面（行为描述是否准确）仍需人工审查
- **Automated linting**：用工具检查文档质量。如 `markdown-link-check` 检测死链、`lichemarkdown` 检测引用有效性、`vale` 做文风一致性检查。这些工具覆盖链接层面但不覆盖语义一致性。本提案的语义审计可视为对自动化 linting 的补充
- **Periodic audit**：定期人工或半自动审查。适用于无法自动化的语义层面（如文档描述的流程是否与代码行为一致）。本提案属于此类

| 执行现有5个提案约140个task（注：plugin-consistency-audit ~56 + skill-ecosystem-audit ~14 + skill-instruction-audit ~23 + prompt-template-audit ~19 + test-pipeline-consistency-audit ~34） | 现有提案 | 已有任务定义 | 覆盖不完整（缺知识库、缺用户文档层），部分任务可能过时 | Rejected: 不够全面 |
| CI 集成持续检查 | Doc-as-code 实践 | 自动防止新增不一致 | 开发成本高，需先清理存量，当前存量问题未解决前 CI 噪音过大 | Rejected: 是审计完成后的后续改进方向，非当前首选 |
| **分层系统性审计** | 本提案 | 覆盖完整，可直接执行 | 工作量较大，一次性审计无法防止未来漂移 | **Selected: 最符合当前需求** |

**选择理由**：选择一次性审计而非持续方案（Doc-as-code + CI）基于以下考量：(1) 当前已积累大量不一致，CI 检查只能防止新增问题，无法清理存量；(2) 语义层面的一致性（文档描述的行为 vs 代码实际行为）难以通过自动化 CI 规则检测，仍需人工/AI 代理逐条判断；(3) 建立持续方案的前提是先清理到一致状态，否则 CI 规则会因存量问题产生大量噪音。本审计是建立持续机制的必要前置步骤。

**三层 vs 单层全量审计**：三层分层优于单层全量审计的原因：(1) 不同文档类型的审计方法论差异显著——用户文档需比对行为描述，规范文档需比对代码常量/路径，知识库需做有效性判断和去重，混合审计会导致每条 Task 的执行流程不一致，增加质量方差；(2) 分层允许按优先级分批产出——L1 产出后可立即启动 L1 修复，无需等待全量审计完成；(3) 质量门控按层独立执行，避免单层问题影响整体审计可信度；(4) 层级间反馈机制（L1/L2 发现的代码结构变化同步到 L3）在分层架构下实现更自然高效——分层使跨类型交叉验证（用户文档 vs 知识库 vs 规范文档）成为结构化步骤而非事后补充，单层审计虽可记录发现并在完成后做交叉检查，但缺乏显式的跨层传播机制，容易遗漏。

**后续 CI 实施路线**：审计完成后，分两阶段引入持续一致性检查——第一阶段（审计后 1-2 周）：为 docs/ 新增 CI 步骤，自动检测死链和引用的文件路径是否存在（覆盖 P1 级别问题）；第二阶段（审计后 4-6 周）：基于审计中发现的高频不一致模式，编写自定义 linter 规则检测特定模式（如文档引用的 CLI 参数名与代码常量是否匹配）。


**与 consolidate-specs skill 的关系**：consolidate-specs 从 feature 文档中提取业务规则和技术规范到 conventions/business-rules，侧重"提取+合并"的增量漂移检测。本审计是全量一致性校验，验证已有文档与代码的当前状态是否匹配——两者互补，consolidate-specs 防止未来新增漂移，本审计清理存量不一致。

## Feasibility Assessment

### Technical Feasibility

完全可行。AI 代理已具备代码阅读和交叉比对能力，无需额外工具开发。语义判断准确率的验证策略：L1 审计前先对 1 个文件（如 README.md）进行试点审计并人工复核，产出准确率基线报告（记录试点文件总声明数、正确识别数、遗漏数、误报数），确认准确率达标（漏报率 < 20%）后再全面铺开；若试点复核发现漏报率 ≥ 20%（即遗漏数占实际不一致总数的比例），调整审计流程（如缩小单 Task 审计范围、增加交叉验证步骤）后重新试点，直至基线报告达标。

### Resource & Timeline

- L1 用户文档层：12 文件（README.md + docs/ARCHITECTURE.md + DESIGN.md + user-guide/4 + official-references/5），预计 3-4 个 Task

- L2 规范文档层：约 22 文件（business-rules/4 + conventions/18），预计 3-4 个 Task
- L3 知识库层：约 143 条目，预计 7-10 个 Task（每条 Task 审查 15-20 条目，降低单批条目数以保证每条目有足够 token 做代码路径验证——每条目分配 4k-6k token，高于原 2.8k-4k 预算）
- 总计约 13-18 个 Task

**Token 成本估算**：每个审计 Task 需读取目标文档全文 + 相关代码文件。L1 每条 Task 估计消耗 60k-90k token（12 个大文件需逐一与代码比对），L2 每条 Task 估计 50k-70k token（规范文件较小但代码引用分散），L3 每条 Task 估计 60k-120k token（每批 15-20 条目，每条目 4k-6k token 用于内容读取+代码路径验证）。总计约 750k-1.2M token。


**时间估算**：每个 Task 预计执行时间 1-1.5 小时（含读取文档、定位代码、比对、记录结果）。L1 预计 4-6 小时（3-4 个 Task，含 1 小时试点审计），L2 预计 4-6 小时（3-4 个 Task），L3 预计 7-11 小时（7-10 个 Task，每批 15-20 条目，知识库条目需逐条判断并验证代码路径）。层间反馈协调 2 小时。总计约 17-25 小时有效工作时间（3 个工作日以内可完成）。

### Dependency Readiness

无外部依赖。所有审计目标文件均在项目仓库内。

## Scope

### In Scope

- L1: 根目录用户文档（README.md、DESIGN.md）和 docs/ 用户文档（docs/ARCHITECTURE.md、docs/user-guide/、docs/official-references/）与代码的一致性审计

- L2: docs/business-rules/（4份）、docs/conventions/（18份：顶层15份 + testing/3份）与实现的一致性审计。另含根目录 CLAUDE.md（AI 代理项目级指令文件，属开发工具配置，纳入规范文档层审计）
- L3: docs/lessons/（133条）和 docs/decisions/（10条）的有效性审查
- 审计产出：结构化问题报告（文件路径+行号+严重级别+建议动作）
- 将问题报告转化为可执行 Task（包括知识库条目的删除/合并建议 Task，此类 Task 需人工确认后方可执行）

### Out of Scope


- docs/ 下其他子目录：experts/ 和 superpowers/ 属于 AI 代理配置而非规范文档；features/（182个 feature 目录）的清理——conventions 中的规范是从 feature 文档中提炼的派生产物，feature 文档的不一致可通过 conventions 审计间接发现；若 L2 审计发现 conventions 描述与代码矛盾且根因在 feature 文档，可在修复阶段追溯至对应 feature 文档；proposals/（182个 proposal）属于历史决策记录，其内容正确性不直接影响 AI 代理的执行行为；其余（plan/、research/、harness-reports/、forensics/、self-evolution/、todos/、profile-authoring.md、zcode-redesign-plan.md）属于历史记录或项目规划文档
- Plugin skill/command 内部一致性（已有 plugin-consistency-audit 覆盖）
- CLI Go 代码内部质量改进
- 测试代码审查
- 直接执行任何修复或删除动作（包括手动删除）


**范围完整性说明**：L1 审计目标已覆盖项目中所有面向用户和开发者的顶层文档。根目录下除 README.md 和 DESIGN.md 外，存在 CLAUDE.md（面向 AI 代理的项目级指令文件），该文件属于开发工具配置而非用户文档，纳入 L2 规范文档层一并审计。CONTRIBUTING.md、CHANGELOG.md 等不存在，因此不会因目录层级混合而遗漏审计目标。
