# Proposal: Tech Design Skill 改造 — 数据库设计独立输出与人类审批门控

## Problem

当前 `tech-design` skill 的 Data Models 章节将数据库表设计与其他技术设计混编在 `tech-design.md` 单一文件中。这导致三个问题：

1. **评审焦点模糊**：数据库表结构是高风险设计决策（字段类型、约束、索引直接影响数据完整性和性能），但在长篇 tech-design 中容易被淹没
2. **缺少 SQL DDL 交付物**：表设计停留在 markdown 描述层面，没有可直接评审和执行的 SQL 文件，开发者需要手动翻译
3. **缺少可视化**：表间关系靠文字描述，缺乏 ER 图等直观表达

### Evidence

`templates/tech-design.md` 的 Data Models 章节仅支持伪代码结构体描述：

```
ModelName = {
    fieldName: Type       // constraint or note
}
```

无法表达外键关系、索引策略、数据库注释等实际 DDL 需要的信息。

### Urgency

数据库 schema 变更是成本最高的返工类型之一（涉及数据迁移、兼容性）。越早建立独立审批机制，越能减少实施阶段的 schema 返工。

## Proposed Solution

当 tech-design 流程检测到 feature 涉及数据库表新建或变更时，将数据库设计从 `tech-design.md` 的 Data Models 章节中拆分出来，输出到独立文件，并通过人类审批门控后才能继续后续设计章节。

### 1. 声明式检测：PRD frontmatter 标记

在 `write-prd/templates/prd-spec.md` 的 frontmatter 中新增 `db-schema` 字段：

```yaml
---
feature: "{{FEATURE_NAME}}"
status: Draft
db-schema: new | alter | new,alter | none
---
```

| 值 | 含义 |
|---|------|
| `new` | 本 feature 涉及新建数据库表 |
| `alter` | 本 feature 涉及修改已有表结构 |
| `new,alter` | 两者兼有 |
| `none` 或字段缺失 | 不涉及数据库变更 |

**设计理由**：将检测从"AI 推断"变为"声明式"。`write-prd` 阶段已经明确 feature 涉及哪些数据，此时声明是否有数据库变更是低成本的。`tech-design` 直接读取 frontmatter 即可确定行为，无需启发式猜测。

**涉及的改动**：
- `write-prd/templates/prd-spec.md`：frontmatter 新增 `db-schema` 字段（默认值 `none`）
- `write-prd/SKILL.md`：在 Step 5 Draft PRD 时，提示 AI 根据 feature 内容填写 `db-schema` 值

### 2. 新增输出文件

当 `db-schema` 不为 `none` 时，输出两个文件（与 `tech-design.md` 平级）：

| 文件 | 路径 | 内容 |
|------|------|------|
| ER 图文档 | `design/er-diagram.md` | Mermaid `erDiagram` + 字段详述表格 |
| DDL 脚本 | `design/schema.sql` | `CREATE TABLE` / `ALTER TABLE` + `COMMENT ON` 语句 |

### 3. ER 图文档结构 (`er-diagram.md`)

```markdown
# 数据库设计 — {Feature Name}

## 概述
设计目标、涉及表数量、变更范围一句话总结

## ER 图
Mermaid erDiagram（标注 [NEW] / [MODIFIED]）

## 表详述
### 表名 [NEW] / [MODIFIED]
| 字段 | 类型 | 约束 | 说明 |

## 索引设计
| 表 | 索引名 | 字段 | 类型 | 说明 |

## 关系说明
| 从 | 到 | 关系 | 业务语义 |

## 变更影响分析（仅 ALTER TABLE 时）
对现有数据的兼容性说明
```

**格式要求**：
- Mermaid `erDiagram` 用于视觉化表间关系，标注 `[NEW]` / `[MODIFIED]` 区分新建和变更
- 下方表格提供完整字段级细节（类型、约束、索引、注释等 Mermaid 无法表达的内容）
- 实体间关系标注业务语义（如"用户拥有多个订单"）
- 数据库级行为（`ON DELETE CASCADE` 等）仅出现在 `schema.sql` 中

### 4. DDL 脚本格式 (`schema.sql`)

```sql
-- ============================================================
-- Feature: {feature-slug}
-- Generated by: tech-design skill
-- Description: {一句话描述本 feature 的数据库变更}
-- ============================================================

-- [NEW] 用户偏好表
CREATE TABLE user_preferences (
    id               BIGSERIAL    PRIMARY KEY,
    user_id          BIGINT       NOT NULL REFERENCES users(id),
    preference_key   VARCHAR(100) NOT NULL,
    preference_value TEXT,
    created_at       TIMESTAMPTZ  NOT NULL DEFAULT now(),
    updated_at       TIMESTAMPTZ  NOT NULL DEFAULT now()
);

COMMENT ON TABLE user_preferences IS '用户偏好设置表';
COMMENT ON COLUMN user_preferences.user_id IS '关联用户ID';
COMMENT ON COLUMN user_preferences.preference_key IS '偏好键名';

-- [MODIFIED] users 表: 新增头像字段
ALTER TABLE users
    ADD COLUMN avatar_url VARCHAR(500);

COMMENT ON COLUMN users.avatar_url IS '用户头像URL';
```

**格式约定**：
- 文件头注释标注 feature 名称和变更描述
- `[NEW]` 标注 + `CREATE TABLE` 用于新建表
- `[MODIFIED]` 标注 + `ALTER TABLE` 用于已有表变更
- 字段对齐，表名注释
- `COMMENT ON` 语句为数据库引擎提供持久化注释
- 仅包含 DDL（`CREATE TABLE`、`ALTER TABLE`、索引、约束），不包含迁移语句

### 5. tech-design.md 的 Data Models 章节改造

当存在数据库变更时，Data Models 章节替换为摘要 + 引用：

```markdown
## Data Models

涉及 3 张表（2 新建，1 变更），核心变更：新增用户偏好表、users 表增加头像字段。

详见 [ER 图设计](er-diagram.md) 和 [DDL 脚本](schema.sql)。
```

当不存在数据库变更时，Data Models 章节保持现有内联编写方式不变。

### 6. 审批门控融入现有流程

不引入额外审批轮次，而是融入现有的 per-section 审批机制（`SKILL.md` Step 6）：

1. Draft Design 阶段（Step 5），在编写 Data Models 章节时：
   - 读取 `prd-spec.md` frontmatter 的 `db-schema` 字段
   - 若 `db-schema` 不为 `none`：生成 `er-diagram.md` 和 `schema.sql`，tech-design.md 的 Data Models 章节改为摘要引用
   - 若 `db-schema` 为 `none` 或缺失：跳过，Data Models 章节照常内联编写
2. Get Approval 阶段（Step 6），在审批 Data Models section 时：
   - 展示 `er-diagram.md` + `schema.sql` 内容供人类审批
   - 通过后继续下一章节
   - 被拒后修改并重新提交（复用现有 section 审批机制）

### 7. Cross-Layer Data Map 保持不变

`tech-design.md` 模板的 Cross-Layer Data Map 章节描述的是跨层数据流转（Storage → Backend → API → Frontend），不只是数据库设计。保留在 `tech-design.md` 中，字段名引用 `er-diagram.md` 中的定义即可。

## Alternatives Considered

| Approach | Pros | Cons | Verdict |
|----------|------|------|---------|
| Do nothing | 零改动成本 | 数据库设计评审质量低，schema 返工成本高 | Rejected: 数据库 schema 变更是成本最高的返工类型 |
| 数据库文件放独立子目录 `design/db/` | 便于未来扩展 migration 文件 | 当前只需要两个文件，子目录过度设计 | Rejected: 同级目录更简洁，需要时再拆分 |
| SQL 包含迁移语句 | 一步到位 | tech-design 阶段是设计评审，不是实施 | Rejected: 迁移逻辑模糊评审焦点，应在 task 执行阶段处理 |
| 额外增加独立审批轮次 | 审批更严格 | 增加流程步骤，拖慢设计阶段 | Rejected: 融入 per-section 审批即可达到同样效果 |

## Scope

### In Scope

| # | Item | Effort | Depends on |
|---|------|--------|------------|
| 1 | 修改 `write-prd/templates/prd-spec.md`：frontmatter 新增 `db-schema` 字段 | S | — |
| 2 | 修改 `write-prd/SKILL.md`：提示 AI 根据 feature 内容填写 `db-schema` 值 | S | 1 |
| 3 | 修改 `tech-design/SKILL.md`：Data Models 章节条件分支逻辑 + 审批流程调整，读取 `db-schema` 字段 | M | 1 |
| 4 | 修改 `tech-design/templates/tech-design.md`：Data Models 章节改为条件模板 | S | — |
| 5 | 新增 `tech-design/templates/er-diagram.md`：ER 图文档模板 | S | — |
| 6 | 新增 `tech-design/templates/schema.sql`：SQL DDL 格式约定模板 | S | — |
| 7 | 修改 `tech-design/templates/manifest-update-design.md`：注册新文件到 Documents 表 | S | 4, 5 |
| 8 | 修改 `eval-design/SKILL.md`：评审维度扩展，覆盖 er-diagram.md 和 schema.sql | M | 3-7 |
| 9 | 修改 `breakdown-tasks/SKILL.md`：Element Mapping 表新增数据库变更 → schema 任务映射 | M | 3-7 |

**Effort scale**: S = < 30 min, M = 30 min–2 h

### Out of Scope

- SQL 迁移脚本（`INSERT`/`UPDATE`/`DELETE`）的自动生成
- 数据库版本管理工具（如 migration 文件编号、回滚支持）
- `er-diagram.md` 的增量更新（修改已有 feature 的数据库设计时的差异追踪）
- 自动检测数据库 schema 与设计文档的一致性

## Key Risks

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| 数据库变更检测误判（漏检或误检） | Low | M — 漏检导致缺少审批，误检产生空文件 | 采用 PRD frontmatter 声明式标记 `db-schema`，tech-design 直接读取字段值，不依赖 AI 推断。`write-prd` 阶段已明确 feature 数据需求，此时声明成本极低 |
| er-diagram.md 与 tech-design.md 描述不同步 | Medium | M — 两处维护，更新时遗漏 | tech-design.md 中仅保留摘要引用，不重复详细内容，从结构上避免重复 |
| Mermaid erDiagram 语法限制 | Low | L — 复杂关系（如多态、自引用）表达受限 | Mermaid 图用于视觉直觉，复杂场景在表格详述中补充 |
| eval-design 评审维度扩展引入新的评分偏差 | Low | L — 新维度权重不合理 | 初版将数据库设计纳入现有的 "Interface & Model Definitions" 维度，不新增独立维度 |

## Success Criteria

- [ ] `write-prd` 生成的 `prd-spec.md` frontmatter 包含 `db-schema` 字段（值为 `new` / `alter` / `new,alter` / `none`）
- [ ] `tech-design` skill 读取 `db-schema` 字段，值不为 `none` 时输出 `design/er-diagram.md`（含 Mermaid 图 + 表格详述）和 `design/schema.sql`（含 DDL + COMMENT ON），tech-design.md 的 Data Models 章节替换为摘要引用
- [ ] `tech-design` skill 读取 `db-schema` 为 `none` 或缺失时，Data Models 章节照常内联编写，不生成额外文件
- [ ] `er-diagram.md` 包含 ER 图、表详述、索引设计、关系说明四个核心章节；有 ALTER TABLE 时包含变更影响分析
- [ ] `schema.sql` 中新建表用 `[NEW]` + `CREATE TABLE`，变更表用 `[MODIFIED]` + `ALTER TABLE`，字段和表均有 `COMMENT ON` 注释
- [ ] per-section 审批 Data Models 时，同时展示 `er-diagram.md` 和 `schema.sql`，通过后才继续下一章节
- [ ] `manifest.md` 的 Documents 表包含 ER Diagram 和 Schema SQL 行（仅在有数据库变更时）
- [ ] `eval-design` 评审覆盖 `er-diagram.md` 和 `schema.sql`
- [ ] `breakdown-tasks` 在有数据库变更时自动生成独立的 schema 执行任务
