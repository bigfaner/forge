- Cross-domain inspiration: 23/35 — Improved from iteration 1. ## ATTACKS

1. **Problem Definition (Urgency justified): urgency lacks concrete deadline or stakeholder commitment.** — Quote: "按规划下季度还有 8-10 个 go-test feature，累计约 4-5 小时纯重复劳动." The "按规划" reference has no link to a planning document, sprint board, or stakeholder commitment. The 30-minute per-feature estimate and 8-10 feature count are unsourced. What must improve: cite the planning source (e.g., link to a sprint board, roadmap document, or team planning note) and state the basis for the 30-minute estimate (e.g., "measured from git history: last 3 features averaged 28 minutes of helper setup").

2. **Requirements Completeness (Scenario coverage): scenario 9 leaves a runtime behavior gap.** — Quote: "`run-e2e-tests` 的 `go test ./...` 会同时执行 old nested 和 new flat 文件. 建议：在 scope 之外保留这些旧文件，但 marker 系统只跟踪 flat 模式生成的文件。旧文件需手动清理（Out of Scope 已声明）." Declaring it "Out of Scope" does not address the runtime consequence: if both old and new files exist for the same feature, `go test ./...` may execute duplicate tests or report conflicting results. What must improve: describe the actual runtime behavior when both files exist (do they test the same functions? will Go compile fail on duplicate test function names?) and either add a cleanup step to scope or add a risk entry for this scenario.

3. **Requirements Completeness (Non-functional requirements): test isolation NFR contradicts existing init() usage with no migration path.** — Quote from NFR: "每个测试函数使用 `t.Parallel()` 独立运行，禁止在测试中修改 package-level 变量." Quote from Problem section: "init() chdir hack" exists in multiple features (e.g., "tui-ui-design/ 有 init() chdir" and "test-scripts-per-type/ 有 init() chdir"). The NFR forbids modifying package-level state but the existing features already do this via `init()`. What must improve: either acknowledge that existing init() chdir hacks must be removed before flat mode can work safely (and include that in scope), or describe how the new flat-mode files will avoid the init() pattern while old files continue using it in the shared directory.

4. **Solution Creativity (Cross-domain inspiration): parallels are post-hoc validation, not generative inspiration.** — Quote: "The pattern of 'detecting that a processing stage is unnecessary and skipping it' appears in several other domains: Database migration frameworks (Flyway, Liquibase)... CI/CD conditional stages (GitHub Actions if: guards, GitLab CI rules)." The section describes the parallels but does not show how studying these domains influenced the proposal's design. What must improve: frame the cross-domain references as design influences rather than just validations. For example, explain what specific mechanism from Flyway's checksum-based skip logic or CI/CD's declarative guard pattern was adopted or adapted into the staging-mode design.

5. **Logical Consistency (Scope ↔ Solution ↔ Success Criteria): pytest/rust-test manifest declares flat mode but no code path honors it.** — Quote from In Scope: "manifest.yaml 加 staging-mode 字段（go-test、pytest、rust-test）." Quote from Out of Scope: "pytest/rust-test 的 flat 模式运行时行为...gen-test-scripts 对这两个 profile 的 flat 模式输出路径适配不在本次 scope 内." The manifest will declare `staging-mode: flat` for pytest/rust-test but gen-test-scripts will not read or act on that field for those profiles. This means the manifest makes a promise that the system does not keep. What must improve: either (a) scope pytest/rust-test manifest changes out entirely and only add `staging-mode` to go-test's manifest, or (b) add a note in the solution section explaining that pytest/rust-test manifest fields are declarative placeholders for future implementation and will not affect current behavior.

6. **Solution Clarity (User-facing behavior described): "After" walkthrough shows no error scenario.** — Quote from "After" walkthrough: all outputs show "OK", "PASS", and successful messages. The "Before" walkthrough explicitly shows "ERROR: compilation failed -- undefined: runCLI" but the "After" walkthrough has zero error or edge-case outputs. What must improve: add at least one error scenario to the "After" walkthrough (e.g., what happens when `gen-test-scripts` detects a merge conflict in helpers.go, or what the developer sees when `run-e2e-tests` fails in flat mode).

7. **Risk Assessment (Likelihood + impact rated): backward compatibility risk rating is unjustified.** — Quote from risk table: "向后兼容性 | L | M | staging-mode 缺失时默认 nested，现有行为不变." The mitigation "defaults to nested" is a strong guarantee that existing behavior is preserved, which would suggest L/L rather than L/M. The impact of M is not explained. What must improve: either justify the M impact rating (what specific medium-impact failure could occur despite the default?) or adjust the rating to L/L if the default truly prevents any impact.

8. **Success Criteria (Criteria are measurable and testable): "已毕业的 feature 不受影响" is unmeasurable.** — Quote: "已毕业的 feature 不受影响." No definition of what "not affected" means. Does it mean compilation still passes? Tests still pass? Markers are still readable? Graduation workflows still work? What must improve: replace with a specific, testable criterion such as "After the change, running `go test ./tests/e2e/... -tags=e2e` with only pre-existing graduated features produces identical pass/fail results as before the change, and all existing `.graduated/` markers remain readable."
