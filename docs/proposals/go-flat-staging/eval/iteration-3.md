## ATTACKS

1. **Requirements Completeness (Scenario coverage): scenario 9 describes a real runtime inconsistency that is deferred rather than resolved.** — Quote: "Go 的 `go test ./...` 会递归编译子目录，导致 old nested 和 new flat 文件同时被编译...`go test` 会执行两遍测试——old nested 中的测试可能引用已 diverge 的 helper 副本，产生不一致结果. **建议**：在 scope 之外保留这些旧文件，但 marker 系统只跟踪 flat 模式生成的文件. 旧文件需手动清理（Out of Scope 已声明）." Mitigation via `-run` filter in `run-e2e-tests` helps for that specific command, but a developer running bare `go test ./...` will get duplicate tests with potentially diverged results. What must improve: either add a mandatory cleanup step to scope (remove old nested files for features that get flat-mode equivalents), or add this as a formal risk with the `-run` filter as partial mitigation and acknowledge that bare `go test ./...` will produce inconsistent results until manual cleanup.

2. **Requirements Completeness (NFRs): test isolation migration strategy has a logical flaw.** — Quote from NFR: "旧的 nested 文件（`features/<slug>/`）保留在 Out of Scope，不迁移到 flat 目录，因此其 `init()` 不会污染 flat 模式测试." But scenario 9 states: "Go 的 `go test ./...` 会递归编译子目录，导致 old nested 和 new flat 文件同时被编译." If both old nested and new flat files are compiled together, old `init()` functions in nested directories will execute and could change working directory or modify global state, contradicting the isolation claim. What must improve: either acknowledge that old `init()` in nested files will execute during `go test ./...` and describe the impact, or reconcile by noting that `init()` in a subdirectory with a different package declaration runs independently.

3. **Logical Consistency (Scope ↔ Solution ↔ Success Criteria): pytest/rust-test manifest declares flat mode but no code path acts on it.** — Quote from In Scope: "manifest.yaml 加 staging-mode 字段（go-test、pytest、rust-test）." Quote from gen-test-scripts scope: "仅 go-test；pytest/rust-test 的 `staging-mode: flat` 字段为声明性占位，`gen-test-scripts` 当前不会为这两个 profile 读取或处理该字段." The manifest will declare `staging-mode: flat` for pytest/rust-test but no code path reads or acts on that declaration for those profiles. What must improve: either (a) scope pytest/rust-test manifest changes out entirely and only add `staging-mode` to go-test's manifest, or (b) explicitly document in the solution that pytest/rust-test manifest fields are forward declarations that will be honored in future work, and add a comment in the manifest itself (e.g., `# TODO: implement flat mode runtime behavior`) to prevent confusion.

4. **Logical Consistency (Solution addresses the stated problem): misleading graduate.md documentation is not updated.** — Quote from problem section: "go-test 的 `graduate.md` 声明 'Import Rewrite: None required'，理由是 Go 使用 module path. 但实际毕业后...没有任何一个已毕业的 go-test feature 实际经历了 'features/ staging → 搬移 + import rewrite → 毕业' 这个流程." The solution adds flat-mode skip logic to `graduate-tests/SKILL.md` but does not update the existing `graduate.md` documentation that the problem section identifies as misleading. What must improve: add "update graduate.md to document flat mode rationale" to scope, or explicitly note in out-of-scope that graduate.md documentation cleanup is deferred and explain why.

5. **Solution Clarity (User-facing behavior described): no walkthrough for run-e2e-tests failure in flat mode.** — Quote from "After" walkthrough: all outputs show "OK", "PASS", and successful messages. The "After - Error Scenario" covers `gen-test-scripts` merge conflict but not `run-e2e-tests` failure. What does the developer see when `go test` fails in flat mode? Does marker writing get skipped? Does the error output differ from nested mode? What must improve: add a brief error walkthrough for `run-e2e-tests` failure in flat mode (e.g., test fails, no marker written, re-generation step).

6. **Feasibility (Resource & timeline feasibility): estimate is undifferentiated and unjustified.** — Quote: "2-3 个任务：文档改动 + 代码改动 + 测试。估计 2-3 小时." No per-task breakdown, no team skill assessment, no bandwidth consideration. The estimate is a single range without explaining what assumptions it rests on. What must improve: provide a brief per-task time estimate (e.g., "manifest + SKILL.md changes: ~30 min; profile/embed.go + task/testgen.go: ~1-1.5 hrs; test cases: ~30 min") and note any assumptions (e.g., "implementer is familiar with the profile and taskgen code").

7. **Risk Assessment (Risks identified): no marker file race condition risk.** — The proposal describes `run-e2e-tests` automatically writing graduation markers: "run-e2e-tests 通过后自动写 graduation marker." If two features are developed in parallel and both `run-e2e-tests` invocations write markers simultaneously, there is a potential race condition on the marker directory. This risk is not identified. What must improve: add a risk entry for concurrent marker file writes, or note that markers are per-feature files (`.graduated/<slug>`) and therefore inherently isolated, making race conditions impossible.
