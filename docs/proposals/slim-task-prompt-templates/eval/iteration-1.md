## Phase 3: Blindspot Hunt

1. **[blindspot] Out of Scope "不改动 task-executor 的行为" contradicts In Scope step-merge and output format change**: The Out of Scope section states "不改动 task-executor 的行为或 prompt 消费方式" (line 445), but In Scope includes "Execution Protocol 步骤合并（11 步 → ≤8 步）" (line 112) and "输出格式从多行描述改为紧凑单行摘要" (line 115). Changing the number of execution steps and the output format IS changing the task-executor's prompt — the agent that reads the Execution Protocol will encounter a different instruction structure. The proposal likely means "不改变 task-executor 的执行语义" (the behavioral outcome), but it writes "行为" (behavior), which encompasses both the process and the outcome. This is not just ambiguous wording — it creates a verification problem for SC2, since "no behavior change" will be tested against an executor that received a structurally different prompt. — Quote: Out of Scope (line 445): "不改动 task-executor 的行为或 prompt 消费方式"; In Scope (line 115): "输出格式从多行描述改为紧凑单行摘要". — What must improve: Rewrite the Out of Scope item as "不改变 task-executor 的执行语义（输出质量和行为结果不变），但允许步骤合并和输出格式精简" to resolve the contradiction.

2. **[blindspot] SC2's 90% threshold is unenforceable without a difference classification rubric**: SC2 states "轨迹一致性 ≥ 90%（容差：步骤顺序因 LLM 生成随机性导致的非功能性差异）" (line 485). The proposal defines the threshold but not the measurement method for classifying differences. In practice, an agent's trajectory can differ in many ways: step order variation, tool call parameter differences, additional clarification steps, different error handling paths, varying output verbosity. Without a classification rubric distinguishing "functional" (prompt-induced behavior change) from "non-functional" (LLM randomness), two evaluators can legitimately reach opposite conclusions about whether SC2 passes. The SC2 automation script (mentioned in iteration 1 report) can compare trajectories but cannot classify differences without rules. — Quote: line 485: "轨迹一致性 ≥ 90%（容差：步骤顺序因 LLM 生成随机性导致的非功能性差异）视为通过". — What must improve: Provide a classification rubric with 3-5 examples of functional vs. non-functional differences and a decision rule (e.g., "any difference in tool call sequence is functional; differences only in narrative text within a step are non-functional").

3. **[blindspot] No sequencing risk between content trimming and frontmatter restructuring**: The proposal treats content trimming and frontmatter restructuring as parallel work streams, but they both modify the same 21 prompt template files. Content trimming modifies the template body; frontmatter restructuring modifies the metadata header. If content trimming is completed first and then frontmatter restructuring begins, the snapshot checklist (SC1 baseline) must be re-created because the files have changed. If frontmatter restructuring is completed first, the PhaseSummary section migration adds new content to the template body that was not in the original content trimming plan. The proposal does not define the execution order or how the verification artifacts (snapshot checklists, baselines) interact between the two work streams. — Quote: line 56: "双线并行"; line 433: "在所有 21 个 prompt 模板正文中添加 `## PhaseSummary` 独立 section". — What must improve: Define the execution sequence (frontmatter first, then content trimming, or vice versa) and specify how SC-Pre baselines are invalidated/recreated when one work stream modifies files that the other work stream's baseline references.

4. **[blindspot] `TaskFile` is classified as `identity` in prompt templates but is not a unique identifier**: The validation table (line 328) lists "TaskID, TaskFile" under identity for prompt templates. `TaskID` is indeed a unique instance identifier. But `TaskFile` is a file path input — multiple task instances can reference the same task file. The grouping judgment rule defines identity as "唯一标识模板实例的字段——缺少该字段导致模板无法定位具体实例" (line 63). A missing `TaskFile` does not prevent template instantiation (the template can still load without it — the field is populated at render time). This misclassification could cause validation errors if `TaskFile` is validated against a struct where it has different semantics than `TaskID`. — Quote: line 63: "identity：唯一标识模板实例的字段——缺少该字段导致模板无法定位具体实例"; line 328: "identity 分组: TaskID, TaskFile". — What must improve: Either move `TaskFile` to the `context` group (it provides execution context, not identity) or refine the identity definition to include "fields required for template rendering initialization."

5. **[blindspot] No cost-benefit analysis for the frontmatter restructuring work stream**: Content trimming has a clear ROI: remove ~190 lines of waste, save 8K-22K tokens/day. Frontmatter restructuring touches 41 templates + Go parser + tests for an estimated 2 days of work, but the proposal states "frontmatter 重构不改变模板运行期行为，仅改善机器校验能力和人类可读性" (line 45). The benefit is qualitative ("改善") while the cost is quantified (2 days + ongoing maintenance of grouped format). The proposal does not identify a specific pain point that the current flat variables list causes — no bug reports, no developer confusion incidents, no failed validations. Without a quantified or evidenced problem, the frontmatter restructuring is a nice-to-have attached to a must-do (content trimming). — Quote: line 45: "frontmatter 重构不改变模板运行期行为，仅改善机器校验能力和人类可读性"; line 373: "Frontmatter 重构部分...总计约 2 天". — What must improve: Either (a) quantify the benefit of frontmatter restructuring (e.g., "validateMetadataVariables currently cannot distinguish identity fields from content fields, leading to false-positive validation errors on X templates") or (b) separate it into an independent proposal with its own cost-benefit analysis.

6. **[blindspot] The "三类指令" classification framework is unvalidated**: The instruction classification (lines 290-300) distinguishes positive instructions (A), negative constraints (B), and behavioral demonstrations (C), each with different retention policies. This framework is the methodological foundation for the entire content trimming approach — it determines what gets kept and what gets deleted. However, the classification is proposed without empirical validation. The proposal asserts that "C类" (behavioral demonstrations) can be reduced to "1 个代表性示例" per principle without losing effectiveness, but provides no evidence that LLM behavior is equivalent with one example vs. multiple examples. The "方法论依据" column (line 300) cites "作用于 LLM 的示范学习（few-shot）路径" but the few-shot learning literature consistently shows that the number of examples affects output quality. — Quote: line 300: "作用于 LLM 的示范学习（few-shot）路径，与指令路径正交；全部删除则失去该调节手段"; line 135: "每原则保留 1 个代表性示例（视觉分隔功能）+ 压缩边界说明为 1 行概括". — What must improve: Acknowledge the risk that reducing from multiple examples to one may degrade few-shot effectiveness, and make the per-principle example count an SC1-evaluable parameter rather than a fixed design decision.
