## ATTACKS

1. **Feasibility (6): Unverified foundational assumption** — The entire design depends on Claude Code executing Stop hooks sequentially in array order. The proposal acknowledges this as a "Pre-condition" and rates it M/H in the risk table, but does not confirm it works. The fallback ("single combined hook") would collapse the SoC argument and invalidate the Innovation Highlights. **What must improve**: Either verify the assumption before proposing (document test results), or design the proposal to work without this assumption (e.g., a single hook with conditional stages, closer to the Jenkins `post` model).

2. **Requirements Completeness (4): Mode detection has no failure scenario** — The mode detection mechanism relies on `proposal.md` existence: "exists = quick mode; absent = full pipeline." No scenario covers what happens when `proposal.md` exists but is empty, corrupt, or leftover from a previous quick-mode run. A stale `proposal.md` in a full pipeline would incorrectly trigger quick mode, causing an unwanted `proposal.md` status update. **What must improve**: Add a scenario for "proposal.md exists but context does not match quick mode" and specify how the command validates the mode, not just detects it.

3. **Logical Consistency (10): Singleton constraint not in Constraints section** — The "Multiple active features" scenario states "Reads active feature from `.forge/config.yaml` (singleton). At most one feature per session" but this constraint is not listed in the Constraints & Dependencies section. A reader building the solution from the constraints section alone would miss this. **What must improve**: Add "Feature singleton: at most one active feature per session (enforced by `.forge/config.yaml`)" to the Constraints & Dependencies list.

4. **Risk Assessment (8): Unjustified likelihood ratings** — Risk ratings lack justification. "Double-commit on edge case" at L/L — what edge case? "Git push fails (network, auth)" at M/M — why M likelihood? "Sequential hook execution assumption wrong" at M — if this is the foundational design assumption, M seems understated; it should be H or explicitly justified as M. **What must improve**: Add one-sentence justifications for each likelihood/impact rating. For example: "M likelihood — network interruptions are occasional in development environments."

5. **Success Criteria (9): Fragile grep-based verification** — Criterion 6 (`grep -c "status.*completed" quick.md` returns 0) and criterion 7 (`grep -c "gitPush|git push" run-tasks.md` returns 0) use regex patterns that can match in comments, documentation strings, or other non-transition contexts. These criteria test the absence of a string, not the absence of the functionality. **What must improve**: Replace grep-based criteria with structural tests (e.g., "quick.md does not contain a `complete-if-done` or status-transition step in its instruction sequence") or functional tests (e.g., "running `/quick` on a completed feature does not perform status transition within the skill execution").

6. **Solution Clarity (2): CLI invocation interface unspecified** — The command is named `forge feature complete-if-done` but no arguments or flags are shown. Does it auto-discover the feature directory from `.forge/config.yaml`? Does it accept `--dir`? The "Constraints & Dependencies" section mentions "Stop hooks receive `stop_hook_active` in stdin JSON" but the proposal does not specify whether the command reads stdin or discovers context automatically. **What must improve**: Add the full CLI interface, e.g., `forge feature complete-if-done` (no arguments, auto-discovers feature directory from `.forge/config.yaml`).

7. **Industry Benchmarking (3): Single-domain inspiration, no published patterns** — All industry references are CI/CD tools. No published design pattern name is cited (e.g., "pipeline stage gating," "post-condition pattern," "saga pattern"). No other domains are explored. The cross-domain inspiration is limited to one domain. **What must improve**: Cite at least one published pattern name from an authoritative source, and explore at least one additional domain (e.g., database transaction commit protocols for the atomicity requirement).
