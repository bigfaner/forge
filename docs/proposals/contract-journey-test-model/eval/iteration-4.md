## ATTACKS

1. **Logical Consistency**: "认证/授权契约" NFR is an orphan requirement -- quoted: "Contract 的 Preconditions 必须包含 auth 状态声明...gen-test-scripts 生成的 Setup 代码注入对应 auth 策略" -- but no success criterion verifies auth Contract generation. Add a criterion like "Web-UI Journey 的 Contract 中 Preconditions 包含 auth 状态声明，gen-test-scripts 生成的 Setup 包含对应 auth 策略代码" to close the gap.

2. **Logical Consistency**: "分批生成" NFR describes auto-split at "15 Contracts or 50k tokens" but no success criterion tests this behavior. If the batch-split logic breaks, no test will catch it. Add a criterion like "生成超过 15 个 Contract 的 Journey 自动拆分为多批，每批独立完成且结果与不分批生成一致" to verify the mechanism.

3. **Industry Benchmarking**: Multi-axis comparison table uses single-word ratings (High/Medium/Low) without scoring or weighting. Quoted: "实现成本: 高（一次性）, 维护开销: 低, 学习曲线: 中" -- these are self-assessed assertions without evidence. The Journey-Driven approach's "Workflow Alignment: Excellent" and "Contract Verification: Excellent" ratings are unsubstantiated. Provide quantitative backing (e.g., "Journey-Driven maps 100% of user stories to test files vs 40% for current model").

4. **Success Criteria**: "通过六维度完整性校验（必选维度非空，语义描述符不含 regex 语法）" -- the "不含 regex 语法" check is not formally defined. What counts as regex syntax? Escaped characters? Character classes in semantic descriptions? This criterion needs a precise definition of what constitutes a regex violation to be truly testable.

5. **Risk Assessment**: Fact Table staleness is not identified as a risk. The bootstrap strategy for verify relies on "126+ 已知正确的输出->Contract 映射" but if the codebase evolves significantly, these ground-truth mappings may become stale. The proposal assumes Fact Table is self-maintaining but no risk addresses the scenario where Fact Table snapshots diverge from current code behavior.

6. **Solution Clarity**: The `forge test verify` CLI output format is described at the criterion level but no example output is shown in the proposal. Quoted from success criterion: "报告受影响的 Contract（列出 Contract 文件路径和断裂维度）" -- but the actual terminal output format (tabular? JSON? plain text?) is unspecified. Add an example verify report output to match the quality of the Contract file example.

7. **Problem Definition**: Urgency is directional but not quantified. Quoted: "扩展成本线性增长" and "契约回归问题已在 task-pipeline-hardening 中暴露（11 个契约修复点）" -- but no estimate of how many regressions occur per month, or measurable productivity impact of the current model. Quantify the cost of delay (e.g., "each new language addition costs ~X engineer-days of forge-cli modification").

8. **Requirements Completeness**: No scenario covers the tag promotion lifecycle (feature -> regression) end-to-end, and no scenario covers the batch-split auto-triggering. The 7 scenarios cover main Journey types but not the operational mechanisms (promotion, batching) that are core to the solution.
