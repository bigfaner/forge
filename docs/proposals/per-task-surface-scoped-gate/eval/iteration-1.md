## Phase 1: Reasoning Audit

### Argument Chain Trace

**Problem -> Solution**: Problem: `RunGate()` ignores surface-key, runs all gates. Solution: (1) `RunGate()` resolves prefixed recipes by key, (2) surface rules generate prefixed gate recipe stubs. Chain holds: if prefixed recipes exist and `RunGate()` uses them, backend tasks skip frontend gates. **Gap**: Solution assumes `mixed.just` prefixed recipes (`backend-compile`) already work correctly — the proposal doesn't verify this, only that they exist.

**Solution -> Evidence**: The `resolveRecipe()` pattern is proven in `quality_gate_lifecycle.go`. `mixed.just` already has prefixed recipes. Code-level root cause is precise (`ResolveScope()` probes parameter mode, not prefixed mode). Chain is strong for the technical mechanism.

**Evidence -> SC**: SC-1 directly tests the core mechanism. SC-3 tests the guard condition. SC-4 tests surface rule completeness. SC-5 tests regression. Chain from evidence to SC is solid for the happy path.

**Self-contradiction check**: 
- Solution says "复用 `resolveRecipe()` 的模式" but Innovation Highlights correctly identifies the dispatcher semantics differ (fan-out vs select). No contradiction, but the reuse claim is now properly qualified.
- SC-2 says "行为与改动前完全一致" for single-surface projects, but In-scope item 2 adds prefixed recipes to ALL five surface rules. If a single-surface project has `api.md` as its surface rule, `init-justfile` will generate `api-compile` alongside `compile`. When `surface-key: backend` maps to `api` type, `RunGate()` will find `api-compile` and use it — changing behavior. **Contradiction**: SC-2 claims unchanged behavior, but the solution itself changes it by adding prefixed recipes to all surface rules.

### SC Consistency Deep-Dive

**Cluster 1: RunGate() behavior (SC-1, SC-2, SC-3)**
- SC-1: `scope=backend` -> `just backend-compile` (prefixed exists)
- SC-2: single-surface project -> unchanged behavior
- SC-3: `scope=""` -> unchanged behavior
- **Intra-group check**: SC-1 and SC-3 are satisfiable (guard `scope != ""`). SC-1 and SC-2 are **ambiguous**: a single-surface project with `surface-key` set to its sole surface will have prefixed recipes generated by the new surface rules. `RunGate()` will find and use them, changing the execution path. SC-2 says "行为与改动前完全一致" but the observable behavior (which recipe names are invoked) changes. Whether this matters semantically (prefixed recipe should produce same result as generic) is not stated. Marked as: **ambiguous — requires author clarification**.

**Cluster 2: Surface rule completeness (SC-4)**
- SC-4: all 5 files have stub recipes and Recipe Invocation Contract entries
- Satisfiable independently. No intra-group conflict.

**Cluster 3: Test regression (SC-5)**
- `go test ./internal/cmd/task/...` and `go test ./pkg/just/...`
- Satisfiable independently. No conflict.

**Cross-cluster**: SC-4 (all surface rules get prefixed stubs) interacts with SC-2 (single-surface unchanged). Adding prefixed recipes to all surface rules means even single-surface projects will have prefixed recipes. Whether SC-2 considers this "unchanged" depends on whether "unchanged" means "same output" or "same recipe invocation". **Ambiguous**.

## Phase 3: Blindspot Hunt

1. **[blindspot]** `ResolveScope()` 与 `resolvePrefixedRecipe()` 的共存架构未定义 — quote: "当任务有 surface-key（如 `backend`）时，优先尝试 prefixed recipe" — 提案引入了新的 `resolvePrefixedRecipe()` 但没有说明它与现有的 `ResolveScope()` 是什么关系。是替代？是补充？是串联调用？如果在 `RunGate()` 中两者都存在，它们作用于同一个 `scope` 参数但逻辑不同（一个探测参数模式，一个探测前缀模式），是否会产生冲突？这不是单一维度的问题——它横跨 D2（Solution Clarity 缺少集成架构）、D10（两个 scope 解析机制并存，未说明交互方式）。新引入的伪代码使这个问题更加显眼。

2. **[blindspot]** Surface rule 改动对单 surface 项目的副作用未分析 — quote: "api.md、web.md、cli.md、tui.md、mobile.md 各增加 `<key>-compile`/`<key>-fmt`/`<key>-lint`/`<key>-unit-test` 的 stub recipe" — 这意味着即使单 surface 项目（如只有 `api` 的项目）也会在 justfile 中生成 `api-compile`、`api-lint` 等 prefixed recipe。如果该项目的任务带有 `surface-key: backend`（因为用户在 config.yaml 中定义了 `backend=api`），`RunGate()` 会找到 `backend-compile`（如果 key 是 `backend`）或找不到（如果 key 是其他值）。这改变了单 surface 项目的 justfile 内容，而 SC-2 声称单 surface 项目行为不变。但 "行为不变" 的定义不清晰——如果 justfile 多了几个 recipe 但实际执行路径相同，算不算 "行为不变"？提案没有分析这个边界。

3. **[blindspot]** `NormalizeSurfaceKey` 字符集约束的验证性断言 — quote: "输出必须限定为 `[a-z][a-z0-9-]*`" — 这个约束被作为前提条件陈述，但提案没有验证当前 `NormalizeSurfaceKey()` 实现是否真的满足这个约束。如果 `NormalizeSurfaceKey()` 实际上允许大写字母或下划线，那 `HasRecipe()` 探测会产生无效的 recipe 名称，prefixed resolution 会静默失败（回退到 generic），用户不会收到任何提示。这不是一个约束声明就能解决的问题——它需要代码层面的验证或运行时断言。

## Attacks

1. **Industry Benchmarking**: 行业参考仍然过于薄弱 — quote: "Maven 的 `-pl module`，Nx 的 `--projects=`" — 仍然只有一句话提及两个工具名。Iteration 0 的 baseline report 已经指出这个问题，iteration 1 完全没有改进。需要补充至少 2-3 个具体的行业方案（Bazel affected targets、Turborepo dependency graph、Gradle incremental builds），分析其 scoped validation 策略与本提案的异同。

2. **Industry Benchmarking**: 替代方案仍然不够多元化 — quote: "仅改 RunGate()" — 这是所选方案的子集，不是真正独立的替代。Iteration 0 baseline report 已指出，iteration 1 未改动。需要增加至少一个行业验证过的替代方案（如基于依赖图的 affected detection）。

3. **Solution Clarity**: `resolvePrefixedRecipe()` 与现有 `ResolveScope()` 的集成架构未定义 — quote: 伪代码 `resolvePrefixedRecipe()` 函数定义 — 新增的伪代码解决了"是否做什么"的问题，但没有解决"放在哪里"的问题。`RunGate()` 中已有 `ResolveScope()`，新的 `resolvePrefixedRecipe()` 是替代它、在它之前调用、还是在它之后调用？两者作用于同一个 `scope` 参数但逻辑完全不同。需要增加集成伪代码或架构说明。

4. **Solution Clarity**: SC-2 "行为与改动前完全一致" 仍然模糊 — quote: "单 surface 项目中，任务提交行为与改动前完全一致" — Iteration 0 baseline report 已指出，iteration 1 未修改。需要具体化：哪些行为一致？recipe 调用名称一致？执行结果一致？退出码一致？

5. **Success Criteria**: Scenario 6（prefixed recipe 失败模式）没有对应的成功标准 — quote: "onFail 回调收到的 step name 为 `backend-compile`（含 surface 上下文），可直接映射到对应 surface 的错误处理路径" — 这个行为描述是需求（scenario 6）但不是可验证的标准。需要增加 SC 条目，如 "prefixed recipe 执行失败时，错误信息包含 surface key 上下文"。

6. **Feasibility**: `NormalizeSurfaceKey` 字符集约束未经验证 — quote: "输出必须限定为 `[a-z][a-z0-9-]*`" — 用"必须"陈述了一个对现有代码的要求，但没有验证当前实现是否满足。如果 `NormalizeSurfaceKey()` 实际允许其他字符，prefixed recipe 名称会无效，导致静默回退到 generic。需要在 Dependency Readiness 中验证或添加运行时断言。

7. **Risk Assessment**: 部署中间态的 Likelihood 评级偏低 — quote: "Prefixed recipe 不存在时回退到 generic，但 generic recipe 仍跑全量 | L | M" — 所有现有项目在 CLI 升级后、justfile 重新生成前都处于这个中间态。Likelihood 应为 H（初始阶段），随时间降低。

8. **[blindspot]**: `ResolveScope()` 与 `resolvePrefixedRecipe()` 共存架构缺失 — quote: 伪代码中 `resolvePrefixedRecipe()` 是独立函数 — 提案引入新解析机制但未说明与现有机制的关系。`RunGate()` 当前使用 `ResolveScope()` 做参数模式探测，新函数做前缀模式探测。两者是否都需要？`ResolveScope()` 是否应被废弃？如果共存，调用顺序是什么？这不是 D2 单独的问题——它影响 D10（逻辑一致性：两个机制作用于同一参数）和 D6（可行性：共享函数的变更范围）。新增的伪代码使这个结构性空白更加显眼。
