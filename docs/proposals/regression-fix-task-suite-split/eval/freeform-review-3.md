# Freeform Expert Review: Regression Fix Task Suite Split

## Background Assessment

This proposal addresses a real and well-documented operational failure: the quality-gate hook creates a single fix task covering all regression test failures, and when failures span multiple test suites (20+ failures across 4 suites, per the evidence), the agent stalls. The root cause analysis in `gotcha-fix-task-broad-scope.md` is thorough and correctly identifies two layers: task scope too broad (Layer 1) and inability to distinguish pre-existing failures from new ones (Layer 2).

The core approach is sound in principle: group regression test output by test file path, create one fix task per test file, and reduce each agent's context to a single file's failures. The proposal correctly identifies a prerequisite bug (SourceTaskID not set in the struct field, making the cap ineffective) and introduces `extractFileLineMap` as a richer replacement for the flat `extractSourceFiles`. The phased approach (Phase 0: improve description info, Phase 1: split by test file) is pragmatic -- testing whether information insufficiency alone causes the stall before investing in the split logic.

However, the proposal carries several risks and logical gaps that deserve scrutiny.

## Key Risks

风险：The proposal claims "fallback 到现有按目录分组的 `addFixTask` 行为" as a "零功能损失" safety net (Constraints & Dependencies, and Key Risks table), but this fallback has never been tested against the specific scenario where test file naming conventions fail. The current `addFixTask` itself relies on `extractSourceFiles` which uses `sourceFileRe` -- the same regex the proposal plans to reuse. If the regex fails to extract file paths from a particular test framework's output (e.g., Rust's `cargo test` output format differs substantially from Go's), the fallback to `addFixTask` produces the exact same monolithic task that causes the stall. The fallback is therefore not truly "zero functionality loss" -- it is "identical to the broken behavior we're trying to fix." There is no observability into when fallback triggers, so operators cannot distinguish "splitting worked correctly" from "splitting was silently skipped."

问题：The proposal states "后续迭代按需扩展其他语言（Python/JS-TS/Java/Ruby）的提取模式，扩展时所有模式同时执行，结果合并去重，不尝试识别输出使用的语言" (Constraints & Dependencies). Running all language patterns simultaneously on every output has an unbounded false-positive surface. Consider: a Go test output that mentions `test_helper.rb:42` in an error message string, or a Java stacktrace embedded in a test assertion message. The proposal explicitly chooses "宁可多包含" (prefer over-inclusion) as the association algorithm strategy, but this compounds with multi-language pattern execution -- every additional language pattern adds false positive risk. The proposal provides no mechanism to detect or bound false positive rates across pattern combinations.

问题：The `extractFileLineMap` association algorithm (Scope, item 4) states "一行匹配多个测试文件时，该行及其上下文归入所有匹配的测试文件." This is the primary source of output pollution between fix tasks. The proposal acknowledges this risk in the Key Risks table ("过度包含导致相邻测试文件输出互相污染") and the scenario analysis (stack trace referencing auxiliary test file), but dismisses it with "成本可控" -- acceptable cost. This dismissal is insufficiently justified. If `utils_test.go` appears in 15 different test files' stack traces, the `utils_test.go` fix task receives output from all 15 test files' contexts, effectively recreating the broad-scope problem for that specific task. The agent assigned to `utils_test.go` would then need to read output from 15 different failing tests to determine which are relevant, which is exactly the context overload the proposal aims to eliminate.

问题：The proposal claims to verify that "现有 compile/fmt/lint/unit-test 步骤的 fix task 创建不受影响" (Success Criteria), but the architectural change has a subtle coupling risk. The proposal plans to extract a shared `createFixTask` helper from `addSingleFixTask`, with both `addRegressionFixTasks` and `addSingleFixTask` calling this helper. Any shared helper carries the risk of behavioral drift: if the helper is modified to support regression-specific needs (e.g., different surface inference behavior for per-file tasks), the compile/fmt/lint/unit-test paths inherit those changes. The proposal does not specify test coverage requirements for the shared helper to guard against this.

问题：The SourceTaskID bug description in the proposal states "当前 `addSingleFixTask` 设置了 `Vars["SOURCE_TASK_ID"]`（模板渲染用）但从未设置 `opts.SourceTaskID`（结构体字段），导致 `countFixTasks` 的过滤条件 `SourceTaskID != ""` 永远为 false" (Proposed Solution). However, examining the actual `countFixTasks` implementation in `quality_gate.go` (lines 614-628), the function filters by title prefix `"fix " + step + ":"` and excludes terminal statuses -- it does NOT check `SourceTaskID` at all. The `gotcha-quality-gate-fix-task-loop.md` lesson mentions `countActiveFixTasks` with a `SourceTaskID != ""` filter, but the current code has `countFixTasks` which uses title-based matching. Either the lesson references an older version of the code, or the proposal is describing a bug that has already been partially fixed. This discrepancy needs resolution before implementation: if `countFixTasks` doesn't actually filter by SourceTaskID, then setting `opts.SourceTaskID = "quality-gate"` won't make the cap work differently. The proposal's "前置修复" may not have the effect claimed.

风险：The proposal removes `maxFixTasksPerStep` constraint for regression path and replaces it with a soft cap of 10 ("regression 专用软上限（10 个）"). The justification is "每个 fix task scope 已收窄到单文件，降低循环触发概率." But the proposal does not analyze the resource implications of 10 concurrent fix tasks. Each fix task is claimed by an agent, and agents may run concurrently. In a system where the dispatcher assigns tasks to subagents, 10 concurrent file-scoped tasks still means 10 agents reading different parts of the codebase simultaneously, potentially editing overlapping production code (since different test files often test the same production code). The proposal mentions RELATED_TASKS as a mitigation, but this is an informational hint to the agent, not a concurrency control mechanism. There is no analysis of the dispatcher's capacity or the merge conflict risk under concurrent multi-agent execution.

风险：The proposal's Key Scenarios section describes "栈 trace 引用辅助测试文件" as an acceptable false positive ("可接受：误报 task 执行后会发现无需修改，成本可控"). This underestimates the cost. A fix task claimed by an agent consumes a claim slot, execution time, and token budget. If the agent reads the output, determines no fix is needed, marks the task completed, and triggers another quality-gate cycle, the cost is one full agent execution cycle plus another quality-gate run. In a system where quality-gate is a stop hook, this extends session time linearly with the number of false-positive tasks. The "cost" is not just "agent reads and moves on" -- it's a full task lifecycle multiplied by false positives.

问题：The `extractFileLineMap` function signature is `func extractFileLineMap(output string) map[string][]string`. The proposal says it "输入原始 output 字符串，返回文件路径到提取行的映射" (Scope). But the proposal also says MVP only implements Go extraction patterns (`sourceFileRe` + `--- FAIL:` block parsing). The `sourceFileRe` regex (`([\w][\w./-]*\.\w{1,10})(?::\d+){1,2}`) matches any file path with a line number, not just test files. Combined with the `isTestFile` function (which identifies test files by `*_test.go`), the flow would be: extract all file paths from output, then group only those that match test file naming. But what about non-test file paths that appear in the output (e.g., `handler.go:42` in a stack trace)? These are extracted by `sourceFileRe` but filtered out by `isTestFile`. The proposal does not specify what happens to output lines that reference only non-test files -- are they lost? Or are they included in the fallback task? This gap could cause incomplete error information in the per-file fix tasks.

风险：The proposal explicitly states "不尝试识别输出使用的语言" (Constraints & Dependencies) for multi-language support. This means the system will never be able to optimize its parsing strategy based on the project's actual tech stack. As language patterns accumulate, the "all patterns simultaneously" approach will increasingly produce cross-language false positives. A Rust project's test output will be parsed with Go/Python/Java/Ruby patterns, each potentially matching false positives in stack traces, assertion messages, or file paths mentioned in test descriptions. Without language identification, there is no way to bound this growth.

## Improvement Suggestions

建议：Replace the unqualified "fallback to addFixTask" with an explicit fallback mode that logs a structured warning. Specifically, when `isTestFile` returns no matches (zero test files identified), log a message like `"WARNING: no test files identified via naming conventions, falling back to directory-grouped fix task. If this is unexpected, check test file naming against isTestFile rules."` This addresses the observability gap in the fallback path. After implementation, the log output would allow operators to distinguish between "splitting worked" and "splitting was silently skipped," enabling data-driven decisions about expanding language support.

建议：Add a false-positive containment strategy for the "one line matches multiple test files" scenario. Instead of assigning the line to all matching files, assign it only to the file that appears on the same line (i.e., `file_test.go:N` where the `:N` line number matches the current output line number). If no line number match is found, only then fall back to the "assign to all" strategy. This addresses the output pollution risk where `utils_test.go` appearing in 15 stack traces causes its fix task to contain output from all 15 tests. The proposal after this change would produce cleaner per-file task descriptions with less noise for the agent.

建议：Cap the false-positive task budget explicitly. The proposal should add to Success Criteria: "When a test file appears only in other tests' stack traces and has no direct `--- FAIL:` entries, it should not generate a standalone fix task." This addresses the risk of wasting agent execution cycles on false-positive tasks. Implementation: only create a fix task for a test file if the output contains at least one `--- FAIL:` line or a direct `file_test.go:N: error` pattern where the file is the primary match (not a stack trace reference). Files appearing only as secondary references get their lines folded into the fallback task.

建议：Resolve the SourceTaskID / countFixTasks discrepancy before implementation. The proposal should verify against the current code whether `countFixTasks` actually filters by `SourceTaskID`, or only by title prefix. If the current `countFixTasks` uses title-based matching (as the code shows), then the "前置修复" of setting `opts.SourceTaskID = "quality-gate"` is solving a different problem than what the lesson describes. The proposal should clarify: (a) what the actual current cap behavior is, (b) what the pre-requisite fix actually changes, and (c) whether the cap works correctly after the fix. This addresses the risk of implementing a fix that doesn't actually enable the cap.

建议：Define a concurrent execution budget for regression fix tasks. Instead of a soft cap of 10 with no analysis of dispatcher capacity, specify: "Regression fix tasks are created with a concurrency hint (e.g., max 3 should be actively claimed at any time). Tasks beyond the concurrency budget remain in `pending` status until active tasks complete." This addresses the risk of overwhelming the dispatcher with 10 concurrent fix tasks that may edit overlapping production code. The soft cap remains at 10 for total tasks created, but active execution is throttled.
