# Freeform Expert Review

## Background Assessment

这份提案试图解决 `forge task list` 在任务依赖可见性上的根本缺陷。目前列表按自然 ID 顺序平铺输出，开发者需要借助多个命令（`check-deps`、`query`）或直接阅读 `index.json` 才能理解任务的真实执行顺序。提案的核心技术方案是：将 Kahn's algorithm 引入列表排序逻辑，使默认输出呈现拓扑序，同时保留 `--sort id` 作为回退，并额外提供一个 `--tree` 的 TUI 树视图用于交互式探索。

作为一个长期与任务编排系统打交道的人，我理解这个痛点。当任务数量超过 10 个且依赖链涉及多层嵌套时，扁平列表几乎毫无价值。提案引用了现有的 `Task.Dependencies []string` 字段作为数据源，声称"无需 schema 变更"——这是正确的判断，也是该提案技术上轻量的根本原因。提案将自身定位为"增量改进"，不阻塞现有工作流，urgency 标记为"低"，这些定位是诚实的。

但作为熟悉 autogen 流水线和 index.json 生成机制的人，我在阅读过程中产生了一些关于依赖链完整性、索引稳定性、以及 scope 边界的担忧。以下展开。

## Key Risks

问题：提案对拓扑排序的输入数据质量缺乏质疑，默认"依赖信息足够构建完整 DAG"。

> "依赖数据已就位（`Task.Dependencies` 字段在所有 task .md 文件中）" — Section "Dependency Readiness"

这正是需要审慎挑战的假设。在 autogen 框架中，task .md 文件由 AI 生成，`Dependencies` 字段的准确性和完备性依赖于 prompt 质量和后处理逻辑。如果某个任务的依赖声明了 `1.x` 但展开后指向的具体任务 ID 并不存在，或者某个任务的依赖遗漏了某个实际必要的前置任务，拓扑排序的结果将表面正确（按依赖序排列）但实际误导（依赖关系本身不完整）。提案对 `[missing: <id>]` 标记的处理是面向"依赖指向不存在 ID"的情况，但没有讨论依赖数据本身的语义正确性。一个数据质量不可控的 DAG 经过拓扑排序后仍然是不可靠的 DAG。

风险：TUI 树视图与默认拓扑排序表的 scope 边界模糊，可能导致其中一项因复杂度而被放弃从而影响整体交付。

> "拓扑排序表格解决了 80% 的查看需求，TUI 树是额外的交互增强。用户明确选择两者都要" — Section "Assumptions Challenged", 第三行

作为经历过"两者都要"最终两者都延误的人，我需要指出：拓扑排序表格和 TUI 树视图分别依赖不同的技术栈——前者是纯计算逻辑（Kahn's algorithm + 表格渲染），后者依赖第三方 Go TUI 库（`bubbletea` 或 `gocui`）。两者的实现难度、测试覆盖要求、以及跨环境兼容性测试量完全不同。TUI 树在 SSH、CI 环境、screen/tmux 下的渲染行为需要大量调优。提案将两者的 timeline 分别估算为"1 天"和"2-3 天"，但这个估算没有考虑 TUI 库引入后的以下隐藏成本：依赖版本管理、安全审计（新依赖的 CVE 风险）、以及 bubbletea 在 darwin/linux/windows 跨平台下的终端能力检测。如果 TUI 实现过程中发现与某个常用终端模拟器不兼容，排期将不可预测地延长。

风险：非 TTY 环境下自动回退到 ID 排序的策略造成了排序结果的不一致性，可能引入管道处理错误。

> "非 TTY（管道）环境仍按 ID 排序，仅 TTY 启用拓扑排序" — Section "Key Risks", Mitigation 列

这在技术面上是一个合理的保守策略（向后兼容），但从任务编排的视角看，它制造了一个危险的语义陷阱：同一个命令 `forge task list` 在终端和管道中产生不同的排序结果。如果开发者编写脚本 `forge task list | head -n 5` 来获取"前 5 个需要执行的任务"，在 TTY 下测试时看到的是拓扑序的前 5 个，但放入 CI 脚本或 cron 后输出的是 ID 序的前 5 个——两者几乎没有语义上的对应关系。这本质上违反了"同一命令同一语义"的 Unix 哲学原则。提案对此没有给出任何警告机制或文档化策略。

问题：通配符依赖（`1.x`）的展开规则仅被提及而未定义，这是实现中的关键模糊点。

> "通配符依赖（`1.x`）展开为 1.1, 1.2, 1.3..." — Section "Edge cases"

通配符依赖的展开在数据模型层面并非无代价的。当 `Task.Dependencies` 中包含 `"1.x"` 时，这意味着在构建 DAG 时需要实时查询 task 列表并做前缀匹配。这个操作的时间复杂度取决于实现方式——线性扫描所有 task 做字符串前缀匹配是 O(N) 的，对于 100 个任务不是问题。但关键问题是：展开结果是否稳定？如果 task 列表在两次 `forge task list` 调用之间发生了变化（例如新的 task 1.4 被添加），同一个 `1.x` 依赖可能展开为不同的具体 ID 集合。这导致拓扑排序的结果在不同时间点不一致。此外，通配符匹配的边界没有定义：`1.x` 是否匹配 `10.x`？精确的匹配语义（glob? prefix? regex? semver range?）需要明确指定，否则不同实现者会产生不同的行为。

问题：Scope 中的"跨 feature 的全局 DAG 视图"被排除，但提案没有说明单 feature DAG 的局限性。

> "跨 feature 的全局 DAG 视图" — Section "Out of Scope"

在实际的 task orchestration 场景中，feature 之间的依赖关系是客观存在的——一个 feature 可能依赖另一个 feature 中某个任务的完成（例如：数据 schema 定义在 feature A，数据消费在 feature B）。提案选择将 scope 限定在单 feature 内是可以理解的（避免 scope creep），但需要承认：这个限制意味着拓扑排序的"全局执行顺序"视图是不完整的。开发者从单个 feature 的拓扑排序列表中看到的"正确的执行顺序"只在 feature 边界内成立。

## Improvement Suggestions

建议：在 proposal 中增加对 DAG 输入数据质量的验证机制定义。

Addresses: 依赖数据准确性的问题（第 2 段风险）

> 具体变更：在实现拓扑排序之前，增加一个 DAG 数据质量检查步骤。这个步骤应独立于排序逻辑本身，其职责包括：(1) 验证每个声明的 `Dependencies` 条目（包括展开后的通配符）确实指向存在的 task ID；(2) 收集所有 task 中存在但未被任何其他 task 引用的"孤岛"任务（无入边的节点），确认这些是设计预期而非遗漏；(3) 针对环检测的结果，不仅标记 `[cycle]`，还应提供环中节点的详细路径日志，帮助开发者定位环的成因。此外，建议在 `Assumptions Challenged` 部分将"依赖信息的语义正确性"作为一个独立的假设接受挑战，而不是假定数据一定正确。

建议：将 TUI 树视图拆分为独立的里程碑，与拓扑排序表格解耦。

Addresses: TUI scope 边界模糊的风险（第 3 段风险）

> 具体变更：将交付分两阶段——Phase 1 仅包含拓扑排序表格 + `--sort id` 回退 + 环/缺失依赖标记，Phase 2 为 TUI 树视图（`--tree`）。这确保了即使 TUI 库集成遇到问题，核心的拓扑排序能力仍可按期交付。在 Phase 1 的 success criteria 中不应包含 `--tree` 相关条目。Phase 2 的 timeline 估算应调整为"3-5 天"，包含 TUI 库选型、跨环境终端兼容性测试、以及依赖版本管理的开销。

建议：为非 TTY 管道场景设计显式的排序策略，而非简单地回退到 ID 排序。

Addresses: 管道输出与 TTY 输出排序不一致的风险（第 4 段风险）

> 具体变更：管道环境下仍输出拓扑排序，而不是回退到 ID 排序。这是一个更有原则的选择——让 `forge task list` 在 TTY 和管道下产生一致的拓扑排序结果。对于需要 ID 顺序的脚本，它们可以使用 `forge task list --sort id` 来显式声明需求。如果必须保留 ID 排序作为管道默认（为了 zero-breakage），则在 TTY 启用拓扑排序时应输出一行警告到 stderr，例如 `Warning: output order has changed from ID sort to topological sort. Use --sort id for the previous order.` 该警告仅在首次变更后出现（可通过环境变量或配置文件抑制）。这遵循了"surprise minimization"原则。

建议：明确定义通配符依赖的展开规则和边界条件。

Addresses: 通配符展开规则模糊的问题（第 5 段问题）

> 具体变更：在 Requirements Analysis 中增加一个子节专门定义通配符依赖规范。至少明确以下几点：(1) 匹配范围——`1.x` 是否匹配 `10`？建议定义为严格前缀匹配（`1.` 后接任意字符），而非全局匹配或正则；(2) 匹配结果的排序方式——多个匹配结果应按照版本号语义排序（1.1, 1.2, 1.3...）而非字符串排序（1.1, 1.10, 1.11...）; (3) 稳定性保证——一次 `forge task list` 调用内的多次展开结果应当一致，即使底层存储的 task 列表发生了并发写入（使用快照或缓存机制）；(4) 无匹配情况——通配符未匹配任何 task 时的行为，应输出 `[missing: 1.x]` 警告。
