# Proposal: Task Stage-Gates Auto-Generation

## Problem

Phase-level `.summary` and `.gate` tasks are currently generated by the `/breakdown-tasks` AI skill as markdown files. This approach has two gaps:

1. **Quick mode has no stage-gates**: `/quick-tasks` generates business tasks but never creates `.summary` or `.gate` tasks, leaving quick-mode features without phase-level verification.
2. **No deterministic guarantee**: Gate/summary generation depends on the AI agent correctly following skill instructions. If the agent skips or misformats these tasks, there's no recovery mechanism.

### Evidence

- `plugins/forge/skills/quick-tasks/SKILL.md` generates business tasks only â€” no gate/summary template references.
- `plugins/forge/skills/breakdown-tasks/SKILL.md` generates gate/summary but relies on AI compliance with templates.
- `forge task index` (in `pkg/task/build.go`) already auto-generates test tasks from embedded templates â€” the pattern for deterministic task generation exists.

### Urgency

Quick mode is actively used. Without stage-gates, quick-mode features have no phase-level quality checkpoints. As more features use quick mode, the gap widens.

## Proposed Solution

Extend `forge task index` to automatically detect numbered phases from existing business tasks and generate missing `.summary` and `.gate` task files for each phase. This follows the same pattern as test task auto-generation already in `build.go`.

### How It Works

After scanning `.md` files (step 5 in `BuildIndex`) and before generating test tasks (step 7):

1. **Detect phases**: Scan all task IDs to find distinct phase numbers (e.g., tasks `1.1`, `1.2`, `2.1` yield phases `1` and `2`). Count business tasks per phase.
2. **Threshold check**: Only generate gate/summary for phases with **>=2 business tasks**. Single-task phases lack sufficient complexity to justify the overhead.
3. **Check existence**: For each qualifying phase, check if `<N>.summary.md` and `<N>.gate.md` already exist in the tasks directory.
4. **Generate missing files**: For each missing file, generate from embedded template with:
   - `<N>.summary`: `depends_on` = all business tasks in phase N (IDs matching `<N>.<digit>` pattern, excluding `.summary` and `.gate`)
   - `<N>.gate`: `depends_on` = `["<N>.summary"]`, `breaking: true`
5. **Continue to index**: Generated files are picked up by the normal `.md` scanning loop.

The AI agent still generates business task files, then calls `forge task index --feature <slug>`. One command handles everything: stage-gate generation + index building.

### Innovation Highlights

This is a straightforward adoption of the existing test-task generation pattern (`generateTestTasks` in `build.go`) to a new task type. The innovation is recognizing that gate/summary generation belongs in the CLI's deterministic layer rather than the AI agent's probabilistic layer.

## Requirements Analysis

### Key Scenarios

- **Happy path**: AI generates 4 business tasks across 2 phases, calls `forge task index`. Two `.summary` and two `.gate` files are auto-generated with correct dependencies.
- **Idempotent re-run**: Re-running `forge task index` after a gate task is completed preserves the existing file and runtime state.
- **Quick mode**: Same behavior as full mode â€” any numbered phase with >=2 business tasks gets gate/summary.
- **Single-task phase**: Phase with only 1 business task â†’ no `.summary` or `.gate` generated (not enough complexity to justify overhead).
- **Phase with only test tasks**: Phases containing only T-test/T-quick tasks are excluded.

### Non-Functional Requirements

- Templates embedded via `go:embed` (same as test task templates)
- Generation time: negligible (file writes only)
- No external dependencies

### Constraints & Dependencies

- Depends on existing `gate-task.md` and `phase-summary-task.md` template structure
- Must not break existing `forge task index` behavior
- Must respect `--no-test` flag (stage-gates are NOT test tasks, so flag does not affect them)

## Alternatives & Industry Benchmarking

### Industry Solutions

Stage-Gate(R) is a standard product development methodology (Cooper, 1990). Each phase ends with a gate that evaluates quality before proceeding. Most CI/CD systems implement similar patterns (pipeline stages with gate approvals).

### Comparison Table

| Approach | Source | Pros | Cons | Verdict |
|----------|--------|------|------|---------|
| Do nothing | â€” | No dev cost | Quick mode has no gates; AI-dependent | Rejected: both gaps are real |
| Standalone `forge task stage-gates` command | â€” | Separation of concerns | Extra step for AI; more commands to learn | Rejected: unnecessary indirection |
| **Integrate into `forge task index`** | Existing pattern (test tasks) | One command; deterministic; same pattern as test tasks | Slightly increases `build.go` complexity | **Selected: minimal API surface, maximum reliability** |

## Feasibility Assessment

### Technical Feasibility

Fully feasible. The `build.go` file already has the exact pattern needed: detect â†’ check existence â†’ generate from embedded template â†’ merge with existing. Phase detection is a simple ID pattern scan (`strings.Split(id, ".")`).

### Resource & Timeline

Single developer, ~2-3 hours. Changes are localized to `pkg/task/build.go`, two new embedded template files, and template generation functions.

### Dependency Readiness

No external dependencies. Uses existing Go standard library and `go:embed`.

## Scope

### In Scope

- Phase detection from task IDs in `BuildIndex`
- Auto-generation of `<N>.summary.md` and `<N>.gate.md` from embedded templates
- Dependency wiring: `.summary` depends on all phase business tasks; `.gate` depends on `.summary`
- Idempotent: skip if file exists
- Update `/breakdown-tasks` SKILL.md to delegate gate/summary generation to `forge task index`
- Update `/quick-tasks` SKILL.md to call `forge task index` for stage-gate generation
- Version bump

### Out of Scope

- Modifying existing business task dependencies (AI agent's responsibility)
- Gate/summary generation for T-test or T-quick phases
- Overwriting existing `.summary` or `.gate` files
- New standalone CLI command

## Key Risks

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| Phase detection misidentifies tasks (e.g., `T-test-1` parsed as phase 1) | L | M | Filter: only match `<digit>.<digit>` format, exclude T-test/T-quick, count >=2 business tasks |
| Template content diverges from skill-authored gate/summary | M | L | Use the same templates from `breakdown-tasks/templates/` via `go:embed` |
| Dependency cycle if AI manually sets deps that conflict with auto-generated ones | L | M | `forge task validate-index` already checks for cycles |
