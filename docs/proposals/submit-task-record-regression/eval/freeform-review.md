# Freeform Expert Review: Submit-Task Record Regression Proposal

**Reviewer Profile:** Test Infrastructure Architect (golden/snapshot regression for CLI tools)
**Date:** 2026-05-24
**Document:** `docs/proposals/submit-task-record-regression/proposal.md`

---

## Section 1: Background Assessment

This proposal addresses a concrete gap in the forge ecosystem: the submit-task skill produces per-type records through a multi-step rendering pipeline (task type -> template selection -> field population -> markdown output), yet no systematic validation exists to confirm that each step of this pipeline remains correct as the system evolves. The rendering pipeline spans two distinct codebases -- the submit-task skill (TypeScript/YAML, selects record-format template and populates JSON) and the forge-cli (Go, validates RecordData and renders markdown via `RenderRecord`) -- which makes schema drift between these layers a real and growing risk.

The proposed approach is orthodox golden-dataset testing: extract historical records as fixtures, organize them by task type, and run them through `validateRecordData` + `RenderRecord` to detect divergence. The 10 identified feature sources yield a healthy corpus (19 coding.feature, 16 coding.enhancement, 20 coding.refactor, etc.), and the proposal correctly identifies that this data already exists, making extraction cheap. The stated goal of 2-3 records per type across 12 active types targets roughly 24-36 fixtures -- a manageable corpus.

A key architectural observation: the proposal's title suggests testing "submit-task record construction," but the actual test target is the Go CLI's rendering pipeline (`validateRecordData` -> `RenderRecord`), not the submit-task skill's template selection and JSON population logic. The submit-task skill itself (the TypeScript/YAML code in `plugins/forge/skills/submit-task/`) is not under test. This is a meaningful scoping decision that the proposal does not explicitly call out.

## Section 2: Key Risk Identification

风险：The proposal tests only one half of the schema drift problem. The stated test surface is "validateRecordData 校验 + markdown 渲染对比" (proposal.md, line 26), which validates the Go struct -> rendered markdown direction. But the primary risk identified in the Evidence section is the reverse direction: "record-format 模板中的示例 JSON 可能与 Go 端实际接受的 schema 存在偏差" (line 17). The record-format templates (e.g., `record-format-coding.md` lines 8-20) show JSON examples that guide the LLM agent in populating RecordData fields. A golden dataset test that feeds historical JSON into Go's `validateRecordData` will never catch a mismatch between what the record-format template *tells the agent to produce* and what the Go struct actually accepts, because the historical records were already successfully validated when created. The test would only catch future Go-side regressions, not the current template-documentation-to-Go-struct gap.

问题：The fixture selection criteria lack temporal awareness. The proposal states "每种 task type 选 2-3 条代表性记录" (line 129) but does not define what makes a record "representative." During the v3.0.0 refactor, the RecordData struct and rendering templates underwent multiple iterations. Records from `forge-architecture-simplification` (dated 05-22, line 106) and `forge-cli-clean-code` (dated 05-24, line 97) were generated against potentially different Go template versions. If fixture selection is purely random or based on convenience, a pre-refactor record might fail against the current rendering pipeline for benign reasons (intentional format changes), while a post-refactor record might pass but miss testing the specific template boundary that changed. The proposal should specify selection criteria based on template iteration boundaries -- e.g., at least one record from before and one from after each known template change.

风险：The `-update` flag pattern lacks a review gate. The proposal mentions "测试用 `-update` flag 模式，确认是有意变更则更新 fixture" (line 145), but this is a purely manual social process. In Go's standard golden-testing pattern, `go test -update` silently overwrites all `.golden` files. There is no diff review, no confirmation prompt, and no CI enforcement. A developer who runs `go test -update ./...` after a template change will automatically update every affected fixture, making it impossible to distinguish intentional format changes from regressions. The proposal's mitigation for this risk is "确认是有意变更则更新 fixture" -- but this is a statement of intent, not a mechanism.

问题：The `fix` (bare) vs `coding.fix` distinction is underspecified in the test plan. The proposal mentions both types in the coverage matrix (lines 117-118: "coding.fix" with 15 records and "fix (legacy bare)" with 8 records) and lists "fix" in scope (line 133: "12 种 task type：coding（5 子类型）、doc（5 子类型）、gate（1 种）、fix（legacy bare type）"). But examining `CategoryForType` in `category.go`, the bare string `"fix"` does not match any explicit case; it falls through to the `default` branch and returns `CategoryCoding`. This means `fix` and `coding.fix` dispatch to the same category and thus the same rendering template (`record-coding.md`). The proposal's claim of "明确区分两种 type 的 fixture，验证各自走正确的模板" (line 148) is misleading -- they walk the *same* template path. Testing them as genuinely separate dispatch paths would create redundant fixtures that exercise identical code. The real question is whether `validateRecordData` applies different validation rules to `fix` vs `coding.fix` (it does not -- both get `isCoding = true`), which the proposal does not analyze.

风险：The proposal claims "新增 fixture 时只需复制文件 + 加一行测试用例" (line 155) as a success criterion, but this understates the actual maintenance cost. Adding a fixture for a new task type requires: (1) extracting a representative historical record, (2) ensuring its JSON input passes `validateRecordData` against the current Go schema, (3) running it through `RenderRecord` to produce the expected markdown, (4) manually verifying the rendered markdown matches the historical record (which may differ due to template iteration), and (5) if they differ, deciding whether the fixture should match the old or new format. Step 4-5 is a judgment call that scales linearly with the number of new types and template changes. The "copy + one line" claim is accurate only if the new record was generated against the current template version, which is not guaranteed.

问题：The CI performance budget of "<30s" (line 45) is asserted without decomposition. With 12 task types x 2-3 records = 24-36 fixture files, each requiring `validateRecordData` + `RenderRecord` invocation, the test performs roughly 48-72 Go function calls plus file I/O for loading fixtures and comparing output. On a typical CI runner, this should comfortably fit within 5 seconds, not 30. The 30s budget suggests either the author anticipates the test will do something heavier (compile checks? subprocess invocation?) or the number is a rough placeholder. Without a baseline measurement or a stated breakdown of what contributes to the 30s, this requirement is not actionable for CI pipeline configuration.

风险：The proposal conflates two distinct testing objectives under one umbrella. The title is "Submit-Task 按任务类型构建 Record 的回归验证" (line 7), which implies testing the submit-task skill's type dispatch logic (task type -> record-format template selection). But the proposed implementation tests the Go CLI's rendering pipeline (`validateRecordData` + `RenderRecord`), which is downstream of submit-task. The submit-task skill's template selection is implemented in the TypeScript skill code (`plugins/forge/skills/submit-task/`), not in Go. A Go-level test cannot verify whether submit-task selected the correct record-format template for a given task type -- it can only verify that the JSON it receives renders correctly. This conflation could lead stakeholders to believe the submit-task skill itself is under test when it is not.

## Section 3: Improvement Suggestions

建议：Add a bidirectional schema validation layer. In addition to the Go-level golden tests, implement a lightweight script that extracts the JSON schema examples from each `record-format-*.md` template and attempts to unmarshal them into Go's `RecordData` struct. This directly addresses the stated goal of detecting "record-format 模板中的示例 JSON 可能与 Go 端实际接受的 schema 存在偏差" (line 17). This test would run as a separate Go test case per record-format template, reading the embedded JSON example and verifying it round-trips through `json.Unmarshal` -> `validateRecordData` without error. This addresses the schema drift risk from the template-to-Go direction that the current proposal only covers from the Go-to-rendered-markdown direction. The implementation would be approximately 20 lines of Go per template and could live alongside the golden dataset tests.

建议：Define explicit fixture selection criteria based on template change history. Replace the vague "2-3 条代表性记录" (line 129) with a concrete protocol: for each task type, select (a) the most recent record available (closest to current template version), (b) the oldest record available (greatest distance from current template), and (c) if the type has a known template change commit, one record from immediately before that change. This ensures the test corpus spans the widest possible format variation, increasing the probability of catching regression at template boundaries. This directly addresses the temporal-awareness gap in fixture selection and makes the "representative" criterion auditable.

建议：Implement fixture diff gating for the `-update` flag. Instead of relying on the standard Go pattern of silently overwriting `.golden` files, wrap the update mechanism to produce a diff file (e.g., `fixture_name.diff`) whenever a fixture changes. CI should then fail if any `.diff` files are present, requiring the developer to review and delete the diff file (or commit the updated fixture with the diff deleted). This transforms the `-update` flow from "run and forget" to "run, review diff, explicitly acknowledge." This addresses the review-gate risk by adding a mechanical enforcement layer on top of the social convention.

建议：Consolidate `fix` (bare) and `coding.fix` into a single test case with an alias assertion. Since `CategoryForType("fix")` and `CategoryForType("coding.fix")` both return `CategoryCoding` and dispatch to the same `RenderCodingRecord` path, there is no value in maintaining separate fixture files for each. Instead, add a focused test case that verifies `CategoryForType("fix") == CategoryForType("coding.fix") == CategoryCoding` and then use a single fixture for both. This reduces the fixture count by 2-3 files and eliminates the misleading impression that these are genuinely separate dispatch paths. The proposal's table should document this as an alias relationship rather than listing them as independent types.

建议：Decompose the <30s CI budget into a measurable specification. State the expected number of fixture files (e.g., "~30"), the expected number of function calls per fixture (2: validate + render), and a target per-fixture time budget (e.g., "<100ms including file I/O"). Run a quick baseline measurement against the current codebase and record it in the proposal. This makes the 30s requirement falsifiable and gives future maintainers a clear signal when the test corpus grows to the point where performance needs attention. With the current fixture count of ~30, the realistic budget should be well under 10 seconds on any modern CI runner.

建议：Rename or explicitly scope the proposal to reflect what is actually under test. If the test target is the Go CLI's rendering pipeline (not the submit-task skill), the title and problem statement should reflect this. For example, "Go CLI Record Rendering Pipeline Regression" would set accurate expectations. Alternatively, if the intent is to also test submit-task's template selection, the proposal should include a separate test layer that validates the skill's type-to-template mapping, which would require a different testing approach (likely fixture-driven validation of the skill's output before it reaches the Go CLI). Clarifying this scope prevents the stakeholder communication gap where the proposal promises end-to-end testing but delivers only the downstream half.
