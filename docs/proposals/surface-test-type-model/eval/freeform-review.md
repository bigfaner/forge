# Freeform Expert Review

**专家视角**: 测试类型语义精确性审查
**审查对象**: Surface-Specific Test Type Model（Surface 测试类型模型）

---

## Background Assessment

本提案试图解决一个真实的术语精确性问题：Forge 将所有自动生成的测试统一标记为"e2e 测试"，但实际上不同 Surface（CLI、TUI、API、Web、Mobile）的测试在执行模型、验证维度和语义定位上存在本质差异。提案的核心主张是取消"e2e"作为通用标签，代之以 Surface → Test Type 的一一映射：CLI 集成测试、终端集成测试、API 契约测试、Web 端到端测试、移动端 UI 测试。

提案自认为这不是创造新概念，而是"精确命名已有实践"——因为 gen-test-scripts 的 types/ 目录已经按 surface 分化了生成策略，只是这种分化从未被提升到概念层和命名层。提案的技术基础是 Forge 已有的 surface 分化基础设施（5 种生成策略、5 种编排规则、5 种 justfile 模板、5 种 surface 规则），因此可行性层面阻力较小。

提案的核心哲学论证位于"Assumptions Challenged"部分：它对"CLI 测试是 e2e"进行了归谬——如果 CLI 测试是端到端的，那它端到端测试了什么？答案只是一个子进程调用。由此得出结论：只有 Web surface 的测试才是"真正的端到端"。

这一哲学论证虽然有力，但恰恰是我审查的起点。一个分类体系的价值不在于它解决了多少命名问题，而在于它的分类标准是否自洽、映射是否精确、边界是否无歧义。

## Key Risks

提案建立了五个 Surface → Test Type 的映射关系，但这五条映射并非站在同一条分类逻辑之上。分类标准的混用会导致后续推理的不一致，而提案对这一问题没有意识。

风险："集成"一词在 CLI 和 TUI 两条映射中的语义模糊
> "CLI 集成测试：编译独立二进制，通过子进程调用，验证命令行参数解析、输出格式、退出码、错误处理。黑盒视角，不测试内部函数。" — 这里"集成"一词指什么？集成测试在行业中有两种主流理解：一是"集成多个模块"（integration of components），二是"集成外部系统"（integration with external systems）。CLI 测试编译独立二进制并通过子进程调用，既不是集成多个内部模块（它是黑盒的，根本不接触内部函数），也不是集成外部系统（它只是启动自己的二进制）。这里唯一合理的"集成"含义是"测试了二进制作为完整可执行体的集成行为"，即"编译 + 链接 + 运行"这一整体流程。但这是一种非常弱的集成语义——如果仅仅是"编译后的二进制能正确运行"，那每个非白盒的命令行测试都可以叫"集成测试"，这个术语就失去了区分价值。更精确的描述应该是"CLI 功能验证测试"或"CLI 行为测试"，因为它验证的是命令行接口的功能行为，而非组件之间的集成。

问题：TUI 的"半黑盒视角"标准未定义，且与 CLI 的"黑盒视角"边界不清
> "终端集成测试：编译独立二进制，通过 stdin pipe 模拟用户输入，验证终端渲染输出（ANSI 序列处理、布局、异步 Cmd 响应）。半黑盒视角。" — 提案为 TUI 标注了"半黑盒视角"而为 CLI 标注了"黑盒视角"，但从未定义判定标准。半黑盒与黑盒的区别通常在于：测试是否了解被测系统的内部结构或实现细节。TUI 测试通过 stdin pipe 模拟输入并检查终端输出——如果它不读取内部状态，这与 CLI 测试通过子进程调用验证 stdout/stderr 的方式在黑盒程度上没有本质区别。唯一的差异可能是 TUI 测试需要关心 ANSI 序列和布局渲染，这更接近于"渲染验证"，但这不等于"半黑盒"。如果"半黑盒"是指测试者需要了解终端渲染的某些内部机制（比如哪些 Cmd 会触发异步操作），那这个标准需要明确定义，否则读者无法判断为何 CLI 是黑盒而 TUI 不是。更深层的问题是：如果"半黑盒"没有精确定义，它就会成为一个标签式的分类——只是为了说"TUI 测试比 CLI 测试复杂"而加的修饰语，而不是一个可推理的分类维度。

风险："API 契约测试"与行业已有的契约测试概念（Pact、Spring Cloud Contract）产生语义冲突
> "API Contract Test | API 契约测试 | HTTP 状态码 + 响应体 + Header | HTTP 客户端" — 以及语义定义："Contract 即契约。" — 行业中"契约测试"（Contract Testing）有明确且特定的含义：它是指服务消费者与提供者之间关于 API 接口行为的约定的验证，典型框架是 Pact。其核心特征是：消费者定义期望（consumer contract），提供者验证自己是否满足这些期望（provider verification），且两者可以独立执行。而本提案中的"API 契约测试"实际行为是：启动 HTTP 服务器，发送请求，验证响应符合 Contract 定义的六个维度。这更接近于"API 功能验证测试"或"API 接口测试"——它验证的是一个 API 是否按规范响应，而不是消费者-提供者之间的契约约定。将这种测试称为"契约测试"会让熟悉 Pact 或 Spring Cloud Contract 的工程师产生错误预期。提案说"Contract 即契约"——这里的 Contract 是 Forge 的 gen-contracts 生成的 Contract 文件，但这个 Contract 与行业中的 Consumer Contract / Provider Contract 是完全不同的概念。用"契约"这个术语来命名，是将 Forge 内部概念（Contract 文件）的名称直接投射到了测试分类学上，而没有考虑该术语在测试领域已有的精确含义。

问题：将"端到端"保留给 Web Surface 的论证不完整
> "Assumption Flip：如果 CLI 测试是 e2e，那它端到端测试了什么？答案是一个子进程调用，这不是'端到端'" 和 "Overturned: 只有 Web surface 的测试真正端到端" — 提案的归谬论证只考察了 CLI 这一个反例，就得出"只有 Web 真正端到端"的结论。但这个结论需要更强的支撑。考虑 API Surface 的测试：如果它启动了真实的 HTTP 服务器，请求经过路由层、中间件、业务逻辑、数据库，最后返回响应——这条链路从 HTTP 入口到数据库持久化，覆盖了完整的请求生命周期。从架构角度看，这不也是"端到端"的吗？提案将 API 测试定义为"契约测试"而非"端到端测试"，其隐含的假设是：端到端必须包含用户界面层。但这是一个需要显式声明的分类前提，而非不言自明的真理。如果端到端的定义是"从系统最外层入口到最底层存储"，那 API 测试也可以是端到端的；如果端到端的定义是"必须包含用户可见的界面交互"，那 Web 和 Mobile 都应该是端到端的，而提案却将 Mobile 定义为"UI 测试"而非"端到端测试"。这个不一致性说明提案对"端到端"的定义实际上是基于直觉而非基于明确的分类标准。

风险：Mobile Surface 被定义为"移动端 UI 测试"而非"端到端测试"，与 Web 的分类逻辑不一致
> "Mobile UI Test | 移动端 UI 测试 | UI 渲染 + 用户交互 | Maestro YAML / 手动验证" — 如果 Web 的测试被称为"端到端测试"是因为它通过浏览器模拟用户操作、验证 UI 渲染和交互逻辑，那么 Mobile 的测试通过 Maestro 驱动移动端 UI、验证渲染和交互，在逻辑上与 Web 测试是同构的。唯一的差异是执行模型（Playwright vs Maestro），但执行模型不应该是决定测试类型的分类标准——提案自己的分类标准是 Surface。如果同一个分类维度下的两个成员（Web 和 Mobile）在分类逻辑上应该同构却被赋予不同的类型名称（"端到端" vs "UI"），那么分类体系的正交性就受到了损害。提案可能想表达的是：Mobile 测试覆盖不够完整（"Best-effort 模式，部分场景标记为 manual-only"），因此不宜称为"端到端"。但覆盖率不应该是分类命名的依据——一个只覆盖了一半用户旅程的 Web 测试仍然叫"端到端测试"，只是覆盖率低而已。

问题：验证维度列表的粒度不一致
> CLI 的验证维度是"退出码 + stdout/stderr"，API 的验证维度是"HTTP 状态码 + 响应体 + Header"，Web 的验证维度是"UI 渲染 + 用户交互 + 状态流转" — 这三组验证维度的粒度不在同一个层次。CLI 和 API 的维度是具体的输出属性（退出码、标准输出、HTTP 状态码、响应体、Header），而 Web 的维度中"状态流转"是一个更高层的概念——它描述的不是某个具体的输出属性，而是多个步骤之间的状态变化。如果按统一粒度要求，CLI 的验证维度应该也包含"状态流转"类的概念（比如：多个命令之间的状态传递），或者 Web 的验证维度应该被分解为更具体的输出属性（DOM 状态、网络请求结果、页面路由变化等）。粒度不一致会导致不同 Surface 的验证维度无法横向比较，进而影响用户对不同测试类型覆盖范围的理解。

问题：分类体系实际上混合了两个分类维度——Surface 和验证视角（黑盒/半黑盒/白盒），但提案未承认这一点
> 提案以 Surface 为一级分类键，但语义定义中又引入了验证视角（"黑盒视角"、"半黑盒视角"）作为二级属性。问题在于：如果验证视角是独立于 Surface 的分类维度，那么理论上应该存在"CLI 白盒测试"或"API 灰盒测试"的可能性，但提案的映射表暗示每种 Surface 只有一种验证视角。如果验证视角不是独立维度而是 Surface 的固定属性，那么它不应该出现在语义定义中作为一个需要独立声明的概念——它应该是 Surface 定义的固有部分。当前的呈现方式让读者无法判断验证视角是一个自由维度还是一个固定属性。

建议：在提案中增加一段"分类标准声明"，明确定义本分类体系使用了哪些分类维度、它们之间的关系是正交的还是主从的，以及每个维度的取值范围。这将消除读者对分类逻辑自洽性的疑虑。

## Improvement Suggestions

建议：将"CLI 集成测试"重命名为"CLI 行为测试"（CLI Behavioral Test），并明确解释为何不使用"集成"一词
Addresses: "集成"一词在 CLI 和 TUI 两条映射中的语义模糊
> "集成测试"在测试行业中有特定的含义（多组件或多系统的集成验证），CLI 测试的行为（编译二进制、子进程调用、验证输出）不符合这一定义。将其重命名为"CLI 行为测试"更准确地描述了它做的事情——验证 CLI 接口的外部行为。如果保留"集成"一词，应在文档中显式定义此处的"集成"指什么（如"集成编译、链接、运行三个阶段的验证"），并与行业定义进行对比说明，以免读者产生错误联想。

建议：重新审视"API 契约测试"的命名，考虑使用"API 接口验证测试"或"API 规范验证测试"
Addresses: "API 契约测试"与行业已有的契约测试概念产生语义冲突
> 提案将 Forge 的 Contract 文件（由 gen-contracts 生成）直接等同于测试分类学中的"契约"概念，但两者在语义上有本质区别。Forge 的 Contract 是一种测试规格描述文件，而行业中的 Consumer Contract 是一种服务间接口约定。建议将测试类型命名基于测试行为而非内部文件名——API 测试验证的是"HTTP 接口是否按规范响应"，所以"API 接口验证测试"更精确。如果团队内部确实习惯将 Contract 文件称为"契约"，那么至少应在文档中增加一段术语消歧，明确说明此处的"契约"不等于 Pact 意义上的消费者契约。

建议：定义"端到端"的精确语义，要么统一 Web 和 Mobile 的命名，要么给出两者在分类逻辑上确实不同的充分理由
Addresses: Mobile Surface 与 Web 的分类逻辑不一致，以及"端到端"论证不完整
> 当前提案中 Web 叫"端到端测试"而 Mobile 叫"UI 测试"，但两者的执行模型（浏览器自动化 vs Maestro 自动化）和验证维度（UI 渲染 + 交互 + 状态 vs UI 渲染 + 交互）高度同构。建议两种处理方式之一：一是将 Mobile 也称为"Mobile 端到端测试"（Mobile E2E Test），承认两者在分类逻辑上是同构的，Mobile 的"Best-effort"特性是执行层面的限制而非分类层面的差异；二是在文档中显式定义"端到端测试"的充分必要条件（如：必须包含浏览器渲染引擎 + 网络层 + 持久化层的完整覆盖），然后论证 Mobile 测试为何不满足这些条件。当前的隐含假设（端到端 = 浏览器测试）不足以支撑分类结论。

建议：在"半黑盒视角"出现的地方，增加一段判定标准的定义，或者删除黑盒/半黑盒的标注
Addresses: TUI 的"半黑盒视角"标准未定义
> 如果黑盒/半黑盒/白盒的区分是本分类体系的一部分，那么它需要被精确定义——判定标准是什么？测试者是否需要了解实现细节？测试是否通过内部接口（而非公开接口）访问被测系统？如果这些标准无法精确定义，那么黑盒/半黑盒标注就不应该出现在提案中，因为它引入了一个无法推理的分类维度。删除后不会影响提案的核心价值（Surface → Test Type 映射），但会消除一个概念模糊点。

建议：统一验证维度的粒度，要么全部使用具体输出属性，要么全部使用高层概念
Addresses: 验证维度列表的粒度不一致
> 例如，可以统一到"输出属性"层面：CLI 的维度为"退出码、stdout、stderr"，API 的维度为"HTTP 状态码、响应体、响应头"，Web 的维度为"DOM 状态、网络请求、页面路由、视觉渲染"，TUI 的维度为"终端输出、ANSI 序列、布局状态"，Mobile 的维度为"UI 元素状态、手势响应、页面导航"。统一的粒度使得五种测试类型的验证范围可以横向比较，也有助于后续质量门报告的标准化。

建议：在提案中增加"可扩展性分析"一节，展示模型在新增 Surface 时的扩展行为
Addresses: 分类体系的可扩展性未经验证
> 当前提案的 Scope 仅覆盖五种已知 Surface，但一个分类体系的价值很大程度上取决于其可扩展性。建议增加一个"扩展性验证"段落，用 2-3 个假设的新 Surface（如 `desktop`、`embedded`、`lambda`）来演示分类逻辑的扩展过程：给定新 Surface 的执行模型和验证维度，如何推导出其 Test Type 名称？推导过程中是否需要引入新的分类维度？如果需要，说明当前的五元素映射只是特例而非一般模型。这个分析将大大增强提案的理论完备性。
