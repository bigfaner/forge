# Test Strategy Evaluation for Task Pipeline Hardening

基于软件测试组织理论的提案评估。前提：Forge 默认采用 TDD，单元测试是必须的。

## 评估结论

提案在问题诊断和架构修复上一流，但验证策略停留在"成功标准 + e2e 回归"的二元结构。需要补充三个层次：单元测试计划、并发/故障场景覆盖、契约测试机制。

## 按理论的评估发现

### 1. 测试金字塔 — 底部缺失

提案只覆盖了集成层（Success Criteria）和 e2e 层（回归），缺少单元测试层。

**关键遗漏**：

- `ValidateTransition(current, target, opts)` 是纯函数，状态空间有限（~6 状态 × 6 状态 × 若干选项），可穷举测试。TDD 开发，先写转换矩阵再实现。
- `countFixTasks` 的活跃/终态过滤逻辑是独立函数，应独立测试。
- Scorer JSON 解析的两层 fallback（JSON 优先 → 正则）是纯字符串处理，应单元测试。

### 2. 风险导向测试 — 测试强度未映射风险等级

提案的风险分析（严重性评级 + 失败级联）做得好，但 12 条 Success Criteria 全部同一粒度。高严重性缺陷应有更严格的测试：

| 风险等级 | 缺陷 | 建议测试强度 |
|---------|------|-------------|
| 高 + 级联 | SM-1+SM-3（终态绕过）| 穷举状态转换矩阵 + 负面用例 |
| 高 + 级联 | MA-1（并发双写）| 并发压力测试（多 goroutine 同时 claim） |
| 高 | QG-1+QG-2（fix-task 失效）| 端到端场景：gate 失败 → fix → restore → cap 锁死 |
| 中 | EP-1（scorer 解析）| 模糊测试：各种格式偏差的 scorer 输出 |

### 3. 静态分析层（Testing Trophy）— 全部在运行时解决

提案没有利用编译期/静态防御：

- `ValidateTransition` 的合法转换可以用 Go 类型系统编码（const + exhaustiveness check），非法转换编译期报错而非运行时
- `typeToTemplate` 映射缺失（TD-1）可通过 `go generate` 从模板目录同步映射表，compile-time 保证一致性
- Scorer 输出可用 JSON schema 验证，而非仅依赖两层容错解析

建议：不要求全部改为编译期检查（改动面太大），但应在任务拆分时标注哪些防御可静态化。

### 4. 契约测试 — 理念对了，机制缺失

#### 什么是契约测试

契约测试验证的是**组件间接口约定**，而非各自的内部实现。核心模型：

- **Provider（提供方）**：暴露接口的一方（API、CLI 命令、函数签名）
- **Consumer（消费方）**：调用接口的一方

契约测试验证：Provider 实际提供的接口，与 Consumer 期望的接口，是否一致。

#### 契约测试 vs 集成测试

契约测试属于集成测试大类，但与传统集成测试是**替代关系**，不是子集关系。两者解决同一问题（"组件 A 和 B 能不能正确协作"），但方式不同：

| | 传统集成测试 | 契约测试 |
|---|---|---|
| 是否需要双方同时运行 | 需要 | 不需要 |
| 测试环境要求 | 高（要搭建集成环境） | 低（各自独立验证） |
| 反馈速度 | 慢（要跑完两边） | 快（只跑一边） |
| 发现问题时机 | 集成阶段 | 开发阶段 |

契约测试不需要同时运行双方——Provider 和 Consumer 各自独立验证是否符合约定，然后比对。这就是为什么它比传统集成测试轻量。

#### 契约测试 vs E2E — 不同轴，非高低关系

契约测试不是"浅版 E2E"，两者沿不同维度纵深：

| 维度 | 契约测试 | E2E |
|------|---------|-----|
| 深度方向 | 接口精度——字段级、格式级、边界值级 | 业务流程——端到端链路完整性 |
| 接口错误归因 | 精确——直接定位 Provider 或 Consumer 违约 | 模糊——只知道"链路断了"，要猜断在哪里 |
| 业务逻辑覆盖 | 不覆盖——只看边界约定 | 广覆盖——完整用户场景 |

E2E 保证"业务没坏"，契约测试保证"接口约定没被破坏"。两者互补，不可互相替代。

#### Forge 提案的具体应用

Pattern 6（Pipeline Contract Gaps）正确识别了契约断裂，但修复方式是逐个补齐（W4 的 11 个修复），而非建立契约测试机制：

| 契约 | 当前修复 | 建议补充 |
|------|---------|---------|
| TC ID 格式（PC-1） | 改正则 | schema 约束 + producer-consumer 测试 |
| Slug 传播（PC-2） | 补逻辑 | 端到端契约：brainstorm → write-prd slug 一致性测试 |
| 状态查询命令（PC-4） | 统一命令 | CLI 接口 smoke test |
| 模板分派（TD-1） | 补映射 | 映射完整性断言：registry ⊇ mappings ⊇ disk files |

契约补丁修完 10 个，第 11 个会在未来出现。需要持续性契约测试防止回归。

### 5. 敏捷测试象限 — Q1 和 Q4 空白

| 象限 | 覆盖状态 | 需要补充 |
|------|---------|---------|
| Q1 技术+支持 | **未覆盖** | 状态机穷举、锁获取/释放、JSON 解析 |
| Q2 业务+支持 | 已覆盖（12 条 Success Criteria） | 充分 |
| Q4 技术+评价 | **未覆盖** | 并发压力、崩溃恢复、lock 超时 |

### 6. 测试左移 — 预防流程缺失

提案在架构层面左移了（集中式守卫），但流程层面没有：

- `ValidateTransition` 应 TDD 开发：先写状态转换测试矩阵，再实现函数
- Advisory lock 应有 CI guard：grep 检查所有 index.json 写入点是否持锁
- Eval 协议变更应先更新测试 fixture，再改实现

### 7. 高速反馈 — 分级未设计

"现有 e2e 测试全部通过"是分钟级反馈。理想分级：

| 层 | 覆盖内容 | 预期速度 |
|----|---------|---------|
| 单元 | `ValidateTransition`、JSON 解析、`countFixTasks` | 毫秒 |
| 集成 | claim+submit 端到端、eval 协议、契约一致性 | 秒 |
| 并发 | 多 goroutine claim、lock 超时、崩溃恢复 | 秒 |
| E2E 回归 | 现有测试套件 | 分钟 |

## 建议补充到提案 Scope 的内容

以下不应改变 W1-W4 的修复方案，而是补充验证手段：

**W1 补充**：
- `statemachine_test.go`：穷举状态转换矩阵（合法 + 非法转换），TDD 先行
- 失败级联 SM-1+SM-3 的专项负面测试

**W2 补充**：
- 并发 claim 压力测试：N goroutine 同时 claim，断言无重复
- `state.go` 崩溃恢复测试：写入中途截断，验证 `LoadState` 不丢数据
- QG-1+QG-2 端到端场景测试

**W3 补充**：
- Scorer JSON 解析模糊测试：格式偏差、缺失字段、空输出
- Scope 守卫的边界测试：合法白名单文件 vs 越界文件
- Eval 最大迭代回滚的正确性验证

**W4 补充**：
- 模板分派完整性断言：registry types ⊇ typeToTemplate keys ⊇ 磁盘模板文件
- Skill 间契约一致性测试（slug、TC ID 格式、CLI 命令）
