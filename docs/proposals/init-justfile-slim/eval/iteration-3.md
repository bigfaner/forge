# Proposal Evaluation -- Iteration 3

## Phase 1: Reasoning Audit

### Problem -> Solution Trace

**Chain**: 1645-line skill -> bash templates in LLM prompt -> token waste + scattered maintenance + responsibility misplacement -> CLI scaffold command + SKILL.md slimming.

Verdict: The causal chain is tight. Each of the three root causes maps to a specific remediation in the solution. This is the strongest aspect of the proposal.

**Residual gap (carried from iteration-0, never addressed)**: The problem statement added a "用户可见症状" paragraph in an earlier revision, which is present in this version (line 25). This partially addresses the user-facing gap. However, the symptoms listed ("命名风格不一致", "agent 生成耗时 30-60 秒", "每次 surface type 变更需同步更新 3-4 文件") are all developer-experience issues, not end-user pain. The urgency remains anchored to "v3.0.0 not yet released" (line 184) -- a window-of-opportunity argument, not a cost-of-delay argument.

### Solution -> Evidence Trace

Line counts verified: SKILL.md = 548, server-lifecycle.md = 745, surface rules = 318 (67+55+55+69+72), self-correction.md = 34. Total = 1645. Correct.

The "~250 行" SKILL.md target has a detailed breakdown in SC4 (line 14): frontmatter+intro (15) + params (5) + target contract (30) + flow Step 0-5 (100) + self-correction ref (5) + extremely-important (10) + notes (15) = 180 + self-correction.md 34 = 214. This totals 214, not 250. With safety margin 66, the ceiling is 280. The math now checks out, though the "250 行" claim in section header (line 111) is imprecise -- the actual target per SC4 is <= 280 lines total.

**CRITICAL: macOS platform claim is factually wrong**. Quote from line 68: "macOS 通过 `[linux]` 属性覆盖（just 将 macOS 归类为 linux 族）". I tested this with just 1.51.0 on macOS (Darwin 25.3.0):
- A recipe with only `[linux]` + `[windows]` variants produces `error: justfile does not contain recipe 'test-recipe'` on macOS.
- `[linux]` matches Linux only. `[macos]` matches macOS only. `[unix]` matches both.
- The proposal's claim that "just 将 macOS 归类为 linux 族" is false. Just treats macOS as a separate OS family from Linux.

This means **every single recipe generated by the CLI scaffold will be invisible on macOS** -- the primary Forge developer platform. This is not a theoretical risk; it is a confirmed implementation error baked into the proposal's core design.

### Evidence -> Success Criteria Trace

Five SCs are present. SC5 (line 15): "生成的 justfile 通过 `just --list` 列出所有旧版等价 recipe...且每个 recipe 的执行结果（exit code + 标准输出）与旧版一致". This is now more precise than iteration-2's "等价" -- it specifies both listing equivalence and execution equivalence. However, given the macOS bug, SC1 ("所有 5 种 surface type 语法正确，`just --list` 零错误") will fail on macOS for any recipe that only has `[linux]` + `[windows]` variants.

### Self-contradiction Check

1. **`<<URL_KEY>>` resolution source**: Line 85 says "Convention 中定义的 URL key 映射表；无 Convention 时从 config.yaml 结构自动推断（读取第一个 URL-like 字段名），推断失败时保留占位符". This is significantly improved from iteration-2 (which claimed `forge surfaces` as the source). However, "从 config.yaml 结构自动推断" is a CLI behavior claim that is not specified in the CLI command interface. The `--type` and `--key` parameters (line 49-53) do not include config file reading. If the agent is supposed to resolve `<<URL_KEY>>`, the CLI cannot also auto-infer it. The ownership is now muddled: is this an agent placeholder or a CLI auto-resolved value?

2. **`user-customized` marking expansion without justification**: Line 68 says "所有 lifecycle recipes 和 quality recipes 均标记 `# user-customized`". The iteration-2 review flagged this as a behavioral expansion. The proposal still does not justify why `compile`, `fmt`, `lint` should be protected from regeneration. These are typically deterministic toolchain commands that SHOULD update when toolchain changes. Marking them as user-customized means a user who changes their linter will have the old linter command preserved by the protection mechanism.

3. **SC4 target inconsistency**: Line 14 says "prompt 层总行数 <= 280 行" and breaks down to ~214 lines. Line 111 says "SKILL.md（548 行 -> ~250 行）". The two targets (280 total, 250 SKILL.md) may or may not be the same depending on whether the 280 includes self-correction.md (34 lines). If it does, 280 - 34 = 246 for SKILL.md alone, which is consistent with "~250". But the SC4 breakdown sums to 180 + 34 = 214, not 280. The 66-line "safety margin" is the gap, suggesting the proposal targets 214 but allows up to 280. This is defensible but the double-target (250 vs 280) creates ambiguity in the pass/fail criterion.

## Phase 3: Blindspot Hunt

[blindspot-1] **CONFIRMED: macOS platform bug -- all recipes invisible on macOS**
Iteration-2 flagged this. The proposal's response (line 68): "macOS 通过 `[linux]` 属性覆盖（just 将 macOS 归类为 linux 族）". I experimentally verified this is FALSE. Just's `[linux]` attribute matches Linux only. On macOS, recipes with only `[linux]` + `[windows]` variants produce `error: justfile does not contain recipe 'test-recipe'`. This is not a risk; it is a confirmed defect in the proposal's platform strategy. Every generated recipe will be non-functional on macOS.

[blindspot-2] **SC2 contradicts out-of-scope exclusion**
SC2 says "所有下游 consumer（run-tests skill、quality gate 机制）在新 recipe 命名模型下功能不变" (line 12). The out-of-scope section (line 228) excludes "其他 skill recipe 调用方" (fix-bug.md, clean-code/SKILL.md) and acknowledges these "在多 surface 项目中可能需要改为 `just <key>-unit-test`". If these consumers break, SC2's "所有" is not satisfiable. Either SC2 must be narrowed, or the scope must be widened.

[blindspot-3] **`--key` validation requires stateful CLI**
Line 53: "若对 scalar surface 传入 `--key`，CLI 报错退出." Line 55: "CLI 不需要知道任何项目细节." To validate whether `--key` is appropriate for a given surface type, the CLI must determine whether the surface is scalar or named. This requires querying `forge surfaces` or maintaining an internal mapping of which types are always scalar vs. always named. But any surface type can be either scalar or named depending on project configuration. The CLI cannot validate `--key` without project context, contradicting the statelessness claim.

[blindspot-4] **`user-customized` marking on quality recipes creates stale toolchain risk**
Line 68: "所有 lifecycle recipes 和 quality recipes 均标记 `# user-customized`". This means `compile`, `fmt`, `lint`, `unit-test` are protected from regeneration. If a user upgrades their toolchain (e.g., from ESLint to Biome), running `/init-justfile` again will preserve the old `eslint` command because it is marked `user-customized`. The proposal does not address how toolchain migration works with the protection mechanism. The current behavior (in the existing SKILL.md) only marks lifecycle and gate recipes as user-customized, not quality recipes. This is a behavioral expansion that was flagged in iteration-2 and remains unjustified.

[blindspot-5] **`ci` aggregate recipe semantic gap with Go quality gate**
The proposal defines `ci` = lint + compile + unit-test, excluding surface test because "surface test 需要运行时环境" (line 93). The Go `FullGateSequence()` function includes `test` and `probe` steps. If `ci` is meant to be the CI pipeline equivalent of the quality gate, it is a subset. If it is not, the relationship between `ci` and the quality gate is undefined. A consumer running `just ci` in CI may expect it to be equivalent to the full quality gate and be surprised when integration tests are skipped.

[blindspot-6] **Convention cold start fallback for `<<URL_KEY>>` is underspecified**
Line 115: "推断失败则使用语言默认值...无法推断的占位符保留 `<<PLACEHOLDER>>` 原样并在最终报告中列出". But `<<URL_KEY>>` is not a command that has a "language default" -- it is a YAML key name. There is no "default URL key name" for any language. The fallback for `<<URL_KEY>>` can only be "preserve placeholder and report", but this means the generated recipe is non-functional until the user manually fills it. The proposal does not quantify how often this will occur (how many projects lack a Convention file?).

[blindspot-7] **`test-setup` aggregate recipe boundary creates a semantic inconsistency**
Line 96: "`test-setup` 聚合边界：仅编排 mobile surface 的模拟器启动步骤，不包含 api/web 等其他 surface 的启动步骤（其他 surface 的启动由各自独立的 `<key>` aggregate recipe 处理）." But line 96 also says: "当项目存在多个需要运行时依赖的 surface（如 api + web），`--aggregate` 模式额外生成 `test-setup` 聚合 recipe，按依赖顺序编排各 surface 的启动和 teardown". These two statements contradict: the first says `test-setup` only handles mobile, the second says it orchestrates multi-surface startup for api+web combinations. Which is it?

## Summary

The proposal's core insight (separate deterministic template generation from LLM reasoning) remains sound and well-motivated. The revision improved SC4's measurability, clarified `<<URL_KEY>>` resolution strategy, and added Convention cold start fallback.

However, three issues from iteration-2 remain unaddressed or worsened:

1. **macOS platform claim is factually wrong** (blindspot-1, verified experimentally). This is a showstopper -- the proposed recipe generation will produce invisible recipes on the primary Forge developer platform.

2. **SC2 "所有下游 consumer" contradicts the out-of-scope exclusion** (blindspot-2). The success criterion cannot be satisfied as stated.

3. **`user-customized` marking expansion on quality recipes** (blindspot-4) remains unjustified and creates a stale toolchain risk.

These are not marginal issues. Blindspot-1 alone invalidates the platform strategy as specified. The proposal requires a design change before it can be implemented correctly.
