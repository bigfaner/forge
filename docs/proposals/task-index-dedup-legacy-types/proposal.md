# Proposal: Derive Test Types from config.yaml Capabilities

## Problem

`BuildIndex` reads `test-cases.md` at index time via `DetectTypesFromTestCases()` to decide whether to create per-type or generic test tasks. This is architecturally wrong: `BuildIndex` is a pre-creation step that should create tasks based on configuration, not runtime state. The dependency on `test-cases.md` (which is generated by a previous task) causes:

1. **Two-pass duplication**: First run (no test-cases.md) creates generic tasks; second run (after gen-cases) creates per-type variants. Both coexist as orphans.
2. **Non-deterministic output**: Same `forge task index` call produces different task graphs depending on whether a previous task has run.
3. **Dual code paths**: Legacy generic (`else`) branches in `GetQuickTestTasks`, `GetBreakdownTestTasks`, `resolveQuickDeps`, and `resolveBreakdownDeps` exist solely because types might be unknown at index time.

### Evidence

- `docs/features/e2e-test-quality-cleanup/tasks/index.json`: both `quick-gen-scripts-go-test` (T-quick-2, orphaned) and `quick-gen-scripts-go-test-cli` (T-quick-2-cli, active)
- Root cause: `DetectTypesFromTestCases()` in `build.go:273-275` reads a runtime artifact at pre-creation time

### Urgency

Every feature that runs the test pipeline hits the duplication bug. The dual code paths add maintenance burden. Part of 3.0.0.

## Proposed Solution

**Derive test types from `.forge/config.yaml`'s `capabilities` field, not from `test-cases.md`.**

The `forge init` / `forge config set` flow already resolves profile capabilities and lets the user select a subset, writing the result to `config.yaml`:

```
profile manifests → UnionCapabilities() → user selects subset → config.yaml capabilities
```

For this project: profile `go-test` supports `[tui, api, cli]`. The user selected `[cli]`. `BuildIndex` reads this field to create per-type tasks — always, regardless of whether `test-cases.md` exists.

**Fallback chain**: config.yaml capabilities → `UnionCapabilities(active profiles)` → no test tasks (only gen-cases tasks created).

This eliminates:
- The `DetectTypesFromTestCases()` call in `BuildIndex` (function deleted entirely)
- Both legacy `else` branches in all 4 functions
- The non-deterministic two-pass behavior
- The `detectedTypes` parameter from all function signatures

`BuildIndex` creates the full test pipeline in one pass, deterministically, based on configuration. `test-cases.md` is only consumed at task execution time.

### Architecture Change

```
Before (broken):
  BuildIndex → DetectTypesFromTestCases(test-cases.md) → generic OR per-type tasks
  (non-deterministic, depends on runtime state)

After (correct):
  BuildIndex → config.yaml capabilities → UnionCapabilities fallback → always per-type tasks
  (deterministic, config-driven)
```

### Innovation Highlights

This is a straightforward application of the principle "configuration drives structure, runtime data drives execution." The pattern is common in build systems (Make, Bazel) where task graphs are derived from config files, not from build outputs. The innovation is recognizing that `test-cases.md` is a build output (produced by T-test-1), so consuming it during graph construction is a layer violation.

## Requirements Analysis

### Key Scenarios

- **Single profile, multiple capabilities**: `go-test` with `[cli, api]` → T-quick-2-cli + T-quick-2-api
- **User narrows scope**: `go-test` supports `[tui, api, cli]`, user selects `[cli]` → only T-quick-2-cli created
- **Re-run with no changes**: Idempotent — same task graph (config doesn't change between runs)
- **No capabilities configured**: Fall back to `UnionCapabilities(active profiles)`. If still empty, only gen-cases tasks created.
- **Breakdown mode**: Same logic, different task IDs (T-test-2-* instead of T-quick-2-*)
- **First run (no test-cases.md)**: Still creates per-type tasks — test-cases.md is not consulted at index time

### Non-Functional Requirements

- **Determinism**: `forge task index` must produce identical output regardless of runtime state
- **Validation**: Invalid capabilities (values not in `{web-ui, tui, mobile-ui, api, cli}`) must be rejected with a clear error message

### Constraints & Dependencies

- Test type capabilities are a closed enum: `web-ui`, `tui`, `mobile-ui`, `api`, `cli` — now validated by `ValidTestTypes` in the profile package
- `BuildIndexOpts` gains a `TestCapabilities` field (resolved by caller)
- `profile.UnionCapabilities()` used as fallback when config has no capabilities set
- `forge init` and `forge config set` already populate config.yaml capabilities correctly — no changes needed to those flows

## Alternatives & Industry Benchmarking

### Industry Solutions

Build systems (Bazel, Make, Gradle) universally derive task graphs from configuration files, not from build outputs. The current code violates this principle by reading `test-cases.md` (a build output) during graph construction.

### Comparison Table

| Approach | Determinism | Source | Verdict |
|----------|------------|--------|---------|
| Do nothing | Broken | test-cases.md | Rejected: non-deterministic, orphaned tasks |
| Cleanup superseded tasks | Still non-deterministic | test-cases.md | Rejected: patches symptom, not root cause |
| Profile manifest capabilities | Deterministic | manifest.yaml | Acceptable but ignores user's narrowing selection |
| **config.yaml capabilities** | Deterministic | config.yaml | **Selected: respects user's capability selection** |

## Feasibility Assessment

### Technical Feasibility

Straightforward refactor. All plumbing exists: `ForgeConfig.Capabilities` field, `UnionCapabilities()` function, profile manifest capabilities. Only needs to wire config → BuildIndex and remove dead code.

### Resource & Timeline

Estimated 4-6 tasks. No external dependencies. Fits within a single sprint.

### Dependency Readiness

All dependencies are in place. `ReadConfig()` returns capabilities, `UnionCapabilities()` provides fallback, profile manifests define the valid enum.

## Scope

### In Scope

- Add `TestCapabilities []string` field to `BuildIndexOpts`
- Wire capabilities through callers (`index.go`, `add.go`): read from config → fallback to `UnionCapabilities(profiles)`
- Add `ValidTestTypes` constant set and `ValidateCapabilities()` function in profile package
- Remove `DetectTypesFromTestCases()` and `summaryTableRow` regex from `testgen.go`
- Remove legacy `else` branches from `GetQuickTestTasks`, `GetBreakdownTestTasks`, `resolveQuickDeps`, `resolveBreakdownDeps`
- Simplify signatures: remove `detectedTypes` parameter from all 4 functions
- Update unit tests

### Out of Scope

- Changes to skill files
- Changes to profile manifest format
- Changes to `forge init` / `forge config set` flows (already correct)
- Changes to test execution logic (test-cases.md is still consumed at execution time)

## Key Risks

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| config.yaml has no capabilities (not set or legacy) | M | L | Fall back to `UnionCapabilities(active profiles)`; if still empty, only gen-cases tasks created |
| Invalid capability values in config.yaml | L | M | `ValidateCapabilities()` rejects invalid values with clear error message |
| Creates tasks for capabilities with no test cases | M | L | Task succeeds trivially (gen-scripts produces no scripts for empty type) |
| Existing test breakage from signature changes | M | M | Table-driven test refactoring, verify with `go test -race -cover ./...` |
