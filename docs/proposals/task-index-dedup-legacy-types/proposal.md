# Proposal: Derive Test Types from config.yaml Capabilities

## Problem

`BuildIndex` reads `test-cases.md` at index time via `DetectTypesFromTestCases()` to decide whether to create per-type or generic test tasks. This is architecturally wrong: `BuildIndex` is a pre-creation step that should create tasks based on configuration, not runtime state. The dependency on `test-cases.md` (which is generated by a previous task) causes:

1. **Two-pass duplication**: First run (no test-cases.md) creates generic tasks; second run (after gen-cases) creates per-type variants. Both coexist as orphans.
2. **Non-deterministic output**: Same `forge task index` call produces different task graphs depending on whether a previous task has run.
3. **Dual code paths**: Legacy generic (`else`) branches in `GetQuickTestTasks` and `GetBreakdownTestTasks` exist solely because types might be unknown at index time.

### Evidence

- `docs/features/e2e-test-quality-cleanup/tasks/index.json`: both `quick-gen-scripts-go-test` (T-quick-2, orphaned) and `quick-gen-scripts-go-test-cli` (T-quick-2-cli, active)
- Root cause: `DetectTypesFromTestCases()` in `build.go:273-274` reads a runtime artifact at pre-creation time

### Urgency

Every feature that runs the test pipeline hits the duplication bug. The dual code paths add maintenance burden. Part of 3.0.0.

## Proposed Solution

**Derive test types from `.forge/config.yaml`'s `capabilities` field, not from `test-cases.md`.**

The `forge init` / `forge config set` flow already resolves profile capabilities and lets the user select a subset, writing the result to `config.yaml`:

```
profile manifests → UnionCapabilities() → user selects subset → config.yaml capabilities
```

For this project: profile `go-test` supports `[tui, api, cli]`. The user should select which apply (e.g., `[cli]`). `BuildIndex` reads this field to create per-type tasks.

This eliminates:
- The `DetectTypesFromTestCases()` call in `BuildIndex`
- Both legacy `else` branches (generic task creation)
- The non-deterministic two-pass behavior

`BuildIndex` creates the full test pipeline in one pass, deterministically, based on configuration. `test-cases.md` is only consumed at task execution time.

### Architecture Change

```
Before (broken):
  BuildIndex → DetectTypesFromTestCases(test-cases.md) → generic OR per-type tasks
  (non-deterministic, depends on runtime state)

After (correct):
  BuildIndex → config.yaml capabilities → always per-type tasks
  (deterministic, config-driven)
```

### Fix config.yaml

The current `.forge/config.yaml` has `capabilities: [compile, test, lint]` — these are project build capabilities, not test type capabilities. Should be corrected to match the profile's test type enum (e.g., `[cli]` for this project).

## Requirements Analysis

### Key Scenarios

- **Single profile, multiple capabilities**: `go-test` with `[cli, api]` → T-quick-2-cli + T-quick-2-api
- **User narrows scope**: `go-test` supports `[tui, api, cli]`, user selects `[cli]` → only T-quick-2-cli created
- **Re-run with no changes**: Idempotent — same task graph (config doesn't change between runs)
- **No capabilities configured**: Fall back to profile capabilities (union of all active profiles). If still empty, only gen-cases tasks created.
- **Breakdown mode**: Same logic, different task IDs (T-test-2-* instead of T-quick-2-*)

### Constraints & Dependencies

- Test type capabilities are a closed enum: `web-ui`, `tui`, `mobile-ui`, `api`, `cli`
- `BuildIndex` already receives profiles from config; adding capabilities read is straightforward
- `config.yaml` capabilities already populated by `forge init` and `forge config set`
- `profile.UnionCapabilities()` available for fallback when config has no capabilities set

## Alternatives & Comparison

| Approach | Determinism | Source | Verdict |
|----------|------------|--------|---------|
| Do nothing | Broken | test-cases.md | Rejected |
| Cleanup superseded tasks | Still non-deterministic | test-cases.md | Rejected: patches root cause |
| Profile manifest capabilities | Deterministic | manifest.yaml | Acceptable but ignores user's narrowing selection |
| **config.yaml capabilities** | Deterministic | config.yaml | **Selected: respects user's capability selection** |

## Scope

### In Scope

- Read `capabilities` from `.forge/config.yaml` in `BuildIndex` (or receive via `BuildIndexOpts`)
- Replace `DetectTypesFromTestCases()` call with config-driven capabilities
- Remove legacy `else` branches from `GetQuickTestTasks` and `GetBreakdownTestTasks`
- Remove legacy branches from `resolveQuickDeps` and `resolveBreakdownDeps`
- Update unit tests
- Fix `.forge/config.yaml` capabilities value

### Out of Scope

- Changes to skill files
- Changes to profile manifest format
- Changes to `forge init` / `forge config set` flows (already correct)
- Keeping `DetectTypesFromTestCases()` for other uses (evaluate separately)

## Key Risks

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| config.yaml has no capabilities (not set or legacy) | M | M | Fall back to `UnionCapabilities(active profiles)`; if still empty, only gen-cases tasks created |
| config.yaml has wrong capabilities (e.g., `[compile, test, lint]`) | H | M | `forge config set` should validate against closed enum; existing config needs manual fix |
| Creates tasks for capabilities with no test cases | M | L | Task succeeds trivially (gen-scripts produces no scripts for empty type) |
