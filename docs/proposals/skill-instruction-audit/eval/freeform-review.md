# Freeform Expert Review

**Reviewer**: Prompt Compliance Architect
**Document**: `docs/proposals/skill-instruction-audit/proposal.md`
**Date**: 2026-05-28

---

## Background Assessment

This proposal identifies three categories of systemic defects across the Forge plugin's instruction layer (22 skills, 19 commands, 1 agent): CLI behavior description instead of imperative instruction, redundancy between EXTREMELY-IMPORTANT blocks and body text (and between frontmatter and body), and clarity/self-consistency failures such as skipped step numbers, misleading cross-references, and ambiguous fallback logic. The proposed fix is straightforward deletion and correction: remove CLI behavioral explanations, trim redundant emphasis markers, and fix numbering and references.

The core insight the proposal advances is that AI agent instructions should be imperative rather than descriptive -- an agent does not need to understand why a CLI command works, only what to run and what to check. This is a sound principle from prompt engineering: every token that does not directly contribute to correct execution is cognitive load that may interfere with compliance. The proposal explicitly recognizes that this principle inverts the human documentation pattern where repetition reinforces memory, acknowledging that for LLMs, repetition introduces inconsistency risk instead.

The proposal claims approximately 95 modifications across ~40 files, organized into three batches by defect type, and estimates 8-12 coding tasks. It explicitly scopes out cross-file deduplication (keeping quick-tasks and breakdown-tasks independent) and any functional changes to skills.

## Key Risks

The proposal is well-scoped and its underlying principle is sound, but several risks arise from the way it frames the problem, the boundaries it draws, and the assumptions it makes about verification.

风险：The proposal's central claim that "约 22 处描述了 forge CLI 的内部行为而非直接使用命令" conflates two distinct categories of information: true behavioral description (explaining what a command does internally) and output parsing instructions (explaining how to interpret CLI output fields). The proposal acknowledges this partially in its risk mitigation table -- "保留 exit code 契约（0=成功/1=失败）和输出字段名列表，只删除语义解释" -- but the boundary is far subtler than the binary "delete CLI descriptions / keep imperative instructions" framing suggests. In `execute-task.md`, the Step 1 output parsing section lists field names like `TASK_ID`, `FILE`, `MAIN_SESSION`, `SURFACE_KEY`, `SURFACE_TYPE`, `FEATURE` alongside explanations like "(e.g., full absolute path to task file)" and "defaults to empty string if absent". These are not behavioral descriptions -- they are output contracts the agent must understand to correctly extract values. If the proposal treats all such annotations as "CLI behavior description" and deletes them, the agent loses the information needed to parse correctly. If it preserves them, the actual scope of deletion is much smaller than the 22 instances claimed.

> "删除 CLI 行为描述：只保留"运行 X 命令"的指令，删除对 CLI 输出语义、内部实现、分支逻辑的解释" -- The phrase "CLI 输出语义" is dangerously ambiguous. Output field semantics (what `SURFACE_KEY` means, when it is absent) are not the same as internal implementation details (how `forge task index` decides between branches). The proposal does not define a clear boundary, which means different task executors will apply different deletion criteria, leading to inconsistent results across the 40 files.

风险：The proposal claims "EXTREMELY-IMPORTANT 块 10 条中 8 条与正文步骤重复" for `test-guide/SKILL.md`, but my verification shows the overlap analysis is incomplete. Reading the actual file, the EXTREMELY-IMPORTANT block contains 10 items, and while several do correspond to body text, some items -- particularly "When regenerating after rejection, preserve approved sections verbatim and regenerate ONLY rejected sections" and "After 2 retry rejections, write the draft as `.draft.md` for manual editing. Do NOT force-apply." -- encode behavioral constraints that span multiple steps and serve as guardrails against common LLM failure modes (premature auto-application, silent overwrites). These are exactly the type of cross-step constraints the proposal itself says should be kept ("只保留正文未覆盖的跨步骤约束"). The risk is that a task executor, following the instruction "remove items that duplicate body text," may mechanically match keywords and delete these guardrails because they reference concepts that also appear in the body, even though the E-I version encodes a stricter constraint than the body version.

> "精简 EXTREMELY-IMPORTANT 块（只保留正文未覆盖的跨步骤约束）" -- The "正文未覆盖" criterion is itself ambiguous. Does "覆盖" mean "mentions the same topic" or "expresses the same constraint at the same strictness level"? The test-guide E-I block item "Do NOT execute any test commands" corresponds to a body note "No code execution: This skill is entirely LLM-driven file analysis and generation." The body note is informational; the E-I item is a hard prohibition with emphasis markers. They cover the same topic but at different enforcement levels. A mechanical deduplication would delete the E-I item, losing the behavioral anchor precisely where it matters most.

问题：The proposal identifies "quick-tasks ↔ breakdown-tasks 之间 12 处近乎逐字重复" as evidence of redundancy, but then explicitly scopes out cross-file deduplication: "quick-tasks↔breakdown-tasks、execute-task↔run-tasks 保持独立". This creates an internal tension. If 12 near-verbatim duplications exist between two skill files, and both files are being modified in this same proposal (both have CLI descriptions and E-I redundancy to fix), the natural moment to address the cross-file duplication is now, while both files are being touched. Scoping it out means these 12 duplications will persist, and any future change to one file will require manually synchronizing the other -- exactly the maintenance burden the proposal identifies as the core problem.

> "跨文件去重（quick-tasks↔breakdown-tasks、execute-task↔run-tasks 保持独立）" -- This out-of-scope decision is justified by " Constraints & Dependencies" stating "quick-tasks 和 breakdown-tasks 必须各自独立自洽，不抽取共享 rule 文件". But "independent and self-consistent" does not require "identical wording" -- it requires each file to be complete on its own. The two goals are achievable simultaneously: each file can contain its own copy of shared rules, but expressed with wording appropriate to its context (proposal input vs. tech-design input). The proposal conflates structural independence with textual duplication.

风险：The proposal's success criterion for E-I deduplication is unenforceable as stated: "EXTREMELY-IMPORTANT 块中与正文重复的条目全部删除（每个 E-I 块条目在正文中无对应）". This verification method -- checking whether each E-I item "has a corresponding entry in the body" -- is subject to the same ambiguity as the deletion criterion. An E-I item stating "submit-task IS MANDATORY" corresponds to a body step that says "invoke submit-task skill." The body mentions submit-task; the E-I item emphasizes its mandatory nature. Does the body "have a corresponding entry"? By keyword matching, yes. By constraint-level matching, no -- the body step does not encode the MUST NOT SKIP semantics. This verification method will produce false positives (flagging items as "deduplicated" when they actually encode stronger constraints than the body).

> "EXTREMELY-IMPORTANT 块中与正文重复的条目全部删除（每个 E-I 块条目在正文中无对应）" -- The parenthetical verification instruction "(每个 E-I 块条目在正文中无对应)" defines a necessary condition for correct deduplication but not a sufficient one. An E-I item with no body correspondence might still be redundant (if it restates a HARD-RULE or HARD-GATE already present elsewhere). Conversely, an E-I item with body correspondence might encode a genuinely stronger constraint. The verification criterion misses both edge cases.

问题：The proposal identifies the `quick.md` fallback logic error as "配置读取失败时 fallback 为跳过确认门（逻辑倒置）" and proposes fixing it so "fallback 为显示确认门". However, reading the actual `quick.md` source, the current code says: "Non-zero (config missing/read error) | Fallback: skip the confirmation gate (same as quick: true)." This is explicitly a design choice documented with reasoning: "This preserves quick mode's streamlined nature." Whether this is a bug or a feature depends on the threat model. If the threat is "accidentally running tasks the user didn't intend," the fallback should be to show the gate. If the threat is "quick mode becomes unusable when config is broken," the fallback should be to skip. The proposal treats this as an unambiguous bug, but the source file contains an explicit design justification that the proposal does not acknowledge or refute.

> "quick.md 配置读取失败时 fallback 为跳过确认门（逻辑倒置）" and "quick.md 配置读取失败时 fallback 为显示确认门（非跳过）" -- The proposal's fix reverses the explicit design intent documented in the source: "This preserves quick mode's streamlined nature." Without acknowledging and arguing against this design intent, the fix reads as a unilateral reversal rather than a reasoned correction. The proposal should explicitly address why the original design justification is wrong.

问题：The proposal claims `gen-contracts/SKILL.md` 引用不存在的 Section 编号 but does not specify which section number or where in the file. I read the first 50 lines of `gen-contracts/SKILL.md` and found no obvious broken section references. This could be deeper in the file, but the proposal's evidence section does not provide enough specificity to verify the claim or to write a targeted fix task. This is a microcosm of a broader pattern: the proposal provides high-level pattern evidence (which is convincing) but individual instance evidence that is sometimes too vague for verification.

> "gen-contracts/SKILL.md 引用不存在的 Section 编号" -- No specific section number is cited, no line reference, and no quote from the file. A task executor assigned to "fix this issue" would need to re-audit the file to find what the proposal author found, duplicating work and potentially finding a different issue (or finding none if the original was misidentified).

风险：The proposal's verification strategy relies primarily on structural checks (grep for "What .* Does" sections, verify step number sequences, check frontmatter-to-body duplication). But the most consequential category of changes -- deleting CLI behavioral descriptions -- cannot be verified by structural patterns. There is no regex that distinguishes "this sentence is a behavioral description that should be deleted" from "this sentence is an output contract that should be preserved." The proposal's success criteria acknowledge this for the risk table ("保留 exit code 契约... 只删除语义解释") but do not translate it into a verification method for the success criteria section.

> "22 处 CLI 行为描述全部删除，替换为指令性操作（grep 无 'What .* Does' section）" -- The grep-based verification only catches sections titled "What X Does." It does not catch inline behavioral descriptions within step instructions (e.g., the explanatory text in `execute-task.md` Step 1.5 that describes what MAIN_SESSION routing does, or the text in `quick-tasks/SKILL.md` that explains what `forge task index` generates). These inline descriptions are the harder and more numerous category, and the proposal provides no verification method for them.

## Improvement Suggestions

建议：Define a precise boundary between "CLI behavioral description" (to delete) and "output contract" (to preserve) before starting any task. The boundary should be stated as a rule, not a judgment call per instance.
Addresses: Risk about conflation of behavioral description vs. output parsing instructions.
> What changes: Add a "Deletion Boundary Rules" subsection to the proposal that classifies CLI-adjacent text into three categories: (1) imperative instructions ("run X, check exit code 0") -- keep; (2) output contracts ("FIELD_NAME contains X, absent means Y") -- keep; (3) behavioral explanations ("this command does X internally by checking Y and then Z") -- delete. Provide 2-3 examples per category drawn from the actual files. This turns a judgment call into a mechanical classification, reducing inter-task inconsistency.

建议：Replace the keyword-overlap verification method for E-I deduplication with a constraint-level audit. Instead of checking whether each E-I item "has a corresponding entry in the body," verify that each E-I item either (a) encodes a constraint absent from the body, or (b) encodes a constraint at a higher enforcement level than the body.
Addresses: Risk about unenforceable E-I deduplication success criterion.
> What changes: The success criterion for E-I deduplication should read: "每个保留的 E-I 条目满足以下之一：(a) 正文未包含该约束，或 (b) 正文包含该约束但 E-I 版本的强制等级更高（如正文为建议性描述，E-I 为 MUST NOT/MUST）。" This gives task executors a concrete decision procedure and prevents mechanical keyword-based deduplication.

建议：Acknowledge the `quick.md` fallback's explicit design justification and argue against it before proposing the fix.
Addresses: Problem about the `quick.md` config-read-failure fallback.
> What changes: In the Evidence section, add: "quick.md 在 Non-zero exit code 行注释中明确写道 'This preserves quick mode's streamlined nature'，表明这是一个有意的 design choice。然而，该选择忽略了配置读取失败的两种原因：(1) config 文件不存在（新用户，应走确认流程）；(2) config 文件损坏（应 fail-safe 而非 fail-open）。在两种情况下，跳过确认门都不是正确的行为。" This turns the fix from a unilateral reversal into a reasoned argument, and the task executor can incorporate the reasoning into the commit message.

建议：Provide specific line references or quotes for each of the 40 clarity/self-consistency issues, especially the ones that lack clear search targets (like the gen-contracts section number issue).
Addresses: Problem about insufficient specificity for individual instances.
> What changes: For each of the 40 clarity/self-consistency issues, add either a file line number or a direct quote of the problematic text. If the proposal intends for task executors to re-discover these issues, state that explicitly: "执行者需重新审计以下文件，基于描述的模式定位具体位置。" If the locations are known (which they should be, since the proposal claims to have conducted a line-by-line audit), including them saves significant duplicate work.

建议：Reconsider the out-of-scope decision on cross-file deduplication. If both quick-tasks and breakdown-tasks are being modified in this proposal anyway, use the opportunity to reduce (not eliminate) cross-file duplication by applying a "same rule, wording appropriate to context" standard rather than the current "identical wording" standard.
Addresses: Problem about internal tension between claiming cross-file duplication as evidence and scoping it out.
> What changes: Add an intermediate scope category: "In Scope (lightweight): 对于 quick-tasks↔breakdown-tasks 之间逐字重复的段落，将 breakdown-tasks 版本改为引用 tech-design 上下文的措辞（保持语义等价，但措辞区别于 quick-tasks）。这不影响各自独立性，仅减少逐字复制带来的维护同步负担。" This captures most of the benefit of deduplication without violating the independence constraint.
